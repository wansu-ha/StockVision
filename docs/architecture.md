# StockVision 아키텍처

> 최종 갱신: 2026-03-06 | 오프라인 내성, 신호등 통합, 종목 데이터 분리

---

## 1. 시스템 개요

AI 기반 주식 예측 + 시스템매매 자동화 플랫폼.
사용자가 정의한 규칙을 사용자의 PC에서 자동 실행한다.

**법적 포지션**: 시스템매매 (투자일임·투자자문 아님)
- 사용자가 직접 규칙 정의 → 사용자 PC에서 실행
- 우리 서버가 매매 판단/명령을 내리지 않음
- AI/LLM은 데이터 제공자(컨텍스트 변수) 역할만

---

## 2. 프로세스 아키텍처

| # | 프로세스 | 위치 | 역할 |
|---|---------|------|------|
| 1 | **프론트엔드** | 정적 호스팅 | React SPA |
| 2 | **클라우드 서버** | 클라우드 | 인증, 규칙 CRUD, 종목 메타, 시세 수집/저장, AI 분석, 어드민 |
| 3 | **로컬 서버** | 사용자 PC | 엔진 + 키움 주문 + 로그 (프로젝트 핵심) |

> **왜 3프로세스인가?**
> - API 서버와 데이터 서버는 동일한 I/O 바운드 프로파일 → 별도 프로세스 불필요
> - LLM은 Claude API (HTTP 호출) → GPU 서버 불필요, 클라우드 서버 내부 모듈로 흡수
> - 개인 프로젝트 규모에서 마이크로서비스 오버헤드 > 이점

## 3. 전체 데이터 흐름

```
[프론트엔드] (사용자 브라우저) — 순수 UI
 ├── HTTPS ──→ [클라우드 서버]
 │              ├ 인증 (JWT)
 │              ├ 규칙 CRUD
 │              ├ 관심종목 CRUD
 │              ├ 종목 검색 (StockMaster)
 │              ├ 어드민
 │              └ AI 분석 결과 조회
 │
 └── localhost ──→ [로컬 서버]
                    ├ JWT 전달 (로그인 직후 1회)
                    ├ 규칙 sync (규칙 저장 직후)
                    ├ API Key 등록
                    ├ 로그 조회
                    └ WS: 실시간 시세 + 체결 알림

[로컬 서버] (사용자 PC) — 자립 동작
 ├── HTTPS ──→ [클라우드 서버]
 │              ├ 하트비트 전송 (주기적, 30초~1분)
 │              │   → 응답에 rules_version, context_version 포함
 │              │   → 버전 다르면 fetch
 │              ├ 규칙 sync (로컬 변경 시 업로드)
 │              ├ 컨텍스트 fetch (버전 변경 시)
 │              ├ 템플릿 목록 fetch
 │              └ JWT 자동 갱신 (Refresh Token)
 │
 ├── 키움 WS (유저 키) ──→ 실시간 시세 수신
 └── 키움 REST (유저 키) ──→ 주문 실행

[클라우드 서버] — 종목 메타 + 시세 수집 + AI 분석
 ├── 공공데이터포털 (금융위원회) ──→ 종목 메타데이터 수집 (일 1회)
 ├── 키움 WS (서비스 키) ──→ openapi.kiwoom.com
 │   시세 수집/저장 (내부 사용, 유저 재배포 아님)
 └── Claude API (HTTP) ──→ AI 컨텍스트 생성 (감성 점수, 뉴스 요약 등)
```

### 3.1 주문 흐름 (핵심)

```
1. 로컬 서버: 키움 WS로 시세 수신 (유저 키)
2. 로컬 서버: 전략 엔진이 규칙 평가 → "매수" 판단
3. 로컬 서버: 키움 REST로 현재가 직접 재확인 (가격 검증)
4. 가격 일치 → 키움 REST로 주문 실행 / 불일치 → 거부 + 로그
5. 로컬 서버: 체결 결과 → 로컬 로그 저장 (SQLite)
6. 프론트엔드: 체결 알림 수신 (localhost WS)
7. (v2) 프론트엔드 → 클라우드 서버: 성공/실패만 보고 (종목·금액 미포함)
```

**가격 검증 이유**: 클라우드가 시세를 가공한다는 의혹 방지 + 오주문 방지.
로컬 서버가 유저 키움 계정으로 직접 확인한 가격만 신뢰.

### 3.2 규칙 동기화 흐름

```
시나리오 A: 브라우저에서 규칙 수정
  [프론트엔드] → localhost에 저장 → [로컬 서버] → 클라우드에 sync

시나리오 B: 다른 기기에서 규칙 수정 (폰 등)
  [다른 기기] → 클라우드에 저장 → rules_version 증가
  [로컬 서버] → 하트비트 응답에서 버전 차이 감지 → 클라우드에서 fetch
```

- 규칙은 **사용자 개인 소유** (다른 사용자와 공유 아님)
- 클라우드 서버 DB에 원본, 로컬 서버에 JSON 캐시
- 버전 차이 감지는 하트비트 응답 (30초~1분 주기)
- 클라우드 장애 시 캐시된 규칙으로 계속 실행
- 충돌 해소: last-write-wins (각 항목의 `updated_at` UTC 비교)

### 3.3 오프라인 내성 (클라우드 다운 시)

로컬 서버가 있는 PC에서는 클라우드 장애 시에도 대부분의 기능이 동작한다.

**계속 되는 것:**
- 자동 매매 (캐시된 규칙 + 키움 독립 연결)
- 실시간 시세 (유저 키 → 키움 WS)
- 체결 알림 (로컬 WS + 트레이 토스트)
- 실행 로그 조회 (로컬 SQLite)
- 엔진 시작/중지, Kill Switch
- 규칙 생성/수정 (로컬 저장 → 복구 후 sync)
- 관심종목 관리 (로컬 캐시 → 복구 후 sync)
- 종목 검색 (로컬 StockMaster 캐시 fallback)

**안 되는 것:**
- 신규 로그인 (다른 기기에서 처음 접속)
- 다른 기기 간 동기화

**sync 큐**: 클라우드 다운 중 변경사항은 로컬에 큐로 쌓이고,
복구 시 일괄 sync. 충돌은 `updated_at` 비교로 해소.

## 4. 각 프로세스 상세

### 4.1 프론트엔드

React SPA. 정적 호스팅 (Vercel, Nginx 등). **순수 UI**.

**연결**:
- 클라우드 서버 HTTPS: 인증, 규칙 CRUD, 관심종목 CRUD, 종목 검색, 어드민
- localhost HTTP: JWT 전달, 규칙 sync, API Key 등록, 로그 조회
- localhost WS: 실시간 시세, 체결 내역

**역할**:
- 최초 로그인 → JWT를 localhost에 전달 (이후 로컬 서버 자립)
- 종목 검색 (클라우드 StockMaster → 실패 시 로컬 캐시 fallback)
- 관심종목 관리 (클라우드 저장, 다른 기기에서도 접근)
- 전략 규칙 CRUD → 저장 후 localhost에 sync
- 실시간 시세 + 체결 모니터링
- 실행 로그 열람
- 설정 (API Key 등록, 모드 전환)
- 어드민 (관리자만)

**로컬 서버 없이 사용 가능한 범위**:
- 가능: 로그인, 종목 검색, 관심종목, 규칙 생성/수정, 프로필
- 불가: 시세, 체결, 엔진 제어, 수익 확인

### 4.2 클라우드 서버

FastAPI. 인증 + 규칙 + 시세 수집 + AI 분석을 하나의 프로세스로 통합.

**역할**:
- 사용자 인증 (이메일 + JWT)
- 전략 규칙 CRUD (저장소)
- 관심종목 CRUD (Watchlist)
- 종목 메타데이터 관리 (StockMaster — 공공데이터포털에서 수집)
- 종목 검색 API (메타데이터만, 시세 미포함)
- 서비스 키움 키로 시세 수집/저장 (내부 분석용)
- AI 컨텍스트 생성 (Claude API 호출 → 감성 점수, 뉴스 요약 등)
- 어드민 (유저 관리, 시스템 통계, 서비스 키 관리)
- 하트비트 수신 (익명 통계)
- 커뮤니티 전략 공유 (미래)

**내부 모듈 구조**:
```
cloud_server/
├── auth/          # 인증 (JWT, 회원가입)
├── rules/         # 전략 규칙 CRUD
├── watchlist/     # 관심종목 CRUD
├── stocks/        # 종목 메타데이터 검색/관리
├── market_data/   # 시세 수집/저장 (서비스 키)
├── ai_analysis/   # Claude API 호출, 컨텍스트 생성
├── admin/         # 어드민
└── heartbeat/     # 하트비트 수신
```

**보유 데이터**:
- 사용자 계정 (이메일 — 최소 수집)
- 전략 규칙 (조건/지표 — 개인정보 아님)
- 관심종목 (종목코드 — 공개 정보)
- 종목 메타데이터 (이름, 코드, 시장, 섹터 — 공공데이터)
- 시세 데이터 (히스토리컬 바, 분봉 — 공개 정보)
- AI 분석 결과 (감성 점수, 요약 — 캐시)
- 하트비트 (익명 UUID + 버전)

**보유하지 않는 데이터**:
- 키움 API Key/Secret (유저 키)
- 체결 내역, 잔고, 수익률
- 어떤 종목을 거래하는지 (관심종목 ≠ 거래종목)

**종목 데이터 분리**:
- 메타데이터 (이름, 코드, 시장, 섹터) → 클라우드가 관리, 유저에게 제공 (공개 정보)
- 시세 데이터 (가격, OHLCV) → 서비스 키 내부 수집용, 유저 재배포 안 함
- 유저 시세는 각자의 키움 키로 로컬에서 직접 수신

**메타데이터 수집원**: 공공데이터포털 (금융위원회_KRX상장종목정보).
키움 API는 시세/주문에만 사용. 메타데이터는 공공데이터로 분리하여 약관 부담 제로.

**서비스 키움 키**: 우리 소유. 어드민 페이지에서 등록/관리.
시세는 공개 정보이나, **유저에게 재배포하지 않음** (내부 분석용).
유저 시세는 각자의 키움 키로 로컬에서 직접 수신.

**시세 재배포 제한**: 키움 제5조③ 시세 중계 금지.
상용화 시 코스콤 정식 라이선스로 전환 예정.

### 4.3 브로커 추상화 (Adapter 패턴)

로컬 서버와 클라우드 서버는 증권사 API를 직접 호출하지 않고,
**`BrokerAdapter` 인터페이스**를 통해 접근한다.

```
전략 엔진 / 로컬 서버 / 클라우드 서버
        ↓
  BrokerAdapter (ABC)          ← 코드가 의존하는 유일한 인터페이스
        ↓
  ┌─────────────┬──────────────┬──────────────┐
  │ KiwoomAdapter│ KISAdapter   │ ...Adapter   │
  │ (v1 구현)    │ (v3+ 미래)   │ (확장)       │
  └─────────────┴──────────────┴──────────────┘
```

**BrokerAdapter 인터페이스:**

| 카테고리 | 메서드 | 설명 |
|---------|--------|------|
| 인증 | `authenticate()` | OAuth/토큰 발급 |
| 인증 | `is_authenticated()` | 토큰 유효 여부 |
| 주문 | `send_order(symbol, side, qty, price, order_type)` | 매수/매도 |
| 주문 | `cancel_order(order_no)` | 주문 취소 |
| 조회 | `get_current_price(symbol)` | 현재가 |
| 조회 | `get_balance(account_no)` | 예수금/총평가 |
| 조회 | `get_positions(account_no)` | 보유종목 |
| 실시간 | `subscribe(symbols, data_type)` | 시세 구독 |
| 실시간 | `unsubscribe(symbols, data_type)` | 구독 해제 |
| 실시간 | `listen()` | 이벤트 스트림 (AsyncIterator) |

**공통 모델**: `OrderResult`, `BalanceResult`, `Position`, `QuoteEvent` —
모든 Adapter가 동일한 데이터 모델을 반환한다. 증권사별 응답은 Adapter 내부에서 변환.

**설정으로 선택**:
```json
// config.json
{ "broker": "kiwoom" }
```
→ 팩토리가 `KiwoomAdapter` 인스턴스를 생성하여 주입.

### 4.4 로컬 서버 (프로젝트 핵심)

사용자 PC에서 실행되는 FastAPI 서버. **프로젝트의 정수**.

**역할**:
- BrokerAdapter를 통해 시세 수신 + 주문 실행
- 전략 엔진 (규칙 평가 → 주문 판단)
- 주문 전 가격 검증 (BrokerAdapter로 직접 확인)
- 실행 로그 저장 (SQLite)
- 클라우드 서버 HTTP 폴링 (JWT 사용):
  - 하트비트 (30초~1분) → 응답에서 버전 변경 감지
  - 규칙/컨텍스트 fetch (버전 변경 시)
  - 규칙 sync (로컬 변경 시 업로드)
  - 템플릿 목록 fetch
  - JWT 자동 갱신 (Refresh Token → Credential Manager)
- 규칙 캐시 (JSON 파일)
- 수익 리포트 생성 (커뮤니티 공유 시, 유저 선택)

**배포**: 단일 .exe (PyInstaller 번들)
**자동 시작**: 레지스트리 Run 키 (`HKCU\Software\Microsoft\Windows\CurrentVersion\Run`)
**시스템 트레이**: 백그라운드 실행, 아이콘 색상으로 상태 표시, 체결 시 Windows 토스트 알림
**자동 로그인**: Refresh Token이 Credential Manager에 저장되어 PC 재부팅 후 자동 복구

### 4.5 AI 분석 모듈 (클라우드 서버 내부)

별도 프로세스 아닌 클라우드 서버의 내부 모듈. Claude API = HTTP 호출이므로 GPU 불필요.

**역할**:
- 시세 데이터 + 외부 데이터 (뉴스 등) 분석 → Claude API 호출
- AI 컨텍스트 생성 (감성 점수, 리스크 지표, 뉴스 요약 등)
- 결과 캐싱 (동일 요청 재호출 방지)

**주의**: LLM은 "데이터 제공" 역할만 (감성 점수, 요약).
"매수하세요" 같은 직접 조언 → 투자자문업 → 금지.

## 5. 데이터 위치 및 법적 근거

| 데이터 | 위치 | 법적 근거 |
|--------|------|----------|
| 키움 API Key/Secret | 로컬 (Credential Manager) | 키움 약관 제3자 위임 금지 |
| Refresh Token | 로컬 (Credential Manager) | 자동 로그인, 자격 증명 |
| 전략 규칙 (캐시) | 로컬 (JSON 파일) | 클라우드 서버 DB에도 평문 저장, 공유/백업 편의 |
| 실행 로그 (체결가, 수량, 잔고) | 로컬 (SQLite) | 개인 금융정보, OS 레벨 보호 |
| 설정 (모드 등) | 로컬 (config.json) | — |
| 관심종목 (캐시) | 로컬 (JSON 파일) | 클라우드 DB에도 저장, 오프라인 대응 |
| 종목 메타 (캐시) | 로컬 (JSON 파일) | 공공데이터, 오프라인 검색 대응 |
| 종목 상세 (캐시) | 로컬 (JSON 파일) | 조회한 종목만 온디맨드 캐싱 |
| 전략 규칙 (원본) | 클라우드 DB | 개인정보 아님 (조건/지표) |
| 관심종목 (원본) | 클라우드 DB | 종목코드 = 공개 정보 |
| 종목 메타데이터 | 클라우드 DB | 공공데이터포털 수집 (공개 정보) |
| 사용자 계정 | 클라우드 DB | 이메일만 최소 수집 |
| 시세 데이터 | 클라우드 서버 DB | 공개 정보 (내부 사용) |
| AI 분석 결과 | 클라우드 서버 (캐시) | Claude API 응답 캐시 |

### 5.1 로컬 저장 구조

```
%APPDATA%\StockVision\
├── strategies.json        # 전략 규칙 캐시 (평문 JSON)
├── watchlist.json         # 관심종목 캐시 (오프라인 대응)
├── stock_master.json      # 종목 메타데이터 캐시 (전체 ~150KB, 오프라인 검색)
├── stock_detail_cache.json # 조회한 종목 상세 캐시 (온디맨드)
├── config.json            # 모드, UI 설정
├── sync_queue.json        # 클라우드 미연결 시 변경사항 큐
├── logs.db                # SQLite — 체결 로그, 오류 로그 (암호화 안 함)
└── Windows Credential Manager:
    ├── StockVision/app_key          # 키움 API Key
    ├── StockVision/app_secret       # 키움 API Secret
    └── StockVision/refresh_token    # JWT Refresh Token (자동 로그인)
```

### 5.2 키 분리

| 키 | 소유 | 위치 | 용도 |
|----|------|------|------|
| 서비스 키 | 우리 (운영자) | 클라우드 서버 | 시세 수집 (내부) |
| 유저 키 | 각 사용자 | 로컬 PC | 시세 수신 + 주문 실행 |

### 5.3 개인정보 원칙

- 우리 클라우드에 **개인 금융정보 미보유**
- 체결 내역, 잔고, 수익률 → 로컬에만 존재
- **"공유하지 않으면 우리도 모른다"** 가 구조적으로 보장됨
- 커뮤니티 수익 공유 → 유저 자발적 업로드 (로컬에서 계산 → 유저가 공개 선택)
- 공유 데이터 활용: 이용약관에 명시 (서비스 개선, 통계, AI 학습)

## 6. 보안 모델

### 6.1 인증

| 구간 | 방식 |
|------|------|
| 프론트 ↔ 클라우드 서버 | 이메일 + JWT (1h) + Refresh Token (7~30d, Rotation) |
| 프론트 ↔ 로컬 서버 | localhost (인증 불필요, 외부 접근 불가) |
| 로컬 서버 ↔ 클라우드 서버 | JWT (프론트에서 전달받음, Refresh Token으로 자동 갱신) |
| 로컬 서버 → 키움 | OAuth Bearer Token (24h, 유저 키로 발급) |
| 클라우드 서버 → 키움 | OAuth Bearer Token (24h, 서비스 키로 발급) |
| 클라우드 서버 → Claude API | API Key (서버 환경변수) |

**로컬 서버 자동 로그인**: Refresh Token을 Credential Manager에 보관.
PC 재부팅 시 Refresh Token으로 새 JWT 자동 발급 → 브라우저 없이 복구.
Refresh Token 만료 시에만 브라우저 재로그인 필요.

**Claude API 키**: 서버 환경변수로 관리 (`ANTHROPIC_API_KEY`). 유저에게 노출 안 함.

### 6.2 암호화

| 대상 | 방식 |
|------|------|
| API Key/Secret | Windows Credential Manager (DPAPI) |
| Refresh Token | Windows Credential Manager (DPAPI) |
| 전략 규칙 캐시 | 평문 JSON (클라우드 DB에도 평문, 암호화 불필요) |
| 체결 로그 | 평문 SQLite (OS 레벨 보호 위임) |
| 클라우드 설정 blob | AES-256-GCM (서버사이드) |
| 비밀번호 | Argon2id 해싱 |

## 7. 신호등 시스템

### 7.1 프론트엔드 (1개 통합)

**"지금 거래 가능한가?"** 에 대한 단일 신호등.

| 색상 | 의미 | 확인 방법 |
|------|------|----------|
| 🟢 | 거래 가능 (로컬 + 키움 정상) | localhost `/api/status` → 키움 연결 확인 |
| 🟡 | 거래 불가 (키움 미연결 또는 Kill Switch) | localhost 응답 OK, 키움 미연결 또는 Kill Switch 활성 |
| 🔴 | 로컬 미연결 (규칙 관리만 가능) | localhost 무응답 |

- 엔진 상태(시작/중지)는 신호등이 아닌 대시보드 버튼으로 분리
- 키움 세부 상태(토큰 만료, WS 끊김 등)는 퉁침 — 자동 복구 대상이므로
- 클라우드 장애는 에러 배너/토스트로 별도 처리
- hover 툴팁: "로컬 서버: 연결됨 | 키움: 연결됨 | 엔진: 가동 중"

### 7.2 트레이 아이콘

```
아이콘 색상 = 전체 상태 요약 (프론트와 동일 기준)
  🟢 초록 = 로컬 + 키움 정상, 거래 가능
  🟡 노랑 = 키움 미연결 또는 Kill Switch 활성
  🔴 빨강 = 오류 (키움 끊김 + 하트비트 장기 실패 등)

더블클릭: 브라우저에서 대시보드 열기

호버 툴팁: "StockVision - 🟢 정상 | 엔진: 가동 | 규칙: 5 | 체결: 3"

우클릭 메뉴:
  ┌──────────────────────────┐
  │ StockVision v1.0         │  (비활성)
  │ 🟢 정상 운영 중            │  (상태 텍스트)
  ├──────────────────────────┤
  │ 대시보드 열기              │  → webbrowser.open(frontend)
  │ 엔진 시작/중지             │  → 토글
  ├──────────────────────────┤
  │ 오늘 체결: 3건             │  (비활성)
  │ 활성 규칙: 5개             │  (비활성)
  ├──────────────────────────┤
  │ 종료                      │
  └──────────────────────────┘

트레이 = 상태 확인 + 최소 제어
브라우저 = 전략 설정, 로그, 관리 등 모든 복잡한 UI
```

## 8. 통신 프로토콜

### 8.1 프론트엔드 → 클라우드 서버 (HTTPS)

```
POST /api/v1/auth/register      회원가입
POST /api/v1/auth/login         로그인 → JWT + Refresh Token
POST /api/v1/auth/refresh       토큰 갱신
GET  /api/v1/rules              내 전략 규칙 조회
POST /api/v1/rules              규칙 생성
PUT  /api/v1/rules/:id          규칙 수정
DEL  /api/v1/rules/:id          규칙 삭제
GET  /api/v1/stocks/search?q=   종목 검색 (메타데이터만)
GET  /api/v1/stocks/:symbol     종목 상세 메타
GET  /api/v1/watchlist          관심종목 목록
POST /api/v1/watchlist          관심종목 등록
DEL  /api/v1/watchlist/:symbol  관심종목 해제
GET  /api/v1/admin/*            어드민 (role=admin)
```

### 8.2 프론트엔드 → 로컬 서버 (localhost)

```
POST /api/auth/token            JWT + Refresh Token 전달 (로그인 직후)
POST /api/config/kiwoom         API Key 등록
GET  /api/config                설정 조회
PATCH /api/config               설정 변경
POST /api/rules/sync            규칙 동기화 (규칙 저장 직후)
GET  /api/status                서버 + 키움 + 엔진 상태
GET  /api/logs                  체결 로그 조회
GET  /api/watchlist             관심종목 캐시 조회 (오프라인 fallback)
GET  /api/stocks/search?q=      종목 검색 (로컬 캐시, 오프라인 fallback)
POST /api/strategy/start        전략 엔진 시작
POST /api/strategy/stop         전략 엔진 중지
POST /api/strategy/kill         Kill Switch 활성화
POST /api/strategy/unlock       Kill Switch 해제
WS   /ws                       실시간 시세 + 체결 이벤트
```

### 8.3 로컬 서버 → 클라우드 서버 (HTTPS, JWT 인증)

```
POST /api/v1/heartbeat          하트비트 (30초~1분)
                                → 응답: { rules_version, context_version,
                                          watchlist_version, stock_master_version,
                                          latest_version, min_version }
                                → 버전 다르면 아래 API로 fetch
GET  /api/v1/rules              규칙 fetch (버전 변경 시)
PUT  /api/v1/rules/:id          규칙 sync (로컬 변경 시)
GET  /api/v1/context            AI 컨텍스트 fetch (버전 변경 시)
GET  /api/v1/watchlist          관심종목 fetch (버전 변경 시)
GET  /api/v1/stocks/master      종목 마스터 fetch (버전 변경 시)
GET  /api/v1/templates          전략 템플릿 목록 fetch
POST /api/v1/auth/refresh       JWT 자동 갱신
```

> WS 없음 — 하트비트 응답의 버전 비교로 변경 감지. 실시간 필요 없는 구간.

## 13. Phase 1-2 기술 상세 (기존)

> Phase 1-2에서 구축한 백엔드 기술 상세는 여전히 유효함.
> 클라우드 서버 (`backend/`)의 기반 기술.

### 데이터/도메인 모델

- Stock(id, symbol, name, exchange, sector)
- Bar(OHLCV, date)
- IndicatorSnapshot(rsi_14, ema, macd...)
- Signal(type, score)
- Prediction(y_hat, confidence, model_type)
- Account, Order, Fill, Position, Trade (가상 거래)
- BacktestRun(strategy, params, metrics)

### ML 모델

- 베이스라인: RandomForest/LogisticRegression
- 특징: 과거 N일 집계 (수익률, 변동성, 지표)
- 추론 주기: 일 1회 (장 마감 후)
- 데이터 누수 방지: TimeSeriesSplit

### 지표/신호

- SMA/EMA/RSI/MACD/BBANDS
- 신호 스코어: [-1,1] 범위, 히스테리시스 적용

### 가상 거래 엔진

- 체결: Market on Close, Limit 종가 기준
- 수수료: max(고정, 비율*체결금액)
- 리스크: 일손실 한도, 종목 편중, 연속 손실 제한

### 성능 예산

- API P95 < 200ms
- 백테스트 1전략·1종목·5년 < 5s
- 일별 배치 < 10m

## 14. 약관 요구사항

### 이용약관

- 서비스 성격: 시스템매매 도구 (투자자문/일임 아님)
- 투자 손실 면책: 사용자 규칙에 의한 매매 결과는 사용자 책임
- 커뮤니티 데이터 활용: 공유된 전략/수익 데이터는 서비스 개선, 통계, AI 학습에 활용 가능
- 서비스 중단: 클라우드 서버 장애 시 로컬 서버는 캐시 기반 독립 동작

### 개인정보처리방침

- 수집 항목: 이메일 (최소 수집)
- 미수집: 금융정보, 거래내역, 계좌정보
- 로컬 저장 데이터는 수집/전송 대상 아님
- 하트비트: 익명 UUID + 버전 (개인정보 아님)

### 키움 약관 준수

- 제3자 위임 금지: API Key는 사용자 PC에만 저장
- 제5조③ 시세 중계 금지: 서비스 키 시세는 내부 분석용, 유저 재배포 안 함
- 유저 시세는 각자 키움 키로 직접 수신

## 15. 리스크 및 대응

### 15.1 개발 범위

7개 Unit, 3개 프로세스로 범위가 넓다. MVP를 최소한으로 좁혀 동작하는 버전을 먼저 내놓는 것이 중요.
전부 완성 후 출시보다 **핵심 루프(규칙 정의 → 평가 → 주문)** 우선.

### 15.2 키움 REST API 의존

키움이 API 정책을 변경하면 핵심 기능이 영향받는다.
통제 밖 리스크이므로, BrokerAdapter 인터페이스로 추상화하여 교체 가능하게 설계.
(현재 Unit 1에서 추상화 완료)

### 15.3 실매매 안전장치 검증

Kill Switch, 손실 제한, 가격 검증 등 설계는 되어 있지만,
실전에서 버그 하나가 실제 금전 손실로 이어진다.
**모의투자 충분히 돌린 후** 실매매 전환 필수.
