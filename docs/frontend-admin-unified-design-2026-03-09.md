# 프론트엔드/어드민 통합 설계

작성일: 2026-03-09
범위: 사용자 프론트엔드, 관리자 콘솔, 클라우드/로컬 브리지 계약 정렬
대상: 기획, 프론트엔드, 클라우드 서버, 로컬 서버

## 1. 현재 개발 상태

### 이미 있는 것

- `frontend/src` 기준으로 React SPA가 이미 존재한다.
- 사용자 영역에는 인증, 대시보드, 전략, 로그, 설정 화면이 있다.
- 관리자 영역 관련 페이지와 서비스 코드도 이미 일부 구현되어 있다.
- 클라우드 서버는 아래 관리자 API를 이미 제공한다.
  - `GET /api/v1/admin/stats`
  - `GET /api/v1/admin/users`
  - `PATCH /api/v1/admin/users/{id}`
  - `GET|POST|DELETE /api/v1/admin/service-keys`
  - `GET|POST|PUT|DELETE /api/v1/admin/templates`
  - `GET /api/v1/admin/collector-status`
- 로컬 서버는 아래 브리지 API를 이미 제공한다.
  - 인증 토큰 전달
  - 상태 조회
  - 설정 조회/수정
  - 규칙 동기화
  - 로그 조회
  - WebSocket

### 지금 문제가 되는 것

- 어드민 UI가 두 갈래로 나뉘어 있다.
  - `frontend/src/pages/AdminDashboard.tsx`
  - `frontend/src/pages/Admin/*` 와 `frontend/src/pages/Admin/index.tsx`
- 실제 라우팅은 아직 단일 페이지 어드민만 연결되어 있어서, 중첩형 어드민 구조는 사실상 죽어 있다.
- 사용자 프론트도 두 방향이 섞여 있다.
  - 예전 방식의 다중 페이지 앱
  - 최근 스펙의 단일 트레이딩 워크벤치

### 확인된 계약 불일치

- 어드민 프론트가 기대하는 API와 클라우드 서버 실제 API가 다르다.
  - 프론트 기대값: `/stats/connections`, `/data/status`, `/errors`
  - 서버 실제값: `/stats`, `/collector-status`
- 로컬 WebSocket 인증이 프론트에 연결되지 않았다.
  - 서버는 `sec=<local_secret>` 쿼리 파라미터를 요구한다.
  - 프론트는 그냥 `ws://.../ws` 로 붙고 있다.
- 로컬 브로커 키 등록 요청 형식이 다르다.
  - 프론트는 `app_key`, `app_secret`만 보낸다.
  - 서버는 `app_key`, `app_secret`, `account_no`를 요구한다.
- 로컬 로그 응답 형식이 다르다.
  - 프론트는 배열을 기대한다.
  - 서버는 `{ items, total, limit, offset }` 구조를 반환한다.
- 프론트 빌드 검증은 현재 작업트리에서 바로 할 수 없다.
  - `frontend/node_modules`가 없고 `tsc`가 없어 `npm run build`가 실패한다.

## 2. 제품 방향

### 사용자 프론트

사용자 프론트는 메뉴 중심 앱보다 "트레이딩 워크벤치"에 가깝게 정리하는 것이 맞다.
핵심 작업은 아래 네 가지다.

1. 지금 거래 가능한 상태인지 확인한다.
2. 관심 종목과 체결 이벤트를 본다.
3. 전략/규칙을 만들고 관리한다.
4. 로컬 브리지와 브로커 설정을 조정한다.

### 어드민

어드민은 사용자 화면의 확장판이 아니라 운영 콘솔이어야 한다.
핵심 작업은 아래 네 가지다.

1. 서비스 상태를 본다.
2. 사용자/규칙 활동을 집계 수준에서 본다.
3. 템플릿과 서비스 키를 관리한다.
4. 수집 상태와 장애를 확인한다.

어드민에는 개인 계좌 잔고, 개인 체결 내역, 원문 자격증명 같은 민감 데이터가 노출되면 안 된다.

## 3. 제안 정보구조

### 사용자 라우트

사용자 앱의 주요 라우트는 최소화한다.

- `/`
  - 기본 트레이딩 워크벤치
- `/logs`
  - 실행 로그와 문제 추적
- `/settings`
  - 로컬 브리지, 브로커, 프로필 설정
- `/login`
- `/register`
- `/forgot-password`
- `/reset-password`
- `/admin/*`
  - 운영자 전용 별도 셸

### 사용자 워크벤치 레이아웃

`/` 하나를 중심으로 패널형 구조로 통합한다.

- 상단 바
  - 제품 아이덴티티
  - 단일 거래 가능 상태 표시
  - 종목 검색
  - 알림
  - 사용자 메뉴
- 좌측 메인
  - 관심 종목 보드와 실시간 시세
  - 체결 피드
- 중앙 메인
  - 선택 종목 상세
  - 해당 종목의 규칙 요약
- 우측 메인
  - 규칙 빌더 드로어 또는 패널
  - 시장 컨텍스트
  - 엔진 제어

이 구조로 가면 지금 흩어진 대시보드, 종목 상세, 전략 생성 화면을 하나의 작업 흐름으로 묶을 수 있다.

### 어드민 라우트

- `/admin`
  - 개요
- `/admin/users`
  - 사용자 목록과 활동 집계
- `/admin/data`
  - 수집기 상태와 데이터 신선도
- `/admin/templates`
  - 전략 템플릿 관리
- `/admin/service-keys`
  - 클라우드 서비스 키 관리
- `/admin/errors`
  - 오류 피드와 장애 확인

단, 실제 API가 없는 화면은 라우팅만 만들지 말고 보류하는 것이 맞다.

## 4. 사용자 프론트 설계

### A. 상태등은 하나로 합친다

현재 3색 상태등을 각각 따로 보여주기보다, 헤더에는 "지금 거래 가능한가"를 하나의 상태로 보여주는 편이 낫다.

- 초록
  - 로컬 브리지 연결됨
  - 브로커 연결됨
  - 거래 가능
- 노랑
  - 로컬 브리지는 연결됐지만 브로커가 끊겼거나 킬 스위치가 켜짐
- 빨강
  - 로컬 브리지 자체가 연결되지 않음

세부 진단은 펼침 정보로 보여준다.

- 클라우드 인증 상태
- 로컬 브리지 상태
- 브로커 연결 상태
- 킬 스위치
- 엔진 실행 상태

### B. 관심 종목 중심 화면

좌측 메인 영역은 관심 종목 보드가 기준이 된다.

각 행은 최소한 아래를 보여준다.

- 종목코드
- 종목명
- 현재가
- 등락률
- 연결된 규칙 수
- 최근 체결 시각
- 연결 상태 기반 표시

행 선택 시 중앙 상세 패널만 바뀌고, 페이지 이동은 최대한 줄인다.

### C. 종목 상세 패널

중앙 패널은 현재 선택된 한 종목의 작업 컨텍스트다.

표시 항목:

- 최신 가격과 당일 변화
- 현재 백엔드/로컬 캐시가 지원하는 주요 지표
- 연결된 규칙 목록
- 규칙 추가 CTA
- 관심 종목 추가/제거

### D. 규칙 빌더

규칙 빌더는 별도 메인 페이지보다 드로어 또는 우측 패널로 붙이는 것이 낫다.

저장 모델은 클라우드 서버가 이미 이해하는 필드만 기준으로 삼는다.

- `script`
- `execution`
- `trigger_policy`
- `is_active`

즉, UI는 조건 편집기처럼 보일 수 있어도 저장은 DSL 기반 규칙 모델을 표준으로 맞춘다.

### E. 로그와 설정은 분리 유지

`/logs`, `/settings`는 사용 빈도가 낮고 정보 밀도가 높으므로 별도 페이지 유지가 적절하다.

## 5. 어드민 설계

### A. 개요 화면

개요 화면은 "지금 서비스가 건강한가"를 빠르게 답해야 한다.

상단 KPI:

- 전체 사용자 수
- 활성 사용자 수
- 활성 규칙 수
- 최근 1시간 오류 수

중단 상태 영역:

- 수집기 상태
- 마지막 시세 수집 시각
- 활성 클라이언트/하트비트 수
- 템플릿 수

하단 이벤트 영역:

- 최근 오류
- 최근 운영 이벤트

### B. 사용자 화면

이 페이지는 감시가 아니라 운영 집계 목적이다.

표시 항목:

- 이메일
- 역할
- 활성/비활성
- 가입일
- 마지막 로그인
- 규칙 수

보이면 안 되는 것:

- 계좌 잔고
- 개인 체결 내역
- 브로커 키

### C. 데이터 화면

이 페이지는 클라우드 수집기 상태와 1:1로 대응시킨다.

표시 항목:

- 수집기 상태
- 마지막 시세 적재 시각
- 적재 건수
- 마지막 오류
- 소스별 신선도

### D. 템플릿 화면

현재 백엔드 계약상 가장 실제적인 어드민 CRUD는 템플릿 관리다.

표시 항목:

- 템플릿명
- 카테고리
- 공개 여부
- 생성자
- 수정 시각

백엔드가 아직 주지 않는 사용량, 수익률 같은 값은 억지로 붙이지 않는다.

### E. 서비스 키 화면

운영자 전용 화면이다.

표시 항목:

- 마스킹된 키 식별 정보
- 앱 이름
- 활성 상태
- 생성일
- 마지막 사용 시각

### F. 오류 화면

클라우드 서버에 실제 오류 조회 API가 생기기 전까지는 계획 상태로 두고, 완전한 라우트로 연결하지 않는 편이 맞다.

## 6. UI 작업 전에 맞춰야 할 계약

### 클라우드 어드민 계약

둘 중 하나를 선택해야 한다.

옵션 A: 프론트가 기대하는 누락 API를 서버에 추가한다.

- `GET /api/v1/admin/stats/connections`
- `GET /api/v1/admin/data/status`
- `GET /api/v1/admin/errors`

옵션 B: 프론트를 현재 서버 계약에 맞춘다.

- `GET /api/v1/admin/stats` 유지
- 데이터 화면은 `GET /api/v1/admin/collector-status` 사용
- 연결 통계/오류 피드는 서버 지원 전까지 보류

권장안은 옵션 B다. 지금 단계에서는 더 작고 안전하다.

### 로컬 브리지 계약

사용자 워크벤치를 제대로 만들려면 아래가 먼저 맞아야 한다.

- WebSocket 연결 시 `local_secret` 포함
- 브로커 키 등록 폼에 `account_no` 포함
- 로그 UI가 `data.items`를 읽도록 수정
- 설정 PATCH는 `{ updates: ... }` 형식으로 맞춤

## 7. 구현 우선순위

### 1단계: 계약부터 맞춘다

- 어드민 라우팅 구조를 하나로 정리한다.
- 프론트 서비스 함수들을 실제 클라우드/로컬 API에 맞춘다.
- 로컬 인증 이후 받은 `local_secret`로 WebSocket 인증을 연결한다.
- 설정/로그 요청과 응답 형식을 고친다.

### 2단계: 사용자 워크벤치를 만든다

- `/`를 진짜 메인 작업 화면으로 승격한다.
- 전략 생성은 탑레벨 페이지가 아니라 컨텍스트 패널로 내린다.
- 관심 종목, 종목 상세, 체결 피드를 한 흐름으로 묶는다.

### 3단계: 어드민 콘솔을 정리한다

- `/admin/*`를 중첩형 셸로 연결한다.
- 실제 API가 있는 화면만 먼저 살린다.
- 개요, 사용자, 템플릿, 서비스 키, 데이터 화면 순으로 정리한다.

### 4단계: 관측 기능을 확장한다

- 오류 피드
- 연결 추이 차트
- 더 자세한 수집/소스 상태 진단

## 8. 명시적 결정

- 사용자 프론트의 정식 경험은 `/` 기반 단일 트레이딩 워크벤치다.
- 어드민의 정식 경험은 `/admin/*` 기반 운영 콘솔이다.
- 규칙의 기준 데이터는 클라우드 서버다.
- 실행 상태와 거래 가능 여부의 기준 데이터는 로컬 브리지다.
- 어드민 범위는 운영 데이터까지만 허용하고, 개인 금융 데이터는 제외한다.
- 단기적으로는 백엔드 API를 더 늘리기보다 프론트 기대치를 실제 계약에 맞추는 쪽이 우선이다.

## 9. 바로 이어서 할 작업

1. `/admin/*` 라우팅을 중첩형 어드민 셸 기준으로 교체한다.
2. `frontend/src/services/admin.ts` 와 `frontend/src/services/localClient.ts` 를 실제 API 계약에 맞게 수정한다.
3. `/` 대시보드를 단일 워크벤치 구조로 재편한다.
4. 클라우드 어드민 API와 로컬 브리지 API에 대한 간단한 계약 테스트 목록을 만든다.
