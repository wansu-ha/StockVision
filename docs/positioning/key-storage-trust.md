# StockVision 키 저장 신뢰 설계

작성일: 2026-03-11
상태: 초안
관련 문서:
- `docs/architecture.md`
- `docs/product/product-direction-log.md`

## 1. 목적

이 문서는 StockVision에서 증권사 키, LLM 키, Refresh Token 같은 민감한 자격 증명을
어디에 어떻게 저장할지, 그리고 그 선택을 사용자에게 어떻게 설명할지를 정리한다.

핵심 질문은 다음과 같다.

- 왜 서버가 아니라 사용자 PC의 Windows 보안 저장소를 기본 선택으로 두는가
- Windows 보안 저장소는 어디까지 안전한가
- 같은 PC에서 실행 중인 다른 프로그램이 접근할 수 있는가
- 별도 `매직키 암호화`보다 왜 OS 보안 저장소가 더 적절한가
- 이 구조를 사용자에게 어떤 문장으로 설명해야 신뢰를 얻을 수 있는가

## 2. 결론 한 줄

`StockVision은 민감한 키를 서버에 저장하지 않고, 사용자 PC의 Windows 보안 저장소에 보관한다.`

다만 이 선택은 `절대 안전`을 의미하지 않는다.
의미는 `대규모 서버 유출 위험을 줄이고, 평문 파일 저장보다 훨씬 나은 기본값을 택한다`는 데 있다.

## 3. 무엇을 어디에 저장하는가

### Windows 보안 저장소에 둘 것

- 증권사 API Key / Secret
- 사용자 LLM API Key
- 클라우드 Refresh Token

### 로컬 파일에 둘 수 있는 것

- 일반 설정값
- 규칙 캐시
- 관심종목 캐시
- 종목 메타 캐시
- 동기화 큐

### 로컬 DB에 둘 것

- 실행 로그
- 체결 로그
- 오류 로그

### 클라우드에 둘 것

- 사용자 계정
- 규칙 원본
- 관심종목 원본
- 동기화/버전 정보
- 공통 브리핑/컨텍스트 재료

### 클라우드에 두지 않을 것

- 증권사 API Key / Secret
- 사용자 LLM API Key
- 개인 잔고/체결 상세 같은 민감 금융정보

## 4. Windows 보안 저장소란 무엇인가

이 문서에서 말하는 `Windows 보안 저장소`는 주로 아래 계층을 통칭한다.

- Windows Credential Manager
- Windows Vault
- DPAPI(Data Protection API)

앱 입장에서의 동작은 단순하다.

1. 앱이 민감한 값을 직접 평문 파일에 쓰지 않는다.
2. Windows가 제공하는 자격 증명 저장소 API에 값을 맡긴다.
3. Windows가 현재 사용자 맥락에 묶어 값을 보호한다.
4. 앱은 필요할 때만 OS API를 통해 값을 읽는다.
5. 읽어온 값은 실행 순간 로컬 서버 메모리에만 잠시 존재한다.

즉, 앱이 직접 암호키를 만들고 숨기는 대신,
OS가 관리하는 저장소와 암호화 메커니즘을 이용하는 구조다.

## 5. 왜 이 선택이 괜찮은가

### 1. 평문 파일 저장보다 훨씬 낫다

- `config.json`, `.env`, 백업 폴더, 로그 파일에 키가 섞이는 실수를 줄일 수 있다.
- 개발자가 별도 암호키를 코드나 설정에 숨길 필요가 없다.

### 2. 서버 대량 유출 위험을 줄인다

- 서버가 사용자 증권사 키와 LLM 키를 보관하지 않으면,
  서버 침해가 곧 전체 사용자 키 유출로 이어지지 않는다.

### 3. 제품 메시지가 선명하다

- `키를 서버에 저장하지 않습니다`
- `키는 이 PC의 Windows 보안 저장소에 보관됩니다`

이 두 문장은 금융/AI 제품에서 신뢰를 만드는 데 유리하다.

### 4. 자동 로그인과 복구 UX에 유리하다

- Refresh Token을 안전한 저장소에 두면 PC 재부팅 후 자동 복구가 가능하다.
- 사용자가 매번 키를 다시 입력하지 않아도 된다.

### 5. 현재 아키텍처와도 일치한다

`docs/architecture.md`는 이미 Credential Manager + DPAPI 방향으로 설계되어 있다.
즉, 구조와 문서와 제품 메시지가 서로 충돌하지 않는다.

## 6. 어디까지 안전한가

중요한 점은 `꽤 괜찮은 기본값`이지 `절대 안전`이 아니라는 것이다.

### 잘 막는 것

- 평문 파일 유출
- 서버 침해로 인한 대량 키 유출
- 다른 PC나 다른 Windows 계정에서의 단순 파일 복사 공격
- 개발 실수로 인한 노출

### 잘 못 막는 것

- 같은 PC의 악성코드
- 같은 Windows 사용자 권한으로 실행 중인 악성 프로세스
- 키를 실제로 꺼내 쓰는 순간의 메모리 탈취
- 화면 가로채기, 입력 탈취, 세션 탈취

즉,
Windows 보안 저장소는 `다른 곳에서 훔쳐가기 어렵게` 만드는 기술이지,
`감염된 같은 사용자 세션 안의 공격자도 완전히 막는 기술`은 아니다.

## 7. 같은 PC의 다른 프로그램도 가져다 쓸 수 있는가

이 질문에는 정직하게 답해야 한다.

### 짧은 답

- `아무 프로그램이나 무조건`은 아니다.
- 하지만 `같은 Windows 사용자 권한으로 실행되는 악성 프로그램`은 현실적인 위협이다.

### 의미

- Windows 보안 저장소는 사용자 맥락 기반 보호에 강하다.
- 그러나 사용자가 이미 로그인한 세션 안에서 돌아가는 악성코드를 완전히 막아주지는 못한다.
- 따라서 이 저장소는 `로컬 저장의 정석`이지 `감염 호스트 절대 방패`는 아니다.

## 8. 왜 서버가 내려주는 매직키 암호화는 비추천인가

가끔 다음과 같은 절충안이 떠오를 수 있다.

- 서버가 항상 같은 `매직키`를 내려준다.
- 로컬은 그 키로 민감값을 암호화해 저장한다.
- 키 자체는 메모리에만 둔다.

겉보기에는 안전해 보이지만, 핵심 보안 설계로는 약하다.

### 이유

- 항상 같은 문자열이면 사실상 공통키다.
- 서버가 그 키를 안다면 서버도 복호화 가능성의 일부가 된다.
- 로컬이 실제로 복호화해야 하므로, 실행 시 메모리 노출 문제는 여전히 남는다.
- 파일만 훔쳐가는 공격에는 약간 도움이 될 수 있어도,
  신뢰 메시지와 근본 보호 구조 측면에서는 이점이 작다.

### 결론

이 방식은 `강한 보안`보다 `약한 난독화`에 가깝다.
StockVision의 신뢰 포지셔닝과도 잘 맞지 않는다.

## 9. StockVision 기준 권장 구조

### 기본 원칙

- 민감한 키는 서버 미보관
- 민감한 키는 평문 파일 미보관
- 민감한 키는 Windows 보안 저장소 보관
- 실행 시에만 로컬 서버 메모리에서 사용

### 권장 흐름

1. 사용자가 웹 또는 프론트에서 브로커 키를 입력한다.
2. 프론트가 localhost 로컬 서버에 1회 전달한다.
3. 로컬 서버가 Windows 보안 저장소에 저장한다.
4. 이후 주문/LLM 호출이 필요할 때만 로컬 서버가 키를 읽는다.
5. 클라우드에는 키를 절대 올리지 않는다.

### 저장 흐름 요약

- 브로커 키: Windows 보안 저장소
- LLM 키: Windows 보안 저장소
- Refresh Token: Windows 보안 저장소
- 규칙 캐시: 로컬 파일
- 실행 로그: 로컬 SQLite
- 규칙 원본/동기화 정보: 클라우드

## 10. 이 구조에서 반드시 함께 가야 하는 보완책

Windows 보안 저장소만으로 신뢰가 완성되지는 않는다.
권한과 피해 제한 설계가 같이 필요하다.

### 제품 보완책

- 일일 손실 한도 강제
- 종목당 한도 강제
- 총 주문 금액 한도 강제
- 긴급 정지는 어디서나 쉽게
- 실행 재개는 더 어렵게
- 원격 실주문은 매우 좁게 또는 초기 금지
- 외부 주문 감지
- 세션 로그와 이상 징후 기록
- 키 삭제 / 재발급 / 재로그인 UX 단순화

### 고급 사용자 안내

- 전용 PC 사용 권장
- 최소 권한 사용 권장
- 보안 소프트웨어 사용 권장
- 장기간 방치된 세션 최소화

## 11. 사용자에게 어떻게 말할 것인가

### 좋은 문장

- `키는 서버가 아니라 이 PC의 Windows 보안 저장소에 보관됩니다.`
- `평문 파일로 저장하지 않습니다.`
- `서버에 맡기지 않아 대규모 유출 위험을 줄입니다.`
- `다만 PC가 악성코드에 감염된 경우까지 완전히 막을 수는 없습니다.`
- `그래서 손실 한도, 긴급 정지, 재발급 같은 보호 장치를 함께 제공합니다.`

### 피해야 할 문장

- `키를 PC에 저장합니다.`
- `로컬이라 절대 안전합니다.`
- `우리 구조라면 해킹당해도 괜찮습니다.`
- `다른 프로그램은 절대 접근할 수 없습니다.`

## 12. 초보와 전문가에게 각각 어떻게 받아들여지는가

### 초보 사용자

초보는 저장 메커니즘의 세부 기술보다 아래 문장을 원한다.

- `서버에 저장하지 않는다`
- `위험하면 멈출 수 있다`
- `잘못 눌러도 바로 사고 나지 않는다`

### 전문가 사용자

전문가는 아래를 확인하고 싶어한다.

- 실제 저장 위치가 어디인가
- Credential Manager / DPAPI를 정말 쓰는가
- 평문 파일이 남지 않는가
- 같은 사용자 세션 공격은 어떻게 보는가
- 피해 제한 장치가 하단에서 강제되는가

즉,
초보에게는 안심을,
전문가에게는 투명성을 줘야 한다.

## 13. 한 줄 결론

StockVision에서 Windows 보안 저장소는 `완벽한 방패`가 아니라
`민감한 키를 서버와 평문 파일에서 떼어내는 가장 현실적이고 설득력 있는 기본값`이다.

따라서 제품 약속은 `절대 안전`이 아니라,
`서버 대량 유출 위험을 줄이고, 로컬 악성코드 리스크는 권한 제한과 빠른 중지/복구로 완화한다`가 되어야 한다.
