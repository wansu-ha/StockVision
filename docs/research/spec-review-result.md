**등급: 🟢 충분**

| 항목 | 평가 |
|------|------|
| 목표 명확성 | 🟢 명확. COM/HTS/32bit 제거 → REST/WS 전환 명시 |
| 요구사항 구체성 | 🟢 F1~F10 전부 테스트 가능한 형태. 우선순위 구분 적절 |
| 수용 기준 검증성 | 🟢 "모의투자 시장가 매수 10주 → 주문번호 수신" 등 구체적 |
| 범위 명확성 | 🟢 포함/미포함 명확. Unit 2,3,5,6과 경계 정의 |
| 미결 사항 블로킹 여부 | 🟡 부분적 |

**문제점:**
1. **미결 사항 중 블로킹 이슈 1건**: "키움 REST API 정확한 엔드포인트 URL 확인" — 현재 spec에 `openapi.koreainvestment.com`이라는 URL이 이미 기재되어 있으나, 한국투자증권 API인지 키움증권 REST API인지 혼동될 소지 있음. 키움증권이 2025년 공개한 REST API와 한국투자증권 오픈API(`openapi.koreainvestment.com`)는 **별개 서비스**일 가능성이 높으므로, 정확한 URL/문서 확인 전 구현 불가.
2. **WebSocket 바이너리 vs 텍스트 프레임**: 파싱 로직에 영향. 구현 초기에 확인 필요하나 블로킹은 아님.
3. **토큰 갱신 실패 시 알림**: 트레이 알림(Unit 2)으로 자연스럽게 처리 가능. 비블로킹.

**권장 수정:**
- 미결 사항 1번을 해결한 후 spec URL 확정 필요 (실제 키움 REST API 공식 문서 URL 확인)
- `quote.py`에 `get_order_history` (주문 내역 조회) 메서드 추가 검토 — F10(미체결 주문 조회)과 연관

### Unit 2: 로컬 서버 코어 (`spec/local-server-core/spec.md`)

**등급: 🟢 충분**

| 항목 | 평가 |
|------|------|
| 목표 명확성 | 🟢 명확. 프로젝트 핵심 실행 환경이라는 위치 정의 |
| 요구사항 구체성 | 🟢 F1~F12 포괄적. 자동 로그인(F10), PyInstaller(F11) 포함 |
| 수용 기준 검증성 | 🟢 7개 카테고리로 세분화. "< 5초 서빙 시작" 등 측정 가능 |
| 범위 명확성 | 🟢 키움(Unit 1), 엔진(Unit 3)과 명확히 분리 |
| 미결 사항 블로킹 여부 | 🟡 부분적 |

**문제점:**
1. **FastAPI + pystray 이벤트 루프 통합**: 미결 사항이지만 구현에 직접 영향. pystray는 메인 스레드를 점유하므로 uvicorn을 별도 스레드에서 실행해야 함. 설계 방향은 §3.2에 "별도 스레드"로 암시되어 있으나 명시적 설계 필요.
2. **API 서버 WS 재연결 정책**: §7.5에 WS 연결 수용 기준은 있으나, 연결 끊김 시 재연결 전략(백오프 간격 등)이 spec에 정의되지 않음. architecture.md §11에 "1s → 5s → 30s 백오프"가 있으므로 spec에 반영 권장.
3. **로컬 서버 REST API 경로**: `/api/auth/token`, `/api/status` 등 — `/api/v1/` 접두어 없이 정의. API 서버는 `/api/v1/` 사용. 의도적 차이인지 명시 필요.

**권장 수정:**
- 미결 사항에 "FastAPI 스레드에서 uvicorn 실행, pystray 메인 스레드 점유" 설계 명시
- WS 재연결 백오프 정책을 architecture.md에서 가져와 명시
- 로컬 서버 API 경로에 버전 접두어 불포함 이유 주석 추가

### Unit 3: 전략 엔진 (`spec/strategy-engine/spec.md`)

**등급: 🟢 충분**

| 항목 | 평가 |
|------|------|
| 목표 명확성 | 🟢 명확. 법적 포지션까지 정의 (투자일임 소지 회피) |
| 요구사항 구체성 | 🟢 F1~F12 구체적. 가격 검증, 중복 방지, 한도 체크 포함 |
| 수용 기준 검증성 | 🟢 6개 카테고리. "RSI <= 30 AND price <= 50000 → True" 등 테스트 케이스 수준 |
| 범위 명확성 | 🟢 백테스팅(v2), AI 신호(v3+) 명시적 제외 |
| 미결 사항 블로킹 여부 | 🟡 부분적 |

**문제점:**
1. **`custom_formula` 안전 파서**: 미결이지만 F2 조건 평가에 `custom_formula` 타입이 언급되지 않음(§4.2 `_eval_single`에도 없음). 현재 지원 타입은 `price`, `indicator`, `volume`, `context`만 — 미결 사항을 삭제하거나 v2로 이동 권장.
2. **스케줄러 시간 범위 불일치**: §4.1에서 `hour="9-15"`로 설정하면 15:00~15:30이 누락. 키움 장 마감 시간 15:30까지 커버해야 함. `hour="9-15"`, `minute="*"` 조합은 15:59까지 실행하므로 오히려 장외 시간 포함. 정확한 cron 표현 필요.
3. **다중 규칙 동시 충족**: 미결이지만, 동일 종목에 매수+매도 규칙이 동시 충족될 경우 처리가 정의되지 않음.
4. **실시간 시세 데이터 소스**: evaluator가 `market_data`를 받는데, 이 데이터가 키움 WS에서 오는 실시간 데이터인지, 캐시된 데이터인지 명확하지 않음.

**권장 수정:**
- 스케줄러 cron을 `hour="9-15"` 대신 정확한 장 시간 범위 로직으로 수정 (15:30 이후 중지)
- `custom_formula`를 미결 사항에서 제거하고 v2 roadmap으로 이동
- 동일 종목 매수/매도 동시 충족 시 처리 규칙 추가

### Unit 4: API 서버 (`spec/api-server/spec.md`)

**등급: 🟢 충분**

| 항목 | 평가 |
|------|------|
| 목표 명확성 | 🟢 "얇은 웹 서버" + 핵심 원칙(금융정보 미보유) 명확 |
| 요구사항 구체성 | 🟢 F1~F13 포괄적. 인증~어드민까지 전범위 |
| 수용 기준 검증성 | 🟢 5개 카테고리. 인증 flow 전체 포함 |
| 범위 명확성 | 🟢 기존 spec 병합(auth, context-cloud) 관계 명확 |
| 미결 사항 블로킹 여부 | 🟡 부분적 |

**문제점:**
1. **데이터 서버 통신 방식 미확정**: "HTTP / gRPC" 미결. Unit 5와 직접 연관. 구현 시작 전 확정 필요하나, HTTP가 자연스러운 선택이므로 실질적 블로킹은 낮음.
2. **Rate Limiting 구현 미확정**: "Redis vs in-memory" — 개발 환경은 in-memory, 운영은 Redis가 표준. 환경별 분리 결정 필요.
3. **WebSocket 엔드포인트 공유**: §10.2에 "프론트엔드 WS와 로컬 서버 WS는 같은 엔드포인트 사용 가능"이라고 하나, 클라이언트 유형 구분 방식(연결 시 query param? JWT payload?)이 미정.
4. **규칙 조건 JSON 검증 스키마**: 미확정이지만, Unit 3과 Unit 6에서 동일 스키마를 사용하므로 중요. 최소한 지원 타입(`price`, `indicator`, `volume`, `context`)과 연산자(`<=`, `>=`, `==`, `!=`, `<`, `>`) 목록 확정 필요.
5. **JWT 알고리즘**: §4.2에 HS256 명시. 로컬 서버도 JWT를 검증해야 하는지, 아니면 API 서버에 위임하는지 미정. HS256이면 시크릿 키 공유 필요.

**권장 수정:**
- 데이터 서버 통신은 REST(HTTP)로 확정하고 미결에서 제거
- WS 클라이언트 유형 구분 방식 명시 (예: 연결 시 `?client_type=local` query param)
- 조건 JSON 스키마를 §5.3에서 정식 스키마로 확정 (지원 타입, 연산자 목록)
- JWT 검증 주체 명시: 로컬 서버는 JWT를 API 서버에 전달만 하고 자체 검증은 하지 않음 (또는 검증한다면 시크릿 공유 방식 정의)

### Unit 5: 데이터 서버 (`spec/data-server/spec.md`)

**등급: 🟡 보완 필요**

| 항목 | 평가 |
|------|------|
| 목표 명확성 | 🟢 명확. 내부 전용 + 시세 중계 금지 준수 |
| 요구사항 구체성 | 🟡 F1~F8이 상대적으로 추상적. "내부 API 제공"의 상세 부족 |
| 수용 기준 검증성 | 🟡 §8이 다른 spec 대비 간결. 수집 실패 복구 테스트 없음 |
| 범위 명확성 | 🟢 포함/미포함 명확 |
| 미결 사항 블로킹 여부 | 🔴 블로킹 이슈 있음 |

**문제점:**
1. **키움 클라이언트 공유 방식 미확정 (블로킹)**: "패키지 분리 vs 복사" — Unit 1이 `local_server/kiwoom/`에 위치하는데, 데이터 서버는 별도 프로세스(클라우드). 코드 공유 방식이 프로젝트 구조에 직접 영향.
2. **데이터 서버 → API 서버 통신 방식 미확정**: Unit 4의 미결과 동일. 양쪽에서 미결이므로 확정 시급.
3. **서비스 키 등록/관리 흐름 불완전**: §5에 `POST /internal/admin/service-key`가 있으나, 어드민 UI(Unit 7)에서 API 서버(Unit 4)를 경유하여 데이터 서버에 전달하는 전체 흐름이 명확하지 않음. API 서버 §8에는 `/api/v1/admin/service-keys` 엔드포인트가 있으나 데이터 서버와 어떻게 연결되는지 미정의.
4. **내부 API 인증**: `/internal/*` API에 인증이 없음. 네트워크 정책으로 차단한다고 하지만, 인증 없는 API는 보안 우려. 최소한 내부 API 토큰이나 mTLS 언급 필요.
5. **분봉 데이터 보관 정책**: 디스크 사용량 추정 없이 "1년" 보관이라고만 기재. 200종목 x 390분/일 x 250거래일 = 약 2천만 건/년 → 구체적 추정 필요.
6. **기존 yfinance 코드 재사용 범위**: `backend/app/services/data_collector.py` 코드가 데이터 서버로 이동하는지, API 서버에 남는지 미정.

**권장 수정:**
- 키움 클라이언트 공유 방식을 확정 (권장: 공유 패키지 `stockvision-kiwoom`)
- 데이터 서버 통신은 REST(HTTP) + 내부 Bearer Token으로 확정
- 서비스 키 등록 흐름을 E2E로 기술: 어드민 UI → API 서버 → 데이터 서버
- 분봉 디스크 사용량 추정 추가
- 수용 기준에 "WS 연결 끊김 시 자동 재연결" 테스트 추가
- 수용 기준에 "yfinance 보조 수집 정상 동작" 테스트 추가

### Unit 7: 어드민 페이지 (`spec/admin/spec.md`)

### 종합 평가

**전체적으로 양호한 수준의 spec 세트.** 8개 spec이 architecture.md 및 development-plan-v2.md와 높은 일관성을 유지하며, Unit 간 의존성과 개발 순서가 잘 정의되어 있음.

**즉시 해결 필요 (P0):**
1. 키움 REST API 정확한 URL/문서 확인 (Unit 1, 5에 영향)
2. 키움 클라이언트 공유 방식 확정 — 패키지 vs 복사 (Unit 1, 5)
3. 데이터 서버 ↔ API 서버 통신 프로토콜 확정 (Unit 4, 5)

**보완 권장 (P1):**
4. WS 메시지 스키마 정식 정의 (Unit 2, 4)
5. 조건 JSON 스키마 확정 (Unit 3, 4, 6)
6. 주문 결과 보고 API 설계 (architecture.md §3.1 7번)
7. Unit 3 스케줄러 장 시간 범위 수정 (15:30 정확 반영)
8. "키움 프록시" 용어를 "로컬 서버"로 통일

**가장 보완이 필요한 spec:** Unit 5 (data-server) — 블로킹 미결 사항이 가장 많고, 수용 기준과 요구사항의 상세 수준이 다른 spec 대비 부족.
