# 규칙 DSL 설계 리서치

> 작성일: 2026-03-06 | 상태: 설계 초안 (GPT/Gemini/Claude 리뷰 반영)

---

## 1. 동기

### 기존 접근 방식의 한계

| 방식 | 문제 |
|------|------|
| 폼 (드롭다운) | 간단한 조건은 쉬움. 조건 10개 넘으면 카드 지옥. `(A AND B) OR C` 표현 어려움 |
| 노드 그래프 | 단순한 조건에 과함. 모바일 UX 최악. 트레이딩 앱이 아니라 프로그래밍 도구 느낌 |
| 키움 수식관리자 | 사실상 코딩. 화면 분리 (수식관리자/조건검색/시스템트레이딩 각각 별도) |
| JSON 직접 편집 | 개발자 전용 |

### 제안: 수식 기반 DSL

엑셀 수식 수준의 간단한 표현식 언어.
코딩이 아니라 "조건을 글로 쓰는 것"에 가깝다.

**성공 조건**: 기능 추가가 아니라 **문법 축소 + 의미 명확화 + 노출 범위 절제**.

## 2. 핵심 개념

> **실행 모델상 모든 식별자는 현재 시세 컨텍스트에서 평가되는 읽기 전용 값 또는 함수다.**

- 상태 변경 없음
- 사이드이펙트 없음
- 같은 시점이면 같은 값 (참조 투명성)
- 결과는 숫자 or boolean

### 표면 문법의 구분

| 구분 | 표기 | 예시 |
|------|------|------|
| **필드** | 괄호 없음 | `현재가`, `거래량`, `수익률` |
| **함수** | 괄호 필수 | `RSI(14)`, `MA(20)`, `MACD()` |
| **커스텀 함수** | 인자 없는 함수 | `골든크로스()`, `과매도()` |

실행 모델은 전부 함수지만, 표면 문법에서는 필드/함수를 구분한다.
이게 가장 덜 헷갈린다.

## 3. 문법

### 3.1 내장 필드 (시세 데이터)

```
-- 가격
현재가              -- 평가 시점의 최신 체결가
시가                -- 당일 시가
고가                -- 당일 고가
저가                -- 당일 저가
전일종가            -- 전일 종가

-- 거래
거래량              -- 당일 누적 거래량
등락률              -- 전일 대비 등락률 (% 포인트, 3 = 3%)

-- 포지션
수익률              -- 보유 시 평균단가 대비 수익률 (% 포인트, 3 = 3%). 미보유 시 0 (포지션 없음의 편의적 표현)
보유수량            -- 보유 주식 수 (미보유 시 0)
```

> **단위 규칙**: `수익률`, `등락률`은 % 포인트 단위. `3`이 3%를 의미한다. `%` 리터럴 없음.

### 3.2 내장 함수 (지표)

```
RSI(기간)                   -- RSI (기본 timeframe: 일봉)
MA(기간)                    -- 단순이동평균 (일봉)
EMA(기간)                   -- 지수이동평균 (일봉)
MACD()                      -- MACD 라인 (12, 26, 9)
MACD_SIGNAL()               -- MACD 시그널
MACD_HIST()                 -- MACD 히스토그램
볼린저_상단(기간)            -- 볼린저밴드 상단 (일봉)
볼린저_하단(기간)            -- 볼린저밴드 하단 (일봉)
평균거래량(기간)             -- N일 평균 거래량
상향돌파(A, B)              -- 직전: A < B, 현재: A >= B
하향돌파(A, B)              -- 직전: A > B, 현재: A <= B
```

> **기본 timeframe**: 모든 지표의 기본 timeframe은 **일봉**. 분봉 지표는 v2 범위.
> `상향돌파`/`하향돌파`는 직전 분봉 데이터 필요 — BarBuilder 연동.

### 3.3 연산자

```
-- 비교: 숫자 → boolean
<=  >=  <  >  ==  !=

-- 논리: boolean → boolean
AND  OR  NOT

-- 산술: 숫자 → 숫자
+  -  *  /

-- 괄호
( )
```

#### 연산자 우선순위 (높은 순)

| 순위 | 연산자 | 결합 |
|------|--------|------|
| 1 | `()` 괄호, 함수 호출 | — |
| 2 | `NOT` | 우결합 |
| 3 | `*` `/` | 좌결합 |
| 4 | `+` `-` | 좌결합 |
| 5 | `<` `<=` `>` `>=` `==` `!=` | 좌결합, 체이닝 금지 |
| 6 | `AND` | 좌결합 |
| 7 | `OR` | 좌결합 |

> `AND`가 `OR`보다 우선. `A OR B AND C` = `A OR (B AND C)`. 헷갈리면 괄호 사용 권장.
> 비교 체이닝 금지: `10 < 현재가 < 50000`은 에러. `현재가 > 10 AND 현재가 < 50000`으로 작성.

### 3.4 타입 규칙

타입은 **숫자**와 **boolean** 2종뿐. 암묵적 변환 금지.

```
RSI(14) + 1          -- OK (숫자 + 숫자)
RSI(14) <= 30        -- OK (숫자 비교 → boolean)
RSI(14) <= 30 AND 현재가 <= 50000  -- OK (boolean AND boolean)

RSI(14) + true       -- 금지 (숫자 + boolean)
현재가 AND 거래량     -- 금지 (숫자 AND 숫자)
NOT RSI(14)          -- 금지 (NOT 숫자)
NOT (RSI(14) <= 30)  -- OK (NOT boolean)
```

### 3.5 커스텀 함수 (상수 함수 선언)

```
-- 이름() = 수식
골든크로스() = 상향돌파(MA(5), MA(20))
과매도() = RSI(14) <= 30
급등() = 등락률 >= 5 AND 거래량 > 평균거래량(20) * 2
저평가매수() = 골든크로스() AND 과매도()
```

- `=`은 선언 (재대입 불가)
- 선언 순서대로 참조 가능 (후방 참조 불가)
- 재귀 불가
- 결과 타입: 숫자 or boolean
- 인자 없음 (커스텀 함수는 항상 무인자)

> 후방 참조 불가 시 에러 메시지: "`과매도()`가 아직 선언되지 않았습니다. 위쪽에서 먼저 정의하세요."

### 3.6 매수/매도 조건

```
매수: 골든크로스() AND 과매도()
매도: RSI(14) >= 70 OR 수익률 >= 3
```

- `매수:`, `매도:` 뒤의 수식은 반드시 boolean
- **둘 다 필수** — 매도 없는 자동매매는 위험
- 각각 정확히 **1회만** 등장 (중복 선언 금지)
- 스크립트 최하단에 위치
- **미보유 시 매도 보호**: 엔진이 매도 주문 실행 전 `보유수량 > 0`을 암묵적으로 확인. DSL에 명시할 필요 없음 — 이건 실행 정책이지 조건식이 아님. (`매도: 수익률 >= 0`이라도 미보유면 실행 안 됨)

### 3.7 주석

```
-- 한 줄 주석
```

### 3.8 없는 것

- if/else
- 루프 (for, while)
- 배열, 문자열
- 가변 변수 (재대입)
- 함수 인자 (커스텀 함수는 항상 인자 없음)
- import
- `%` 리터럴
- `cross_above` / `cross_below` 중위 연산자 (→ `상향돌파()` 함수로 대체)

## 4. 필드 평가 시점 (의미론)

> **이게 문법보다 중요하다.** 같은 DSL이라도 평가 시점이 다르면 결과가 달라진다.

| 필드/함수 | 평가 기준 | 설명 |
|----------|----------|------|
| `현재가` | 최신 체결가 | 평가 시점(1분 주기)의 마지막 체결가 |
| `시가`, `고가`, `저가` | 당일 | 당일 장중 누적 |
| `전일종가` | 전일 | 전일 종가 확정값 |
| `거래량` | 당일 누적 | 장 시작~현재 누적 |
| `등락률` | 전일종가 대비 | `(현재가 - 전일종가) / 전일종가 * 100` |
| `수익률` | 평균단가 대비 | `(현재가 - 평균단가) / 평균단가 * 100` |
| `보유수량` | 실시간 | 체결 반영된 현재 보유량 |
| `RSI(N)`, `MA(N)` 등 | **일봉 기준** | N일 일봉 데이터로 계산. 당일 값은 현재가를 마지막 봉으로 포함 |
| `평균거래량(N)` | 일봉 기준 | 최근 N일의 일 거래량 평균 |
| `상향돌파(A, B)` | 직전 성공 평가 vs 현재 | 직전 성공적 평가 주기의 값과 현재 값 비교. 스킵/재접속 시에도 마지막 성공 평가 기준. 첫 평가 시 false |

> **`상향돌파`/`하향돌파` 첫 평가**: 엔진 시작 직후 또는 장 시작 첫 평가에는 직전 값이 없으므로 항상 `false`. 두 번째 평가부터 비교 가능.

### 4.1 Repainting 주의

일봉 지표(RSI, MA 등)는 당일 현재가를 마지막 봉으로 포함하므로, 장중 1분마다 값이 변한다.
오전 10시에 `상향돌파(MA(5), MA(20))`가 true였다가 10시 1분에 false로 돌아갈 수 있다.

**v1 정책**: 현재가 포함 (실시간 반응형). 사용자에게 "장중 지표는 확정값이 아닙니다" 안내.
**v2 검토**: 봉 마감(종가) 기준 평가 옵션 제공 가능.

### 4.2 결측치 (Missing Data)

데이터가 부족한 경우 (예: 상장 5일째 종목에서 `MA(20)` 호출):

- 해당 함수는 `null` 반환
- **null 전파 규칙: 전체 블록 실패**
  - 평가 트리 어디서든 `null`이 발생하면 해당 `매수:`/`매도:` 조건 **전체를 미충족(false)** 처리
  - 개별 비교 결과를 false로 변환하지 않음 (NOT 반전으로 true가 되는 버그 방지)
  - 예: `NOT (MA(20) > 50000)` — MA(20)이 null이면 NOT 결과가 true가 아니라 **전체 미충족**
- UI/프리뷰: false와 null 구분 표시 (예: `미충족(데이터 부족)` vs `미충족`)
- 에러 로그에 기록 (프리뷰: "데이터 부족: MA(20)에 20일 데이터 필요, 현재 5일")
- 봇은 중지하지 않음 — 해당 평가 주기만 스킵

### 4.3 런타임 에러

| 상황 | 처리 |
|------|------|
| 0으로 나누기 (`현재가 / 0`) | `null` → 조건 미충족(false). 프리뷰에서 경고 표시 |
| 오버플로 | 실용상 발생 불가 (주가/수량 범위). 발생 시 null 처리 |
| 시세 미수신 | 해당 평가 스킵. 다음 주기에 재시도 |

> **원칙**: 런타임 에러는 봇을 중지시키지 않는다. 해당 주기를 안전하게 스킵하고 로그에 기록.

## 5. 전체 예시

### 5.1 간단한 규칙

```
매수: RSI(14) <= 30 AND 현재가 <= 50000
매도: RSI(14) >= 70
```

### 5.2 커스텀 함수 활용

```
-- 내 전략: 골든크로스 + 과매도 진입, RSI 과매수 또는 목표수익 시 청산
골든크로스() = 상향돌파(MA(5), MA(20))
과매도() = RSI(14) <= 30
거래량확인() = 거래량 > 평균거래량(20) * 2
낙폭제한() = 등락률 >= -3

매수: 골든크로스() AND 과매도() AND 거래량확인() AND 낙폭제한()
매도: RSI(14) >= 70 OR 수익률 >= 3 OR 수익률 <= -5
```

### 5.3 단순 가격 조건

```
매수: 현재가 <= 50000
매도: 현재가 >= 55000
```

## 6. 제품 전략: 단계적 노출

> **DSL을 먼저 만들되, 곧바로 일반 사용자에게 전면 노출하지 않는다.**

### v1: 폼 UI + 내부 DSL

```
[사용자] → 폼(드롭다운/카드)으로 조건 구성
    ↓
[프론트] → 폼 입력을 DSL 텍스트로 변환
    ↓
[저장] → script 필드에 DSL 문자열 저장
    ↓
[엔진] → DSL 파싱 → 평가
```

- 사용자는 DSL을 직접 보지 않음
- 내부 표현은 처음부터 DSL로 통일
- 폼에서 표현 가능한 범위 = v1 문법의 전부

### v2: 고급 모드 — DSL 편집기 공개

```
[기본 모드] 폼 UI (기존과 동일)
    ↕ 전환 버튼
[고급 모드] DSL 에디터 (CodeMirror, 자동완성, 프리뷰)
```

- 고급 사용자만 선택적으로 사용
- 폼 ↔ DSL 양방향 변환 (단순 조건은 양방향, 복잡한 DSL은 폼으로 역변환 불가 경고)

### 장기: 폼은 DSL의 비주얼 프록시

- 폼이 만드는 건 항상 유효한 DSL
- DSL이 만드는 건 폼으로 못 돌아갈 수도 있음
- 이 비대칭은 허용 (고급 모드 사용자는 감수)

## 7. 에디터 UI (v2 고급 모드)

### 7.1 통합 화면 (화면 분리 없음)

```
+--------------------------------------------------+
| 삼성전자 (005930)          [저장] [시뮬레이션]     |
|--------------------------------------------------|
|                                                   |
|  -- 내 함수 (접기/펼치기) -----------------       |
|  | 골든크로스() = 상향돌파(MA(5), MA(20))  |       |
|  | 과매도() = RSI(14) <= 30                |       |
|  ------------------------------------------       |
|                                                   |
|  매수 조건 ------------------------------------   |
|  | 골든크로스() AND 과매도()                |       |
|  ------------------------------------------       |
|  현재: MA(5)=71800 MA(20)=70200 RSI=42.3         |
|  골든크로스=true  과매도=false  >> 미충족           |
|                                                   |
|  매도 조건 ------------------------------------   |
|  | RSI(14) >= 70 OR 수익률 >= 3            |       |
|  ------------------------------------------       |
|  현재: RSI=42.3 수익률=-  >> 미충족                |
|                                                   |
|  실행: 시장가 / 10주 / 하루 1회  [설정 변경]       |
|                                                   |
+--------------------------------------------------+
```

### 7.2 에디터 기능

| 기능 | 설명 |
|------|------|
| 자동완성 | `RS` 입력 → `RSI(기간)` 제안 |
| 구문 강조 | 필드=초록색, 함수=파란색, 연산자=회색, 숫자=주황색, 키워드=보라색 |
| 실시간 프리뷰 | 수식 아래에 현재 값 + 충족 여부 표시 |
| 에러 표시 | `RSI(14) + true` → 빨간 밑줄 + "숫자와 boolean은 더할 수 없습니다" |
| 함수 도움말 | 함수 위에 마우스 → 설명 + 현재 값 + 기본 timeframe 툴팁 |

### 7.3 구현 기술

| 컴포넌트 | 후보 |
|---------|------|
| 에디터 | CodeMirror 6 (경량, 모바일 지원) 또는 Monaco (VS Code 기반, 무거움) |
| 파서 | 직접 구현 (수식 수준이라 단순) 또는 PEG.js/Nearley |
| 구문 강조 | CodeMirror 커스텀 언어 모드 |
| 자동완성 | CodeMirror completion 확장 |

## 8. 데이터 모델

### 8.1 저장 형식

```json
{
  "script": "골든크로스() = 상향돌파(MA(5), MA(20))\n과매도() = RSI(14) <= 30\n\n매수: 골든크로스() AND 과매도()\n매도: RSI(14) >= 70 OR 수익률 >= 3",
  "execution": { "order_type": "MARKET", "qty_type": "FIXED", "qty_value": 10 },
  "trigger_policy": { "frequency": "ONCE_PER_DAY" }
}
```

- `buy_conditions`, `sell_conditions` → `script` 단일 필드로 통합
- `execution`, `trigger_policy`는 DSL 밖 — 조건은 평가식, 실행은 설정. 성격이 다르다
- 파서가 script에서 `매수:`, `매도:` 블록을 추출하여 평가

### 8.2 하위 호환

JSON 조건 배열도 계속 지원:
- `script` 필드가 있으면 DSL로 평가
- `buy_conditions`/`sell_conditions`가 있으면 기존 방식으로 평가
- 폼 UI는 항상 DSL로 저장 (내부적으로)

## 9. 파서 구현 난이도

| 항목 | 난이도 | 비고 |
|------|--------|------|
| 토크나이저 (어휘 분석) | 낮음 | 한국어 식별자 + 연산자 + 숫자 |
| AST 파서 (구문 분석) | 낮음~중간 | 연산자 우선순위, 괄호, 함수 호출 |
| 타입 체크 | 낮음 | 숫자/boolean 2종, 암묵적 변환 금지 |
| 평가기 (인터프리터) | 낮음 | AST 트리워킹, 재귀 없음 |
| 에러 메시지 | 중간~높음 | 한국어 에러 메시지, 위치 표시, 제안형 수정 |
| 에디터 통합 | 중간 | CodeMirror 커스텀 언어 모드 (v2) |

파서 자체는 1~2주. 다만 진짜 비용은 파서가 아니라 **에러 메시지 품질, 에디터 UX, 테스트 케이스, 사용자 문의 대응**이다. 언어를 만든다는 건 파서를 만드는 게 아니라 **UX 계약을 만드는 것**이다.

> **에러 복구 난이도 주의**: `현재가 AND 거래량` 같은 타입 에러에서 "혹시 `현재가 >= ... AND 거래량 >= ...`을 의미하셨나요?" 수준의 제안형 수정을 제공하려면 파서가 문맥을 추론해야 한다. 이 수준의 에러 복구는 단순 파서보다 구현 비용이 높으므로, v1에서는 명확한 에러 메시지까지만, 제안형 수정은 v2에서 점진적으로 추가.

## 10. LLM 연동 (자연어 → DSL 번역)

### 10.1 법적 근거

> **자본시장법 제6조 제6항**: "투자자문업"이란 금융투자상품의 가치 또는 금융투자상품에 대한
> 투자판단(종류, 종목, 취득·처분, 취득·처분의 방법·수량·가격 및 시기 등에 대한 판단)에 관한
> 자문에 응하는 것을 영업으로 하는 것.

**핵심 구분**: LLM이 "어떤 종목을, 언제, 얼마에, 얼마나" 판단하면 투자자문. **문법 변환**은 투자판단이 아니라 **코드 변환 도구**.

### 10.2 운영 원칙

#### 허용 (코드 변환 도구)

| 기능 | 예시 | 근거 |
|------|------|------|
| 자연어 → DSL 번역 | "RSI 30 이하면 매수" → `매수: RSI(14) <= 30` | 사용자가 조건 결정, LLM은 문법 변환만 |
| 문법 오류 수정 | `매수: RSI(14) <= AND` → `AND 뒤에 조건이 필요합니다` | 구문 교정 |
| 함수 도움말 | "`RSI(N)`: N일 기준 상대강도지수, 0~100 범위" | 사전적 설명 |
| DSL 구문 해설 | "이 조건은 RSI가 30 이하이고 현재가가 5만원 이하일 때 매수합니다" | 사용자가 작성한 코드를 읽어주는 것 |

#### 차단 (투자자문)

| 기능 | 예시 | 근거 |
|------|------|------|
| 파라미터 추천 | "RSI 몇으로 설정하면 좋을까?" | 투자판단 자문 (취득 방법) |
| 전략 추천 | "이 종목에 맞는 전략 추천해줘" | 투자판단 자문 (종류, 종목) |
| 성과 기반 암시 | "이 전략 백테스팅 결과 좋네요" | 투자판단 자문 (가치 판단) |
| 종목 추천 | "지금 뭐 사면 좋을까요?" | 투자판단 자문 (종목, 시기) |
| 비교 우위 판단 | "A 전략이 B보다 낫습니다" | 투자판단 자문 (방법) |

#### 경계선: "설명"의 함정

일반적인 지표 설명도 맥락에 따라 추천으로 읽힐 수 있다:

- "RSI 30 이하는 **일반적으로** 과매도 구간입니다" → 위험 (매수 암시)
- "RSI(N)은 N일간 상승/하락 비율입니다. 0~100 범위" → 안전 (사전적 설명)

**원칙**: 값의 해석(좋다/나쁘다/과매도/과매수)을 포함하면 차단. 계산 방법과 범위만 설명.

### 10.3 제품 수준 안전장치

LLM 차단은 정책 문서만으로 불충분. **제품/UI 레벨에서 강제**:

| 장치 | 설명 |
|------|------|
| 출력 형식 제한 | LLM 응답은 **구조화된 JSON만** 허용 (DSL 코드 + 에러 메시지). 자유 형식 텍스트 차단 |
| 프롬프트 가드레일 | 시스템 프롬프트에 금지 영역 명시 + 출력 스키마 강제 |
| 입력 필터 | "추천", "뭐 사면", "좋을까" 등 투자자문 키워드 감지 → LLM 호출 전 차단 |
| 응답 검증 | LLM 응답이 JSON 스키마를 벗어나면 사용자에게 전달하지 않음 |
| UI 범위 제한 | LLM 입력란은 DSL 에디터 내부에만 존재. 별도 "AI 상담" 채팅 UI 금지 |

#### LLM 응답 스키마 (예시)

```json
{
  "type": "translation",
  "dsl": "매수: RSI(14) <= 30 AND 현재가 <= 50000\n매도: RSI(14) >= 70",
  "errors": [],
  "warnings": ["기간이 명시되지 않아 RSI 기본값 14를 사용했습니다"]
}
```

- `type`: `translation` | `correction` | `explanation` (고정 enum)
- `dsl`: 생성/수정된 DSL 코드 (없으면 null)
- `errors`: 파싱 불가 사유
- `warnings`: 가정/기본값 사용 알림
- **자유 텍스트 필드 없음** → "~하면 좋겠네요" 류 출력 구조적 불가

### 10.4 architecture.md 원칙 준수

> "LLM은 데이터 제공자 역할만. '매수하세요' 직접 조언 → 투자자문업 → 금지."

이 원칙의 DSL 적용:
- LLM은 **사용자가 말한 조건을 DSL로 변환**하는 번역기
- 조건 자체를 **제안하거나 평가하지 않음**
- 사용자의 입력이 없으면 LLM도 출력 없음 (능동적 추천 금지)

식별자() = 수식

매도: 불리언식

-- 연산
+ - * /                   (숫자 → 숫자)
< <= > >= == !=           (숫자 → boolean)

( )                       (그룹)

-- 주석
-- 텍스트

-- 없는 것
if/else, 루프, 배열, 문자열, 재대입, 함수 인자,
import, % 리터럴, cross 중위 연산자

## 12. 미결 사항

- [ ] 한국어/영어 혼용 정책: 글로벌 표준 지표는 영어(`RSI`, `MACD`, `MA`), 자체 데이터/액션은 한국어(`현재가`, `상향돌파`) — 국내 HTS 관행과 동일. 최종 확정 필요
- [ ] 내장 함수 이름 최종 확정 (한국어 우선? 영어 병기? alias?)
- [ ] 에러 메시지 한국어 가이드라인 — 개발자 관점("Type mismatch") 아닌 비서 관점("매수 조건의 결과는 '참/거짓'이어야 합니다. 현재 수식은 '숫자'를 계산하고 있습니다")
- [ ] 폼 → DSL 변환 시 포맷 규칙 (줄바꿈, 들여쓰기)
- [ ] DSL → 폼 역변환 가능 범위 정의 — 커스텀 함수 또는 3단+ 괄호 중첩 시 "고급 수식 모드 적용 중" 배지 표시 + 폼 편집 비활성화(read-only). 무리한 역변환 시도 금지
- [ ] 분봉 지표 지원 시점 (v2) 및 문법 확장 방향
- [ ] v2 봉 마감 기준 평가 옵션 (§4.1 repainting 대응)
- [ ] v2 커스텀 함수 인자 도입 검토 — `골든크로스(단기, 장기) = 상향돌파(MA(단기), MA(장기))` 형태. 인자는 **숫자 리터럴만 허용** (변수/함수 인자 전달 금지 → 파서 복잡도 억제). v1은 무인자로 시작
- [ ] 알림 전용 모드 — 매매 실행 없이 조건 충족 시 푸시 알림만. 문법은 변경하지 않고(`매수:`/`매도:` 필수 유지), 제품 레이어에서 `매도: false` 자동 삽입 등으로 흡수
- [ ] 커스텀 함수 공유 범위 — script 안에 인라인 vs 사용자별 라이브러리 분리. 커뮤니티 공유(v2) 시 함수만 추출 가능한 구조 필요
- [ ] trigger_policy 상태의 프리뷰 표시 — 조건 충족됐지만 `ONCE_PER_DAY`로 이미 실행된 경우 "오늘 이미 실행됨 (하루 1회 제한)" 표시. 사용자 혼란 방지
- [ ] 에러 메시지 카탈로그 — 파서 구현 전에 에러 메시지 목록 먼저 설계. 에러 메시지가 문법을 역제약하는 경우 있음
