# E. 백테스트 시점 검토

## E-1. 재활용 가능한 모듈

| 모듈 | 위치 | 백테스트 역할 |
|------|------|--------------|
| RuleEvaluator | `local_server/engine/evaluator.py` | 동일 DSL로 과거 바 평가 |
| IndicatorProvider | `local_server/engine/indicator_provider.py` | yfinance 일봉 + 지표 계산 |
| DailyBar 모델 | `cloud_server/models/market.py` | OHLCV 히스토리컬 바 |
| DSL evaluate | `sv_core/parsing/` | 실전과 동일한 평가 함수 |

핵심: `RuleEvaluator._eval_dsl()`은 `market_data` dict를 받을 뿐. DailyBar를 순회하며 동일 함수를 호출하면 백테스트. DSL 일관성 자동 보장.

## E-2. 구현 위치 비교

| 옵션 | 장점 | 단점 | 적합도 |
|------|------|------|--------|
| **클라우드 서버** | 가입만으로 체험, DB 직접 접근 | 서버 부하 | **권장** |
| 프론트 (JS) | 서버 부하 없음 | DSL 평가기 JS 재구현 필요 → 일관성 깨짐 | 비권장 |
| 로컬 서버 | evaluator 그대로 재사용 | 설치 필요 → 온보딩 목적 위배 | 해당 없음 |

## E-3. 간이 백테스트 최소 범위

```
입력: rule_id + 기간 (3개월/6개월/1년)
처리: DailyBar 순회 → evaluator 호출 → 가상 매수/매도
출력:
  - 수익률 곡선 (일별 포트폴리오 가치)
  - 매수/매도 포인트 (날짜 + 가격)
  - 요약: 총 수익률, MDD, 승률, 거래 횟수
```

제외 (Phase D): 복수 종목, 슬리피지/수수료, 파라미터 최적화, 전종목 스크리닝

## E-4. 필요한 신규 작업

- `BacktestRunner`: DailyBar 시간순 순회 + evaluator + 가상 포지션
- 포지션 모델: 현금, 보유 수량, 평균 단가, 실현/미실현 손익
- 결과 집계: 수익률 곡선, MDD, 승률, 매매 이벤트
- API: POST `/api/v1/backtest/run`
- 프론트 UI: 기간 선택 + 결과 차트 (Recharts 기반)

구현 난이도: ~3-4일 (evaluator 재활용)

## E-5. 타이밍 결론

**권장: Phase C 후반부 선택적 포함 (필수 아님)**

- Phase C 핵심 = "안심" (운영 패널, 킬스위치, 원격 조회)
- 백테스트는 Phase E ("전략 수명주기") 핵심 구성요소
- 단, 온보딩 임팩트가 크면 Phase C 막바지에 간이 버전 투입 타당
- Phase C를 백테스트 없이 출시해도 핵심 가치 충족

## StockVision 차별점 vs 열위점

### 차별점 (경쟁우위)
- **로컬 실행 → 투자일임 회피** — 법적으로 가장 안전
- **DSL 표현력** — AND/OR, cross 연산자, trigger_policy → 젠포트/알파스퀘어 대비 높음
- **증권사 교체 가능** — BrokerAdapter ABC, 특정 증권사 종속 없음
- **오픈소스** — 포트폴리오/확장성 가치

### 열위점 (개선 과제)
- **로컬 설치 필수** → 온보딩 장벽 가장 높음 (트레이딩뷰/알파스퀘어 = 2단계)
- **백테스트 미구현** → 젠포트는 전략+백테스트 한 세트
- **"가입만으로 의미 있는 체험" 부재** → 클라우드만 접속해도 볼 콘텐츠 필요

## 핵심 인사이트

경쟁사 대비 가장 큰 격차: **"증권사 없이 체험 가능한 범위"**

클라우드 기능(규칙 생성 + 백테스트 확인)을 로컬 설치 전에 체험 가능하게 하는 것이 온보딩 전환율의 핵심 과제.
