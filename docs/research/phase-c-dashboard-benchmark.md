# 트레이딩 대시보드 벤치마킹

## 1. 플랫폼별 대시보드 구성

### 1-1. 한국 플랫폼

#### 키움증권 영웅문 / 한국투자증권 eFriend (HTS 표준)
- **계좌 요약**: 예수금, 총평가금액, 총매입금액, 총평가손익, 수익률
- **포지션 테이블**: 종목명, 보유수량, 매입단가, 현재가, 평가손익, 수익률%, 평가금액
- **주문 내역**: 미체결/체결 구분, 주문수량, 체결수량, 주문가, 체결가, 상태
- **호가창**: 10단계 매도/매수 호가, 잔량, 체결량
- **체결 내역**: 시간순 체결 로그
- **조건검색/자동주문**: 조건 → 주문 연동, 잔고/미체결/체결 탭

#### 젠포트 (Genport)
- 전략 백테스트 결과 (승률, 수익률, MDD, 샤프비율)
- 전략별 실시간 운용 현황 (활성/비활성)
- 매매 신호 발생 로그
- 포지션 현황 (보유 종목, 평균단가, 현재손익)
- 전략 성과 비교 차트

#### 알파스퀘어 (AlphaSquare)
- 실시간 주식시세 및 차트
- 종목 발굴/분석 통합 뷰
- 매매 신호 알림 (프리미엄)
- 시스템매매 규칙 설정 및 운용 현황

### 1-2. 해외 플랫폼

#### TradingView
- **차트**: 최대 16개 동시, 캔들/바/라인/Heikin Ashi/Renko 등
- **워치리스트**: 실시간 시세, 등락률, 거래량
- **알림**: 가격/지표/시간 기반 조건 알림
- **트레이딩 패널** (브로커 연동 시):
  - 포지션: 종목, 수량, 진입가, 현재가, P&L, 미실현손익
  - 주문: 활성 주문, 상태, 주문가
  - 체결 내역: 시간, 종목, 가격, 수량
  - 계좌 요약: 잔고, 마진, 구매력
- **스크리너**: 조건 필터링 종목 탐색
- **경제 캘린더**: 예정 이벤트, 실적 발표

#### QuantConnect (라이브 트레이딩 대시보드)
- **런타임 배너**: Equity, Fees, Holdings, Net Profit, PSR, Return, Unrealized, Volume
- **차트**: 자산곡선(equity curve), 낙폭(drawdown), 벤치마크 대비, 롱/숏 익스포저, 턴오버, 마진 사용량
- **포지션 탭**: 종목, 평균단가, 수량, 시가총액, 미실현손익
- **주문 탭**: 제출/체결/부분체결/취소/만료 이벤트 + CSV 내보내기
- **운영 모니터링**: 실시간 로그, 오류 알림, 자동 재시작 상태, 알고리즘 중지/청산 컨트롤

#### Alpaca
- **계좌 메트릭**: equity, portfolio_value, cash, buying_power, unrealized_pl, accrued_fees
- **포지션**: 종목, 수량, 평균매입가, 현재가, 미실현손익률
- **주문**: 주문 ID, 상태(new/filled/canceled/expired), 체결수량, 체결평균가, 타임스탬프
- **포트폴리오 히스토리**: 시계열 자산 가치 추이

#### Interactive Brokers TWS
- **계좌 창**: 잔고, 증거금 요건, 매매 가능 자금, FX 포지션
- **포트폴리오 창**: 기초자산별 정렬, 포지션별 시가
- **주문 관리**: 즉시 주문, 주문 모니터, 일별 체결 리포트
- **리스크 내비게이터**: 전체 포트폴리오 위험, "What-If" 시나리오, 베타 분석, 옵션 그릭스
- **실시간 P&L**: 거래 화면 + 계좌 창에서 실시간 손익

## 3. 공통 패턴

### 3-1. 정보 계층 (모든 플랫폼 공통)
```
1. 계좌 요약 (Account Summary)      ← 최상단, 항상 표시
2. 포지션 현황 (Positions)           ← 실시간
3. 주문 관리 (Orders)                ← 미체결/체결 탭
4. 성과 차트 (Performance Chart)     ← 자산곡선 중심
5. 리스크 지표 (Risk Metrics)        ← MDD, 드로다운
6. 시스템 상태 (System Health)       ← 연결/오류
7. 알림 (Alerts/Notifications)       ← 조건 트리거
```

### 3-2. 실시간 vs 주기적 데이터

| 구분 | 항목 | 전달 방식 |
|------|------|----------|
| 실시간 | 현재가, 포지션 P&L, 주문 상태, 시스템 연결 | WebSocket |
| 주기적 | 계좌 잔고, 성과 통계, 드로다운 | 폴링 (분~시간) |
| 일별 집계 | 일일 실현 손익, 거래 횟수, 수수료 합계 | 장 마감 후 |

### 3-3. 알림 패턴

| 유형 | 트리거 | 심각도 |
|------|--------|--------|
| 체결 알림 | 주문 체결 완료 | info/success |
| 주문 실패 | API 오류, 잔고 부족 | error |
| 손실 한도 | 일/총 손실 초과 | critical |
| 시스템 오류 | 연결 끊김, 엔진 오류 | critical |
| 가격 알림 | 지정 조건 도달 | info |

### 3-4. 한국 플랫폼 특유

- **호가창**: 10단계 매도/매수 호가 + 잔량 (필수 UX)
- **체결강도**: 매수/매도 체결 비율
- **프로그램 매매 동향**: 외국인/기관/개인 수급
- **수수료율**: 거래당 수수료 명시 (한국 투자자 민감)
- **공시/뉴스**: 종목별 공시 알림 실시간 연동

## 5. 얄코 × 키움 자동매매 프로그램 분석

> 출처: [키움영웅전 유튜브](https://youtu.be/_BAdI9THvYc) (2025.11.28, 13만회, 좋아요 2.3천)
> 출연: 키움증권 김미 대리, 얄코(얄팍한 코딩 사전), 이재혁 주임(키움 REST API)

### 5-1. 프로그램 구조

```
[사용자 PC]
  Python 스크립트
    ├─ 키움 REST API (조건검색 + 주문)
    ├─ 키움 WebSocket (실시간 시세)
    └─ 텔레그램 봇 (원격 제어 + 알림)
```

- 비개발자 대상: 노션에서 코드 복사/붙여넣기 → VS Code에서 실행
- "한 번 세팅하면 끝" — 이후 껐다 켰다만

### 5-2. 핵심 기능 (자막 기반)

| 기능 | 설명 |
|------|------|
| 조건검색 → 자동매수 | HTS 조건검색식에 종목 편입 시 자동 매수 |
| 익절/손절 자동 | 수익률 % 기반 자동 매도 |
| 텔레그램 원격 제어 | `start`, `stop`, `help`, 설정 변경, 조건식 전환 |
| 텔레그램 알림 | 매수 완료, 익절 완료, 손절 완료 실시간 톡 |
| 리포트 | `report` 입력 → 보유종목 + 수익률 + 총손익 |
| 설정 실시간 변경 | 장중에 익절%/손절%/투자금 텔레그램으로 변경 |

### 5-3. StockVision 시사점

| 인사이트 | 얄코 | StockVision 현재 | 개선 |
|---------|------|-----------------|------|
| 모바일 제어 | 텔레그램 = 리모컨 | 없음 (웹 대시보드 PC 의존) | 텔레그램/카톡 봇 or 모바일 웹 |
| 알림 채널 | 텔레그램 톡 (즉각적) | WS 토스트 (브라우저 열어야) | 외부 알림 채널 추가 |
| 리포트 요청 | "report" → 보유종목+손익 | 미구현 (daily P&L 없음) | `/api/logs/daily-summary` |
| 설정 원격 변경 | 텔레그램으로 즉시 변경 | 설정 페이지 (로컬 접속 필요) | 원격 설정 변경 API |
| "한 번 세팅" | 1회 셋업 후 켜기만 | 매번 uvicorn 실행 | exe 패키징 + 자동시작 |
| 복붙 온보딩 | 노션 코드 복사/붙여넣기 | 온보딩 위저드 | 더 쉬운 설치 경험 |

**핵심 발견**: 트레이더는 PC 앞에 앉아있지 않다. "폰이 리모컨" 패러다임이 핵심.
StockVision의 웹 대시보드는 PC 의존적 — 모바일 원격 제어+알림 채널이 경쟁력 격차.

### 5-4. 법적 분석

- 이 영상은 **키움증권 공식 콘텐츠** (준법감시인 심사필 제25-02778호)
- 개인이 본인용 자동매매 프로그램 → 법적 제한 없음
- 타인 배포/판매 시 → 투자자문업 등록 이슈 (자본시장법)
- StockVision은 "사용자가 직접 조건 설정" + 오픈소스 → 도구 제공이므로 상대적 안전
- 다만 "종목 추천" 기능 추가 시 법적 리스크 증가 → 주의 필요

### 5-5. Gemini 종합 인사이트 (4가지)

> Gemini 요약 + 자체 분석 통합

#### 인사이트 1: 메신저 리모컨 UX — "친숙한 채널이 대시보드"

복잡한 HTS나 모바일 앱 없이 텔레그램 챗봇이 대시보드+리모컨 역할.
직장인/외출 중 투자자에게 핵심 편의성.

**StockVision 적용 방향:**
- 웹 대시보드는 "분석/설정/리뷰" 용 → PC
- 메신저 봇은 "알림/제어/리포트" 용 → 모바일
- 역할 분리: 복잡한 UI를 웹에 욱여넣지 말고, 핵심 액션은 봇으로 분리
- 후보: 텔레그램 봇 (개발 용이, 해외 사용자 호환) or 카카오톡 알림톡

#### 인사이트 2: 전략(조건식)과 실행(봇)의 분리 — "프로그램은 추천하지 않는다"

프로그램 자체는 "실행자/매니저" 역할만. 종목 추천 X [03:00].
성패는 전적으로 사용자의 조건검색식 퀄리티 [13:45].

**StockVision 차별화 포인트:**
- 현재 구조도 동일 (사용자가 규칙 생성 → 엔진이 실행)
- **차별화 기회**: AI가 "좋은 조건"을 제안 or 백테스트로 검증
- 법적으로도 "조건 도구 제공" ≠ "투자자문" → 안전한 포지셔닝
- Phase C AI 분석: Claude가 시장 컨텍스트 분석 → 조건 제안 (제안만, 실행은 사용자)

#### 인사이트 3: 실시간 익절/손절 커스터마이징 — "유연한 대응력"

장중에 텔레그램으로 익절선/손절선 즉각 조정 [12:48].
기존 HTS 자동감시주문의 "설정 후 방치" 한계를 극복.

**StockVision 적용 방향:**
- 현재: 규칙 수정 = DetailView → cloudRules.update → localRules.sync (PC 접속 필요)
- 개선: 원격 규칙 파라미터 변경 API (익절%/손절%/수량 등 핫 파라미터)
- **한 단계 더**: AI가 장세 감지 → "변동성 급등, 익절 라인 조정할까요?" 제안
  → 사용자 버튼 클릭으로 승인 (Human-in-the-loop)
- Kill Switch/Loss Lock은 이미 엔진에 있음 → 원격 트리거만 추가하면 됨

#### 인사이트 4: 증권사 독점 데이터 활용 — "대중 심리 지표"

키움의 "실시간 조회 순위 (0198)" = 대중 수급 지표 [05:37].
트래픽이 많을수록 유의미한 데이터.

**StockVision 적용 방향:**
- 키움 REST API의 `0198` (실시간 조회 순위) 활용 가능 → 모멘텀 지표
- 거래대금 급증 현황, 프로그램 매매 동향 등 키움 독점 데이터
- 기술적 지표(RSI/MACD) + 수급 데이터 융합 → 단기 트레이더에게 매력적
- 주의: 키움 전용 기능 → 다른 증권사 사용자에겐 미제공 (BrokerAdapter 추상화 고려)

### 5-6. 종합 — 얄코 프로그램 vs StockVision 포지셔닝

| 축 | 얄코 프로그램 | StockVision |
|----|-------------|-------------|
| 타겟 | 코딩 모르는 개인 트레이더 | 시스템매매 입문자 |
| 전략 소스 | HTS 조건검색식 (사용자 제작) | 규칙 빌더 + AI 제안 (향후) |
| UI | 텔레그램 봇 (모바일 퍼스트) | 웹 대시보드 (PC 퍼스트) |
| 제어 | 텔레그램 명령어 | 웹 버튼 + 원격 제어 (계획) |
| 알림 | 텔레그램 톡 (즉각) | WS 토스트 (브라우저 필요) |
| 설치 | 코드 복붙 + Python 실행 | 로컬 서버 + exe (계획) |
| 리포트 | "report" → 보유종목+수익률 | 미구현 |
| 확장성 | Python 코드 수정 | 오픈소스 + API 확장 |
| 법적 | 키움증권 공식 (심사필) | 도구 제공 (오픈소스) |

**StockVision이 이겨야 하는 곳:**
1. **AI 분석** — 얄코는 조건검색식을 "사용자가 직접" 만들어야 함. StockVision은 AI가 도울 수 있음
2. **백테스트** — 조건식의 과거 성과 검증. 얄코는 없음
3. **웹 기반 분석** — 차트, 지표, 재무 데이터 통합 뷰. 텔레그램으로는 불가능
4. **다중 증권사** — BrokerAdapter로 KIS/키움 모두 지원

**StockVision이 배워야 하는 곳:**
1. **모바일 알림 채널** — 텔레그램/카톡 봇 연동 (가장 시급)
2. **원격 제어** — start/stop/설정 변경을 모바일에서
3. **리포트 요청** — "오늘 결과 보여줘" 한마디로 요약 수신
4. **"한 번 세팅" 경험** — exe 패키징 + 자동시작으로 진입장벽 낮추기

## 6. "어디서" 보여줄 것인가 — 정보 배치 분석

트레이더의 관심은 상황(context)에 따라 다르다. 주요 시나리오별로 "어떤 정보를, 어디서" 보고 싶어하는지 정리.

### 6-1. 시나리오별 정보 요구

#### 시나리오 A: 아침 출근 — "오늘 상태 확인" (10초 스캔)
- **필요**: 엔진 돌고 있나? 어제 결과? 오늘 장 상태?
- **현재**: OpsPanel 4도트 + 경고 배너 → **어제 결과 없음**
- **원하는 위치**: 대시보드 진입 즉시 — OpsPanel 또는 바로 아래

#### 시나리오 B: 장중 모니터링 — "문제 없나?" (30초 주기)
- **필요**: 체결 발생? 오류? 현재 손익? 엔진 정상?
- **현재**: 체결 내역 목록 + WS 토스트 → **일일 손익 합산 없음**
- **원하는 위치**: 한 화면에서 스크롤 없이

#### 시나리오 C: 체결 알림 수신 — "잘 된 건가?" (즉시 확인)
- **필요**: 어떤 규칙? 어떤 종목? 체결가? 슬리피지?
- **현재**: WS 토스트 (짧은 메시지) → **상세 정보 부족**
- **원하는 위치**: 토스트 클릭 → 해당 종목 DetailView

#### 시나리오 D: 장 마감 후 — "오늘 리뷰" (5분 분석)
- **필요**: 오늘 수익률, 거래 횟수, 전략별 성과, 자산곡선
- **현재**: 체결 내역 목록만 → **집계/요약 없음**
- **원하는 위치**: 별도 "오늘의 리포트" 또는 DetailView 확장

#### 시나리오 E: 위험 상황 — "뭔가 잘못됐다!" (즉시 대응)
- **필요**: Kill Switch 상태, 손실 규모, 남은 포지션, 복구 방법
- **현재**: Kill Switch/Loss Lock 배너 (방금 추가) → **복구 액션 없음**
- **원하는 위치**: OpsPanel 긴급 배너 + 인라인 복구 버튼

### 6-2. 위치별 정보 배치 제안

#### OpsPanel (항상 보이는 최상단)
```
현재: [로컬●] [브로커●] [클라우드●] [엔진●]     신호 N  체결 N  오류 N
```

## 7. StockVision 강점/약점 종합 분석

벤치마킹 플랫폼(HTS, 젠포트, TradingView, QuantConnect), 경쟁 프로그램(얄코 자동매매), Gemini 분석을 종합한 포지셔닝 평가.

### 7-1. 강점 (Strengths)

| # | 항목 | 설명 | 경쟁 대비 |
|---|------|------|-----------|
| S1 | **AI 분석 통합** | Claude API 기반 종목 분석·규칙 제안. 조건식을 사용자가 직접 만들어야 하는 경쟁 도구와 차별화 | 얄코=없음, 젠포트=없음, HTS=없음 |
| S2 | **웹 기반 통합 뷰** | 차트(5종 타입) + 지표 + 재무 + 규칙 + 실행로그가 한 화면 | 얄코=텔레그램(텍스트), HTS=다창, TradingView=차트 중심 |
| S3 | **다중 증권사** | BrokerAdapter ABC로 KIS/키움 모두 지원. 어댑터만 추가하면 증권사 확장 가능 | 얄코=키움 전용, 젠포트=키움 전용 |
| S4 | **규칙 카드 UX** | 조건/실행/리스크/결과 4블록 시각화. 전략을 코딩 없이 GUI로 설정 | 얄코=코드 직접 수정, QuantConnect=코드 전용 |
| S5 | **3프로세스 아키텍처** | Cloud(인증·AI) + Local(엔진·주문) + Frontend(SPA) 분리. 보안과 확장성 양립 | 얄코=단일 스크립트, 젠포트=모노리스 |
| S6 | **리스크 관리 내장** | Kill Switch, 일일 손실 한도, Loss Lock 등 안전장치 시스템 레벨 내장 | 얄코=사용자가 코딩, HTS=일부 |
| S7 | **오픈소스 + 도구 제공** | 법적으로 "투자자문"이 아닌 "도구 제공" 포지션. 사용자가 직접 전략 설정 | 얄코=개인용(비공개), 젠포트=유료 SaaS |
| S8 | **실시간 WS 파이프라인** | WS 시세 → BarBuilder → Indicator → Evaluator → Executor. REST 폴백 포함 | 얄코=WS 사용, 젠포트=서버사이드 |
| S9 | **차트 이벤트 마커** | BUY/SELL/FAIL 체결 이벤트를 차트 위에 직접 표시 | 얄코=없음, HTS=별도 화면 |
| S10 | **백테스트 기반 구조** | Evaluator 재사용으로 백테스트 확장 가능 (구조는 있음) | 얄코=없음, 젠포트=핵심 기능 |

### 7-2. 약점 (Weaknesses)

| # | 항목 | 심각도 | 난이도 | 설명 | 경쟁 대비 |
|---|------|--------|--------|------|-----------|
| W1 | **모바일 알림 없음** | ★★★ | 중 | 체결/오류/긴급 상황을 PC 웹에서만 확인 가능. 트레이더는 "장중에 PC 앞에 없다" | 얄코=텔레그램 알림+원격제어 |
| W2 | **일일 P&L API 없음** | ★★★ | 하 | 오늘 실현손익 합산 API 미구현. OpsPanel에 가장 필요한 정보 | HTS=기본, QuantConnect=기본 |
| W3 | **설치 진입장벽** | ★★★ | 중 | Python 환경 + 서버 2개 기동 필요. 비개발자 사용 사실상 불가 | 얄코=Python 실행이지만 단일파일, 젠포트=웹 |
| W4 | **원격 제어 없음** | ★★ | 중 | 엔진 시작/정지, 설정 변경을 웹 대시보드에서만 가능. 모바일 원격 제어 미지원 | 얄코=텔레그램 명령어 |
| W5 | **백테스트 미구현** | ★★ | 상 | 구조(Evaluator)는 있지만 실제 백테스트 UI/API 없음. 전략 신뢰도 검증 불가 | 젠포트=핵심, QuantConnect=핵심 |
| W6 | **일일 리포트 없음** | ★★ | 중 | 장 마감 후 "오늘 요약" 자동 생성 없음. 수동으로 체결 내역 확인해야 함 | 얄코="report" 명령어, QuantConnect=대시보드 |
| W7 | **실전 검증 0회** | ★★★ | - | 모의서버만 테스트. 실제 돈으로 주문/체결/슬리피지/부분체결 미검증 | 얄코=실전 운용 중 |
| W8 | **HTS 조건검색 미연동** | ★ | 상 | 증권사 독점 데이터(조건검색식, 실시간 검색순위 0198)를 활용 못 함 | 얄코=키움 조건검색 핵심 활용 |
| W9 | **exe 패키징 미완** | ★★ | 중 | spec은 있지만(`spec/local-exe-deeplink/`) 미구현. 딥링크 자동시작도 미완 | 얄코=스크립트 직접실행 |
| W10 | **리포트/통계 뷰 없음** | ★★ | 중 | 누적 수익곡선, MDD, 샤프비율, 전략별 성과 비교 화면 없음 | 젠포트=기본, QuantConnect=기본 |

```
심각도 ★★★   W2 일일P&L     W1 모바일알림     W7 실전검증
```

### 7-4. 논의 결정 사항 (2026-03-12)

#### W2 일일 P&L — Phase C Quick Win으로 착수
- 로컬 서버에 일일 요약 API 추가, OpsPanel + 계좌바에 표시

#### W1 + W4 통합 → "모바일 대응 (PWA)"
기존에 별개 항목이었던 W1(모바일 알림)과 W4(원격 제어)를 하나로 통합.

| 결정 | 내용 |
|------|------|
| 주 채널 | **PWA** — 기존 React SPA에 반응형 + Service Worker 추가 |
| 알림 | **웹 푸시 (FCM)** — Android 완벽, iOS 16.4+ 지원 (홈화면 추가 필요) |
| 원격 제어 | PWA에서 상태 조회 + 긴급 정지 (Phase C 범위) |
| 알림 경로 | 로컬 서버 → 클라우드 (하트비트 채널) → FCM → 사용자 폰 |
| 인증과 푸시 | 푸시 토큰은 JWT와 별도 — 로그아웃/만료돼도 알림 수신 가능 |
| 메신저 (텔레그램) | 당장은 안 함. iOS 푸시 불만 시 보조 채널로 검토 |
| 네이티브 앱 | 과도. 사용자 규모 커지면 그때 검토 |

**근거**: 프론트엔드 코드베이스 하나로 PC(풀 기능) + 모바일(모니터링/알림/긴급 제어) 모두 커버. 별도 앱/봇 코드베이스 불필요. 한국 Android 70%+ 점유율에서 웹 푸시 충분.
