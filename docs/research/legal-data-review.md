# 데이터 제공 법적 검토 — 논의 정리

## 배경

StockVision의 서버-로컬 구조에서 어떤 데이터를 어떻게 제공할 수 있는지에 대한 법적 검토.

## 1. 데이터 제공 자체는 합법인가?

**합법이다.** 단, "도구/데이터 제공"과 "투자자문"의 경계를 지켜야 한다.

### 자본시장법 기준

- **투자자문업**: 금융투자상품의 **투자판단**(종목, 매수/매도, 시기 등)에 관한 **자문에 응하는 것**을 영업으로 하는 것 (자본시장법 제6조 제7항)
- **도구 제공**: 판단의 주체가 사용자이고, 플랫폼은 데이터/분석 도구만 제공하는 경우 → 투자자문업 미해당

### 업계 사례

젠포트, 트레이딩뷰, 인텔리퀀트 등이 라이선스 없이 운영 가능한 이유:
- 투자판단의 주체가 사용자
- 플랫폼은 도구만 제공
- 불특정 다수에게 동일한 정보 제공

## 2. 서버 데이터 오류 시 법적 리스크

**면책 조항이 있으면 리스크 매우 낮음.**

| 상황 | 리스크 |
|------|--------|
| 외부 데이터 소스 오류/지연 | 낮음 — 원천 제공자 책임 |
| Cloud AI 점수 계산 버그 | 낮음 — 참고용 도구 |
| 의도적 데이터 조작 | **높음** — 사기죄 해당 |
| 지연 데이터를 실시간으로 표기 | 중간 — 허위 표시 |

### 필수 면책 조항

```
- 본 서비스의 데이터는 지연되거나 부정확할 수 있습니다.
- 투자 판단의 책임은 전적으로 이용자에게 있습니다.
- 데이터 오류로 인한 손실에 대해 책임지지 않습니다.
- 본 서비스는 투자 자문이 아닌 정보 제공 목적입니다.
```

## 3. 시스템 매매 vs LLM 매매 — 루트 차이

**판단 엔진은 분리, 실행 엔진은 공유.**

```
[dsl 루트]  현재 구현됨 (strategy_type: "dsl")
로컬 서버 스케줄러 → DSL 평가기 (sv_core/parsing/) → 안전장치 → BrokerAdapter.send_order()

[LLM 루트]  v2 예정, 미구현 (strategy_type: "llm")
로컬 서버 스케줄러 → 컨텍스트 fetch → 사용자 LLM 호출 → 응답 파싱 → 안전장치 → BrokerAdapter.send_order()
```

`strategy_type` 필드로 분기하며, 안전장치와 `BrokerAdapter`(주문 실행)는 양쪽 공유.

## 4. 서버 vs 로컬 역할 분리

### 원칙: 서버는 재료, 로컬은 판단

| 구간 | 서버 | 로컬 |
|------|------|------|
| 데이터 수집 | O | 보충 가능 |
| 지표 계산 | O | 서버 거 사용 |
| **분석/판단** | **X** | **O** |
| **매매 신호 생성** | **X** | **O** |

서버에서 분석까지 해서 내려보내면 "도구 제공"이 아니라 "판단 제공"으로 넘어갈 수 있음.

### 로컬 시스템 vs 로컬 LLM — 법적 차이 없음

받는 주체가 규칙 엔진이든 LLM이든 사용자 기기에서 실행되는 건 동일. 법적으로 구분 없음.

## 5. 데이터 출처별 제공 가능 여부

### 시세 데이터

| 데이터 종류 | 서버에서 재배포 | 근거 |
|------------|---------------|------|
| 실시간 시세 (호가, 체결가) | **X** | KRX 시세 재배포 계약 필요 |
| 지연 시세 (15분+) | O | 대부분 무료 |
| 일봉 OHLCV (종가 기반) | O | 자유 사용 |
| 기술적 지표 (RSI, MACD) | O | 공개 공식 계산 결과 |

**yfinance 데이터 = 지연 + 일봉 위주 → 현재 구조에서는 문제 없음.**

### 콘텐츠 데이터 (미래 확장)

| 출처 | 원본 제공 | 가공 결과 제공 | 비고 |
|------|----------|--------------|------|
| yfinance (공개 API) | O | O | 문제 없음 |
| DART OpenAPI (공시) | O | O | 공공데이터 |
| KRX 실시간 시세 스크래핑 | **X** | O (지표 변환) | 이용약관 위반 주의 |
| 증권사 API (키움 등) | **X** | O (가공) | 개인용 라이선스 |
| 뉴스 본문 크롤링 | **X** | O (감성 점수) | 저작권 |
| 커뮤니티 크롤링 | **X** | O (여론 점수) | 이용약관 |

## 6. 핵심 — 실시간 데이터를 분석해서 결과만 주면?

**된다.**

```
실시간 호가 데이터 그대로 전달        → X (시세 재배포)
실시간 데이터로 계산한 RSI=72         → O (파생 결과물)
실시간 데이터 기반 감성 점수 0.68     → O (변환된 정보)
```

KRX 시세 재배포 규정이 보호하는 건 **"시세 데이터 그 자체"**. 의미 있는 변환(지표 계산, 점수화 등)을 거친 파생 결과물은 원본이 아님.

**단, 가공이 너무 얕으면** (1초 지연시켜서 그대로 보여주기 등) 실질적 재배포로 볼 수 있음.

## 7. 분석 근거 공개 의무

**법적 의무 없음.** 실무적으로는 신뢰도 확보를 위해 데이터 출처 정도는 표시 권장.

- 알고리즘 공개: 불필요 (영업비밀)
- 데이터 출처: 권장 ("yfinance 기반 지연 데이터")
- "AI 기반 분석" 명시 (Cloud AI): 권장 (면책 강화)

## 8. 가공 vs 변조

| | 가공 | 변조 |
|--|------|------|
| 의도 | 원본을 다른 형태로 변환 | 원본을 거짓으로 바꿈 |
| 원본 훼손 | 없음 | 있음 |
| 검증 가능 | O (동일 공식 → 동일 결과) | X |
| 법적 | 정상 | 사기 |

가공 데이터가 변조로 오해받지 않으려면:
- **재현 가능성** — 계산 로직 명시 가능할 것
- **원본 대체 안 함** — "분석 지표"이지 "시세"가 아님을 명시
- **면책 조항** — "가공된 데이터이며 원본 시세가 아닙니다"

## 9. API 응답 설계 가이드라인

### OK — 서버가 내보내도 되는 것

```json
{
  "rsi": 72,
  "macd": { "line": 150, "signal": 140 },
  "sentiment_score": 0.68,
  "news_summary": "실적 호조 기사 다수",
  "volume_change_pct": 15.2,
  "bollinger_position": "upper"
}
```

```json
{
  "recommendation": "매수",
  "prediction": "상승 확률 85%",
  "action": "buy",
  "target_price": 85000
}
```

## 10. StockVision 현재 구조 평가

| 항목 | 현재 상태 | 법적 안전성 |
|------|----------|------------|
| 데이터 소스 (yfinance) | 지연/일봉 데이터 | 안전 |
| 기술적 지표 제공 | RSI, MACD 등 수치 | 안전 |
| Cloud AI 분석 점수 | 서버에서 계산 | 프레이밍 주의 필요 |
| 판단 주체 | 사용자 (로컬 LLM/규칙) | 안전 |
| 면책 조항 | 미구현 | **구현 필요** |

## 결론 — 행동 항목

1. **면책 조항 UI 구현** — 서비스 하단/이용약관에 필수 문구 추가
2. **API 응답 프레이밍 검수** — "추천/예측" 표현 제거, "분석/참고 지표"로 통일
3. **데이터 출처 표시** — "yfinance 기반 지연 데이터" 명시
4. **yfinance 표기** — "실시간"이 아닌 "지연 데이터"임을 명확히
5. **미래 확장 시** — 뉴스/커뮤니티 데이터는 원본 아닌 점수만 외부 제공
6. **로컬 수집** — 사용자 개인 용도의 실시간 수집은 자유, 서버 재배포만 주의
