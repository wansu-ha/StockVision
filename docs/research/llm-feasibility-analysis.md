## 0. 아키텍처 전제

이 분석은 StockVision의 **3프로세스 아키텍처**를 전제로 한다.

```
[프론트엔드]  정적 호스팅, 순수 UI
     ↕
[클라우드 서버]  인증, 규칙 CRUD, 시세 수집, Cloud AI 컨텍스트 생성
     ↕
[로컬 서버]  사용자 PC — 전략 엔진 + 증권사 주문 + 로그 (핵심)
```

### 0.1 세 가지 모듈

시스템에는 데이터 가공·매매 판단에 관여하는 모듈이 **세 개** 있다.

#### A. Cloud AI 분석 모듈 — 데이터 가공 (판단 아님)

| | |
|--|--|
| 위치 | 클라우드 서버 |
| API 키 | 우리 키 (`ANTHROPIC_API_KEY`) |
| 모으는 정보 | 시세 (OHLCV), 뉴스, 외부 데이터 |
| 하는 일 | LLM으로 원본 데이터를 가공 → 감성 점수, 뉴스 요약, 리스크 지표 |
| 산출물 | **Cloud AI 컨텍스트** (가공된 데이터 포인트) |
| 법적 성격 | 데이터 가공 도구 — RSI 계산과 본질적으로 동일 |

매매 판단을 내리지 않는다. 산출물은 아래 B·C의 입력 재료가 된다.

#### B. 전략 엔진 (DSL 규칙) — 규칙 기반 매매 판단

| | |
|--|--|
| 위치 | 로컬 서버 (사용자 PC) |
| LLM 사용 | **안 씀** |
| 모으는 정보 | 클라우드 컨텍스트 (지표 + A의 산출물), 브로커 실시간 시세, 포트폴리오 상태 |
| 하는 일 | DSL 규칙 평가 → 안전장치 검증 → 주문 실행 |
| 산출물 | **매매 신호 → 주문** |
| strategy_type | `"dsl"` |

#### C. Custom LLM 통합 (이 spec) — LLM 기반 매매 판단

| | |
|--|--|
| 위치 | 로컬 서버 (사용자 PC) |
| API 키 | 사용자 키 (Credential Manager) |
| 모으는 정보 | 클라우드 컨텍스트 (지표 + A의 산출물), 브로커 실시간 시세, 포트폴리오 상태 |
| 하는 일 | 프롬프트 조립 → 사용자 LLM 호출 → 응답 파싱 → 안전장치 검증 → 주문 실행 |
| 산출물 | **매매 신호 → 주문** |
| strategy_type | `"llm"` |

### 0.2 모듈 간 관계

```
[A] Cloud AI        →  컨텍스트 생산 (데이터 가공, 판단 아님)
                          ↓ fetch
                    ┌─────┴─────┐
[B] 전략 엔진 (DSL)  │           │  [C] Custom LLM
    규칙 평가 → 주문  │  같은 데이터 │  LLM 평가 → 주문
    LLM 안 씀        │  다른 엔진  │  사용자 키
                    └───────────┘
                          ↓
                      안전장치 (한도 초과 시 주문 차단)
                          ↓
                      BrokerAdapter → 증권사 주문
```

- B와 C는 **같은 데이터를 받아서 다른 엔진으로 판단**하는 병렬 옵션
- 사용자가 규칙별로 `strategy_type`을 선택 (`"dsl"` 또는 `"llm"`)
- 안전장치와 주문 실행 경로는 B·C 공유
- **우리 서버(클라우드)는 매매 판단에 관여하지 않는다** — A는 데이터 가공만

## 1. 가능한 것 / 불가능한 것

### 1.1 한 장 요약

| | 우리가 해도 되는 것 | 하면 안 되는 것 |
|--|---|---|
| **클라우드 데이터** | 지표, 감성 점수, 뉴스 요약 제공 | "매수 추천", "상승 확률 85%" |
| **프롬프트** | 템플릿 구조·변수 바인딩 제공 | — (프롬프트 자체는 도구) |
| **LLM 호출** | — (로컬 서버가 사용자 키로 호출) | 클라우드에서 매매 판단용 LLM 호출 |
| **전략 코드** | 전략 프레임워크·파서·안전장치 제공 | "RSI 30 이하면 매수" 같은 판단 로직 기본 제공 |
| **실행** | 주문 래퍼, 드라이런, 킬스위치 | 사용자 동의 없는 자동 실행 |

### 1.2 상세: 가능한 것

#### A. 프롬프트 템플릿 제공 — 합법

프롬프트는 **도구의 설계**에 해당한다. 망치의 모양을 설계하는 것이지, 대신 못을 박아주는 게 아니다.

```
가능한 것:
✅ 프롬프트 템플릿 구조 제공 (변수 바인딩, 응답 스키마)
✅ 템플릿 버전 관리 & 백테스트 연결
✅ 응답 JSON 스키마 정의 & 파싱
✅ 프롬프트 작성 가이드 (변수 사용법, 포맷 설명)
```

#### B. 클라우드 데이터 가공 & 컨텍스트 빌드 — 합법

```
가능한 것:
✅ OHLCV → JSON/Markdown 자동 변환
✅ 기술적 지표 (RSI, MACD, 볼린저) 자동 계산·제공
✅ 뉴스 감성 점수 (원문 아닌 점수만) — Cloud AI 모듈
✅ 포트폴리오 상태 자동 첨부 (로컬에서 조립)
```

데이터 제공은 "재료 공급". 서버가 판단/추천을 내리지 않으면 도구 제공에 해당.
상세는 `legal-data-review.md` §4~6 참조.

#### C. 안전장치 프레임워크 — 합법 + 필수

```
가능한 것:
✅ 손실 한도, 포지션 한도, 쿨다운 프레임워크 제공
✅ 기본값 설정 (예: 드라이런 ON, 일일 손실 50만원)
✅ 킬스위치, 자동 중단 로직
✅ 안전장치 위반 로그 기록
```

임계값의 "기본값"은 제공해도 된다. "이 값이 적당합니다"라는 추천만 아니면 됨.

#### D. 백테스팅 & 모니터링 — 합법

```
가능한 것:
✅ 과거 데이터로 프롬프트 전략 검증
✅ 수익률, 샤프비율, MDD, 승률 등 숫자 제공
✅ 프롬프트 버전별 성과 비교
✅ 실행 이력 로그, 대시보드
✅ 토큰 사용량 & 비용 추정
```

숫자만 보여주고, "이 전략이 좋다/나쁘다" 평가는 하지 않으면 됨.

### 1.3 상세: 불가능한 것 (투자자문업 등록 없이)

#### A. 클라우드에서 매매 판단 내려보내기 — 불가

```
불가한 것:
❌ API 응답에 "recommendation": "매수"
❌ API 응답에 "prediction": "상승 확률 85%"
❌ API 응답에 "action": "buy"
❌ API 응답에 "target_price": 85000
```

**이유**: 클라우드 서버가 매매 판단을 내려보내면 **자본시장법 제6조 제7항** 투자자문업에 해당.

#### B. 전략 판단 로직 기본 제공 — 불가

```
불가한 것:
❌ "RSI 30 이하면 매수 추천" 같은 기본 전략 프리셋
❌ "이 프롬프트가 성과가 좋습니다" 같은 전략 추천
❌ 백테스트 결과에 "이 전략이 우수합니다" 같은 해석
```

빈 틀과 작성법 가이드는 OK. 채워진 전략은 NO.

### 1.4 경계선 위의 것들 (주의 필요)

| 항목 | 판단 | 근거 |
|------|------|------|
| 프롬프트 작성 가이드 | ✅ OK | "변수 사용법, 응답 포맷 설명" 수준 |
| 전략 작성 예시 (교육용) | ⚠️ 주의 | "예시입니다, 투자 판단이 아닙니다" 명시 필요 |
| 기본 응답 스키마 | ✅ OK | 빈 구조만 (`action`, `ticker` 필드 정의) |
| 백테스트 성과 숫자 | ✅ OK | 숫자만, 해석/평가 없이 |
| 안전장치 기본값 | ✅ OK | "합리적 기본값"이지 "추천"이 아님 |
| Cloud AI 감성 점수 | ✅ OK | 데이터 가공 결과물, 매매 판단 아님 |

## 2. LLM 호출 구조

이 장은 §0의 세 모듈 중 **LLM을 사용하는 두 모듈 (A, C)** 의 기술 구조를 다룬다.
모듈 B (DSL 전략 엔진)는 LLM을 사용하지 않으므로 이 장의 범위 밖이다.

### 2.1 모듈 C: Custom LLM — 로컬 서버 호출

Custom LLM 통합에서 LLM API 호출은 **로컬 서버(사용자 PC)에서** 이루어진다.
브라우저에서 호출하거나 클라우드가 프록시하는 것이 아니다.

```
[로컬 서버 — Python 프로세스, 사용자 PC]
  ├── Credential Manager에서 LLM API 키 로드
  ├── 클라우드에서 받은 컨텍스트 + 캐시된 템플릿으로 프롬프트 조립
  ├── httpx로 LLM API 직접 호출 (Claude, OpenAI, Ollama 등)
  ├── 응답 파싱 → 안전장치 검증 → 주문 실행
  └── 실행 로그 로컬 SQLite에 저장
```

이 구조의 법적·기술적 의미:

| 질문 | 답 |
|------|---|
| 호출 주체가 누구인가? | **사용자** — 사용자 PC의 프로세스가 사용자 키로 호출 |
| LLM 비용은 누가 부담하나? | **사용자** — 사용자 API 키 |
| "우리가 호출했다" 해석 여지? | **없음** — 키움 주문과 동일 구조 (사용자 PC, 사용자 키) |
| CORS 문제? | **없음** — 브라우저가 아닌 Python 프로세스 |
| 프롬프트가 브라우저에 노출? | **없음** — 로컬 서버 내부에서 조립·전송 |

### 2.2 모듈 A: Cloud AI — 클라우드 서버 호출

아키텍처 §4.5의 Cloud AI 분석 모듈. Custom LLM 통합 (모듈 C)과 **별개**다.

```
Cloud AI 분석 모듈:
  ├── 우리 API 키 (ANTHROPIC_API_KEY, 서버 환경변수)
  ├── 시세 + 뉴스 데이터 → Claude API 호출
  ├── 결과: 감성 점수, 뉴스 요약, 리스크 지표 (데이터 가공물)
  └── 캐싱 → 로컬 서버가 컨텍스트로 fetch → 모듈 B·C의 입력 재료
```

이것은 **데이터 가공**이며 매매 판단이 아니다.
RSI를 계산해서 제공하는 것과 본질적으로 동일 (`legal-data-review.md` §6 참조).
우리 키로 호출해도 합법.

### 2.3 API 키 저장 (모듈 C 전용)

LLM API 키는 키움 API 키와 **동일한 패턴**으로 관리한다:

```
Windows Credential Manager (DPAPI):
  ├── stockvision:{user_id}/kiwoom_app_key      # 키움 API Key (기존)
  ├── stockvision:{user_id}/kiwoom_secret_key   # 키움 API Secret (기존)
  ├── stockvision:{user_id}/llm_api_key         # LLM API Key (신규)
  └── stockvision:{user_id}/sv_cloud_refresh_token  # JWT Refresh Token (기존)
```

- OS 레벨 암호화 (DPAPI) — 같은 Windows 사용자 세션 내 격리
- 클라우드 서버에 LLM API 키가 전송되지 않음
- 기존 `local_server/storage/credential.py` 패턴 재사용

### 2.4 프롬프트 템플릿 배포 경로 (모듈 C 전용)

```
[클라우드 서버]                          [로컬 서버]
  규칙 CRUD (템플릿 ID 포함)     →→→     규칙 캐시 (JSON)
  템플릿 목록 API                →→→     템플릿 캐시
                                         ↓
                                       프롬프트 조립 (템플릿 + 컨텍스트)
                                         ↓
                                       LLM API 호출
```

- 템플릿은 클라우드에 저장 (CRUD, 버전 관리, 다기기 동기화)
- 로컬 서버가 하트비트로 버전 변경 감지 → fetch → 캐시
- 프롬프트 조립·호출은 로컬에서만 발생

## 3. 경쟁력 분석

### 3.1 StockVision 포지셔닝

```
[완전 위임]                                              [완전 자유]
핀트/Stoic ←────── Composer ←── StockVision ──→ 젠포트 ──→ QuantConnect/Alpaca
라이선스 필요        AI 전략빌더     AI+DIY       DIY 퀀트      인프라
```

**StockVision = "젠포트의 접근성 + LLM 전략 생성" → 국내 미개척 영역**

### 3.2 경쟁 우위

#### 1. LLM 통합 — 국내 최초 포지션

| 경쟁자 | LLM 지원 | 비고 |
|--------|---------|------|
| 젠포트 | ❌ | UI 기반 조건식만 |
| 인텔리퀀트 | ❌ | JavaScript 코딩만 |
| 트레이딩뷰 | ❌ (웹훅 우회) | Pine Script만 |
| 핀트 | 내부만 | 사용자 LLM 연동 불가 |
| **StockVision** | **✅** | **사용자 LLM 직접 연결** |

젠포트의 가장 큰 진입장벽인 "조건식 설계"를 LLM으로 대체할 수 있다.

#### 2. 로컬 실행 — 법적·보안 이점

```
Composer:    서버에서 전략 실행 → 라이선스 경계 모호, 데이터 서버 경유
StockVision: 사용자 PC에서 전략 실행 → 라이선스 불필요, 데이터 로컬 보관
```

로컬 서버 구조 덕분에:
- 투자자문업 등록 불필요 (판단 주체 = 사용자 PC)
- 체결 내역/잔고가 클라우드에 없음 (개인정보 비보유)
- API 키가 사용자 PC를 벗어나지 않음

#### 3. 안전장치 내장

Composer.trade는 전략 빌더는 강력하지만 네이티브 리스크 관리가 없어 써드파티(Trade Conductor)가 등장.
StockVision은 **안전장치를 내장** (손실 한도, 킬스위치, 드라이런).

#### 4. 풀 파이프라인

```
데이터 수집 → 지표 계산 → LLM 분석 → 안전장치 → 드라이런/실매매 → 백테스트 → 모니터링
     ↑                                                                    ↑
  이미 구현                                                            이미 구현
```

OpenBB는 분석만, Alpaca는 실행만, 젠포트는 LLM이 없다. StockVision은 전 구간을 커버.

#### 5. 비용 구조 유리

| 항목 | StockVision | 핀트 | 젠포트 |
|------|------------|------|--------|
| 플랫폼 수수료 | 무료 (오픈소스) | 수익의 9.5% | 매매 0.1% |
| AI 비용 | 사용자 부담 (API 키) | 내재 | 없음 |
| 데이터 비용 | 무료 (yfinance) | 내재 | 내재 |

사용자가 부담하는 건 LLM API 비용뿐 — Claude Haiku 기준 분석 1회 ≈ $0.001~0.01.

### 3.3 경쟁 약점

#### 1. LLM 의존성 — 환각/오류 리스크

LLM은 틀린 분석을 할 수 있다. 사용자가 LLM 결과를 맹신하면 손실 발생.

**완화**:
- 드라이런 기본 활성화
- 안전장치 프레임워크
- 면책 조항 필수 표시
- 백테스트로 사전 검증 권장

#### 2. 데이터 품질 한계

yfinance는 지연 데이터 + 일봉 위주. 실시간 호가가 없다.

**완화**: 키움 REST API 연동 (로컬 서버에서 사용자 키로 직접 수신).

#### 3. 사용자 진입장벽 — API 키 필요

일반 사용자에게 "API 키를 발급받으세요"는 허들이 될 수 있다.

**완화**:
- Ollama (로컬 LLM) 지원으로 API 키 없이도 사용 가능
- API 키 발급 가이드 UI 내 제공
- DSL 규칙 엔진 (기존)은 LLM 없이도 동작

#### 4. 규모 & 신뢰 — 신생 서비스

젠포트는 검증된 사용자 베이스, 핀트는 AUM 3,000억원. StockVision은 아직 초기.

**완화**: 오픈소스로 투명성 확보, 모의투자로 리스크 없이 시작 가능.

### 3.4 진짜 경쟁력 = 복사할 수 없는 것들

| 경쟁력 요소 | 복사 난이도 | 설명 |
|------------|-----------|------|
| 데이터 파이프라인 | 높음 | DataProvider → 지표 계산 → 컨텍스트 빌드 전체 체인 |
| 안전장치 프레임워크 | 높음 | 손실한도/포지션한도/킬스위치/드라이런 통합 |
| 백테스팅 엔진 (LLM 전략) | 높음 | 날짜별 컨텍스트 재생성 + LLM 호출 + 성과 측정 |
| 로컬 서버 아키텍처 | 높음 | 오프라인 내성, 자동 로그인, 트레이 아이콘, 규칙 동기화 |
| 풀 파이프라인 통합 | 높음 | 데이터→분석→실행→백테스트→모니터링 원스톱 |
| 커뮤니티 & 개선 속도 | 높음 | 오픈소스 기여, 사용자 피드백 루프 |

## 4. 규제 환경 (2026년 기준)

### 4.1 적용 법령

| 법령 | 적용 여부 | StockVision 영향 |
|------|----------|-----------------|
| 자본시장법 제6조 (투자자문업) | 회피 가능 | 판단 = 사용자 PC, 클라우드 = 데이터만 |
| 인공지능기본법 (2026.1 시행) | 적용 | AI 서비스 제공자로서 기본 의무 |
| 금융분야 AI 가이드라인 (통합 개정) | 참고 | 자율규제, 법적 강제력 없음 |
| 금융소비자보호법 | 부분 적용 | 면책 조항, 위험 고지 필요 |
| 개인정보보호법 | 적용 | API 키 로컬 저장 (클라우드 미전송) |

### 4.2 투자자문업 회피 — 왜 가능한가

StockVision의 로컬 서버 구조는 투자자문업 회피를 **구조적으로** 보장한다:

```
투자자문업 = "투자판단에 관한 자문에 응하는 것을 영업으로 하는 것"

StockVision 구조:
  [클라우드 서버]  → 데이터/지표/감성점수 제공 (판단 아님)
  [로컬 서버]     → 사용자 PC에서 사용자 키로 LLM 호출 → 매매 판단
                    → 사용자 PC에서 사용자 키로 증권사 API 호출 → 주문

  판단 주체 = 사용자 (PC에 설치한 프로그램이 사용자를 대신하여 실행)
  플랫폼 역할 = 도구 제공 (젠포트, 트레이딩뷰와 동일)
```

업계 선례: 젠포트(조건식 자동매매), 인텔리퀀트(JS 전략), 트레이딩뷰(웹훅 자동매매) —
모두 투자자문업 등록 없이 운영 중. StockVision은 이들과 동일한 "도구 제공" 포지션.

### 4.3 인공지능기본법 대응

2026년 1월 시행된 AI 기본법의 핵심 원칙 중 StockVision 관련:

| 원칙 | 대응 |
|------|------|
| 보조수단성 | AI는 도구, 최종 판단은 사용자 (현재 구조 부합) |
| 신뢰성 | 드라이런 기본, 안전장치 내장 |
| 보안성 | API 키 Credential Manager 저장, 실행 로그 보존 |

### 4.4 필수 구현 항목

```
[이미 계획됨 — spec에 포함]
✅ API 키 로컬 저장 (Credential Manager, 기존 패턴)
✅ 드라이런 기본 활성화 (FR-5.7)
✅ 킬스위치 (FR-5.6)
✅ 실행 이력 보존 (FR-6.5)

[추가 필요 — 이 분석에서 도출]
⬜ 면책 조항 UI (서비스 하단 + 첫 사용 시 동의)
⬜ "투자 자문이 아닙니다" 고지 (Cloud AI 분석 결과 표시 시 항상)
⬜ 데이터 출처 표시 ("yfinance 기반 지연 데이터")
⬜ AI 분석임을 명시 ("AI 생성 결과이며 오류가 포함될 수 있습니다")
⬜ 이용약관에 AI 서비스 관련 조항 추가
```

## 6. 결론

### 6.1 핵심 판단

| 질문 | 답 |
|------|---|
| LLM을 우리가 붙여줄 수 있나? | **사용자 PC에서 사용자 키로 호출 → 가능** |
| 프롬프트 템플릿을 제공해도 되나? | **된다** — 빈 틀과 변수 바인딩은 도구 |
| 클라우드 AI (감성 점수 등)에 우리 키 써도 되나? | **된다** — 데이터 가공이지 매매 판단이 아님 |
| 전략 코드를 제공해도 되나? | **프레임워크는 OK, 판단 로직은 NO** |
| 투자자문업 등록 필요한가? | **현재 구조에서는 불필요** |
| API 없이 가능한가? | **Ollama로 가능** (API 키 불필요, 품질 제한 있음) |

### 6.2 경쟁력 요약

```
강점:
  • 국내 최초 LLM 통합 DIY 트레이딩 플랫폼
  • 로컬 실행 → 라이선스 불필요 + 개인정보 비보유
  • 안전장치 내장 (Composer 대비 차별화)
  • 풀 파이프라인 (OpenBB/Alpaca 대비 차별화)
  • 비용 구조 유리 (오픈소스 + 사용자 API 키)

약점:
  • LLM 환각 리스크 (면책 + 안전장치로 완화)
  • yfinance 데이터 한계 (키움 REST API로 보완)
  • API 키 진입장벽 (Ollama + 가이드로 완화)
  • 신뢰/규모 부족 (오픈소스 + 모의투자로 극복)
```

### 6.3 행동 항목

| 우선순위 | 항목 | 근거 |
|---------|------|------|
| P0 | 면책 조항 UI 구현 | 법적 필수 |
| P0 | API 응답 프레이밍 검수 | "추천/예측" 표현 제거 |
| P1 | spec을 Phase 3 아키텍처에 맞춰 갱신 | 코드 경로, 키 저장, 호출 위치 등 |
| P1 | Credential Manager LLM 키 저장 | 기존 패턴 확장 |
| P2 | Ollama 지원 | API 키 없는 사용자 대응 |
| P2 | 백테스트 비용 추정 UI | 사용자 비용 인지 |
| P2 | 이용약관 AI 조항 추가 | 법적 보강 |

### 6.4 spec 갱신 필요 사항

현재 `spec/custom-llm-integration/spec.md`는 Phase 2 모놀리스(`backend/app/`) 기준으로 작성되었다.
Phase 3 아키텍처에 맞춰 다음을 갱신해야 한다:

- **코드 경로**: `backend/app/services/...` → `local_server/` (엔진/실행) + `cloud_server/` (CRUD/저장)
- **API 키 저장**: Fernet 암호화 → Credential Manager (DPAPI)
- **LLM 호출 위치**: 명시 안 됨 → 로컬 서버 명시
- **DB 모델 위치**: 프로바이더/템플릿 CRUD → 클라우드 DB, 실행 이력 → 로컬 SQLite
- **실행 이력**: 로컬 SQLite (체결 로그와 동일 패턴)
