## 1. 아키텍처 일치성 검증

### 1.1 데이터 저장 위치 (일치도: 100%)

#### 금융정보 - 로컬 PC 저장 ✅

**Architecture 기준** (섹션 5.1):
```
금융정보: 사용자 PC (Credential Manager, SQLite)
  ├─ 키움 API Key/Secret
  ├─ 체결 로그 (거래가, 수량, 잔고)
  └─ Refresh Token
```

**문서별 반영도**:
| 문서 | 해당 섹션 | 내용 | 일치도 |
|------|---------|------|--------|
| 이용약관 | 6.2 | "서버에 저장하지 않음" | ✅ |
| 개인정보처리방침 | 3.1 | "로컬 PC에만 저장" 명시 테이블 | ✅ |
| 키움 준수서 | 1.2.3 | Credential Manager DPAPI 암호화 명시 | ✅ |
| 투자 위험 고지 | - | 직접 언급 안 함 (불필요) | - |

**✅ 일치**: 모든 문서에서 "키움 API Key는 서버 미저장"을 명확히 함

---

#### 클라우드 서버 - 규칙 & 시세만 저장 ✅

**Architecture 기준** (섹션 4.2):
```
클라우드 DB 보유:
  ├─ 사용자 계정 (이메일, 해시 비밀번호)
  ├─ 전략 규칙 (JSON 평문)
  ├─ 시세 데이터 (히스토리컬, 내부 분석용)
  └─ AI 분석 결과 (캐시)

클라우드 DB 미보유:
  ├─ 키움 API Key
  ├─ 체결 내역
  ├─ 잔고
  └─ 거래 종목
```

**문서별 반영도**:
| 문서 | 해당 섹션 | 내용 | 일치도 |
|------|---------|------|--------|
| 이용약관 | 7.1, 7.2 | 수집/미수집 항목 정확 구분 | ✅ |
| 개인정보처리방침 | 2.1~2.3 | "계정, 규칙, 시세만 저장" 명시 테이블 | ✅ |
| 키움 준수서 | 2.2.1 | "내부 분석용만 저장" | ✅ |
| 투자 위험 고지 | - | 직접 언급 안 함 | - |

**✅ 일치**: 완벽하게 일치

- **미반영**: 구체적인 동기화 메커니즘 (하트비트, 버전 관리) 설명 불필요 (기술 명세)

**결론**: ✅ 완전 일치

#### 주문 실행 흐름 (가격 검증)

**Architecture** (섹션 3.1):
```
1. 로컬 서버: 시세 수신
2. 규칙 평가 → "매수"
3. 키움 REST로 "현재가 직접 재확인" (가격 검증)
4. 가격 일치 → 주문 실행 / 불일치 → 거부
```

**문서 반영도**:

**문제점**: 이용약관/개인정보방침에서 "가격 검증으로 가공 의혹 방지"를 명시하면 신뢰도 향상

## 4. 문서 품질 이슈 (9개 항목)

### 4.1 이용약관 (5개 이슈)

#### 이슈 1: "종가" vs "현재가" 혼용
**위치**: 제5조 1.1 (시스템 오류 위험)
```
"구식 시세 데이터(지연된 가격)를 기반으로 한 주문"
```
**문제**: Architecture에서는 "현재가 직접 재확인"하므로, "지연된 가격"은 오류임
**개선안**:
```
"로컬 서버의 시세 캐시 지연(밀리초 단위)으로 인한 주문"
```
**심각도**: ⚠️ 경미 (사용자 이해도에 영향)

---

#### 이슈 2: "로컬 서버 독립 동작" 설명 부족
**위치**: 제5조 5.3
```
"로컬 서버는 클라우드 서버와 무관하게 독립적으로 동작할 수 있습니다"
```
**문제**: **어떻게 규칙을 업데이트하는가** 명시 부족
- 클라우드 장애 시 캐시 규칙 사용 ✅
- **신규 규칙은 수동 등록? 캐시만 사용?** ❓ 모호함

**개선안**:
```
"로컬 서버 독립 동작:
- 기존 규칙: 캐시에서 로드하여 계속 실행
- 신규 규칙: 클라우드 복구 후 동기화 필요
- 규칙 변경: 클라우드 복구 후 fetch"
```
**심각도**: ⚠️ 경미 (기술 상세는 약관에 불필요하나, 명확성 향상)

---

#### 이슈 3: "예정된 점검" 중 규칙 실행 여부
**위치**: 제5조 5.4
```
"점검 중 새로운 규칙 저장, 수정, 삭제 불가 (기존 규칙 실행은 유지)"
```
**문제**: **로컬 서버도 규칙 실행 불가?** 문맥 모호
- "점검 중" = 클라우드만 미접근? 또는 로컬도?

**개선안**:
```
"점검 중:
- 클라우드 웹 대시보드: 규칙 저장/수정/삭제 불가
- 로컬 서버: 캐시 규칙으로 계속 실행 (영향 없음)"
```
**심각도**: ⚠️ 경미 (명확성 개선)

---

#### 이슈 4: 제6조 "API Key 절대로 서버에 전송하지 않는다"와 "Refresh Token" 혼동
**위치**: 제6조 6.1
```
"사용자는 자신의 API Key와 Secret을 절대로 서버에 전송하지 않습니다."
```
**문제**:
- JWT (서비스 토큰) vs API Key (인증 정보) 구분 명확 ✅
- **하지만 "Refresh Token은 로컬에 저장"이라는 언급이 앞서 있어 혼동 가능**

**개선안**:
```
"사용자 책임:
1. 키움 API Key/Secret: 절대로 서버에 전송하지 않음
   → 사용자 PC의 Credential Manager에만 저장
2. StockVision JWT: 로컬 서버에 전달 (1회, 로그인 직후)
   → 프론트에서 localhost로 전달
3. Refresh Token: 로컬 PC의 Credential Manager에 저장
   → 서버와 무관하게 자동 갱신"
```
**심각도**: ⚠️ 경미 (명확성 개선)

---

#### 이슈 5: 배상액 한도 (제10조 10.3)
**위치**: 제10조 10.3
```
"배상액은 사용자가 지난 3개월간 서비스 이용료의 합계를 초과하지 않습니다."
```
**문제**:
- StockVision은 **현재 무료** → 배상액 = 0원 ❌
- 향후 유료 전환 시 명확해지지만, 현재는 혼동 가능

**개선안**:
```
"배상액 한도:
- 현재 (무료 기간): 배상 책임 불발생 (이용료 없음)
- 향후 유료화 시: 사용자가 지난 3개월간 납부한 이용료 한도"
```
**심각도**: ⚠️ 경미 (향후 유료화 시 수정 필요)

### 4.2 개인정보처리방침 (2개 이슈)

**문제**:
- 개인정보보호법상 "개인정보"는 **"개인을 식별할 수 있는 정보"**
- UUID만으로는 개인을 식별 불가 ✅
- **하지만 "앱 버전, OS, 키움 연결 상태" 등과 결합하면 개인 식별 가능?**
  - 예: UUID + OS + 타임스탐프 → 같은 사용자 추적 가능

**개선안**:
```
"익명 UUID 만으로는 개인정보 아님
단, 타임스탐프/OS와 결합하여 개인 추적 가능할 수 있으므로
집계 통계로만 활용 (개인별 분석 금지)"
```
**심각도**: ⚠️ 경미 (법적 해석에 따라, 보수적 해석 권장)

#### 이슈 8: 과거 성과 표시 기준 부재

**개선안**:
```
"과거 성과 표시 시 반드시 포함할 지표:
- 총 수익률 (누적 % 수익)
- 연평균 수익률 (CAGR)
- 최대 낙폭 (Maximum Drawdown)
- 승률 (Win Rate: 수익 거래 / 전체 거래)
- Sharpe Ratio (수익성 대비 변동성)
- 거래 횟수 및 기간

이 지표들이 모두 표시되지 않으면 신뢰성 낮은 성과로 간주"
```
