# 자동 업데이트 개선 리서치

> 작성일: 2026-03-29

## 1. 현재 시스템 분석

### 1.1 구현 상태 (v0.3.0)

| 기능 | 상태 | 비고 |
|------|------|------|
| GitHub Releases 버전 체크 | ✅ 구현 | 서버 시작 시 1회 |
| 하트비트 버전 감지 | ✅ 구현 | `on_heartbeat()` 존재 |
| 백그라운드 다운로드 | ✅ 구현 | 3회 재시도, SHA256 검증 |
| 자동 설치 (bat → Inno) | ✅ 구현 | `execute_update()` 존재 |
| 백업 (최근 2개) | ✅ 구현 | `backup/v{VERSION}/` |
| OpsPanel 배너 | ✅ 구현 | 노란/빨간 배너 |
| 설정 UI (on/off, 시간) | ✅ 구현 | `UpdateSettings` 컴포넌트 |
| **설치 스케줄러** | ❌ 미구현 | `try_install()` 호출부 없음 |
| **트레이 알림** | ❌ 미구현 | 업데이트 관련 토스트 없음 |
| **수동 설치 버튼** | ❌ 미구현 | "지금 설치" UI 없음 |
| **다운로드 진행률 표시** | ❌ 미구현 | `download_progress` 존재하나 UI에서 미사용 |
| **릴리즈 노트** | ❌ 미구현 | 변경 내용 표시 없음 |
| **주기적 재체크** | ❌ 미구현 | 서버 시작 시 1회만 |

### 1.2 핵심 문제점

#### P1: 설치가 실제로 실행되지 않음
`try_install()`을 호출하는 스케줄러/태스크가 없다. 다운로드까지는 자동이지만 설치는 사실상 불가.

#### P2: 버전 정보가 평소에 안 보임
- OpsPanel 배너는 업데이트 "가능" 시에만 표시
- 현재 버전은 로컬 상태 팝오버 클릭해야 확인 가능 (StatusPopover → "버전: N/A")
- Settings 페이지에 버전 표시 없음

#### P3: 다운로드 진행 UX 부재
- `download_progress` 필드가 API에 있으나 프론트에서 표시하지 않음
- 다운로드 중인지, 완료인지, 실패인지 사용자가 알 수 없음

#### P4: 업데이트 실패 시 복구 수단 없음
- 설치 실패 시 자동 롤백 없음 (수동 backup 복원만)
- 다운로드 실패 시 재시도 UI 없음
- 네트워크 문제로 체크 실패 시 재체크 수단 없음

#### P5: 버전 불일치 감지가 불완전
- 클라우드/프론트 버전과 로컬 버전 비교 로직 없음 (로컬 ↔ GitHub만)
- 프론트엔드 자체 버전 표시 없음

## 2. 벤치마킹

### 2.1 Electron 앱 (VS Code, Slack, Discord)

**electron-updater 패턴:**
- `download-progress` 이벤트: 퍼센트, 속도, 전송 바이트 실시간 제공
- 차등 업데이트 (differential): 변경분만 다운로드 (120MB → 5-20MB)
- 세 단계 UX: `update-available` → `download-progress` → `update-downloaded`
- 사용자가 재시작 시점 선택 (지금/나중에/앱 종료 시)

**VS Code 구체적 패턴:**
- 상태바 우하단에 항상 버전 표시
- 업데이트 시 기어 아이콘에 뱃지
- "Restart to Update" 버튼 제공
- 릴리즈 노트를 Markdown 탭으로 표시

**Discord 패턴:**
- 앱 시작 시 강제 업데이트 (스플래시 화면에서 진행)
- 사용자 선택권 없음 — SaaS 특성

### 2.2 비-Electron (Obsidian, 1Password)

**1Password:**
- 주 1회 체크 (베타/나이틀리는 더 자주)
- 잠김 상태: 자동 백그라운드 설치
- 열린 상태: 알림 후 사용자 확인 대기
- 작업 중단 최소화 설계

**Obsidian:**
- Settings > About에서 "Check for updates" 버튼
- 인앱 알림 바 (비침입적)
- Community 플러그인과 앱 업데이트 분리

### 2.3 트레이딩 앱 (키움 MTS, 토스증권)

- **강제 업데이트** 패턴이 일반적 (금융 규제 준수)
- 로그인 시 버전 체크 → 구버전이면 차단 + 다운로드 유도
- 앱 스토어(Play/App) 기반 배포가 주류
- 데스크탑 MTS (영웅문): 실행 시 버전 체크 → 패치 다운로드 → 재시작
- UX: 진행률 바 + "업데이트 중..." 스플래시 화면

### 2.4 Python + Inno Setup 모범 사례

**일반 패턴:**
1. 서버에서 버전 JSON 체크 → 다운로드 → Inno `/SILENT` 설치
2. PyInstaller onedir 번들 → Inno Setup 래핑
3. SHA256 또는 Authenticode 서명으로 무결성 검증

**Squirrel.Windows (참고):**
- 델타 업데이트 (~700KB)
- NuGet 패키지 기반
- .NET/Electron 중심이나 아키텍처 참고 가치 있음

**핵심 권장사항:**
- 설치 후 smoke test (새 버전 정상 실행 확인)
- 실패 시 자동 롤백 (이전 exe 복원)
- 버전 번호를 바이너리 메타데이터와 동기화

## 4. 개선 권장사항

### 즉시 필요 (P1 — 기능 완성)

1. **설치 스케줄러 추가**: `try_install()`을 주기적으로 호출하는 백그라운드 태스크
2. **수동 설치 API**: `POST /api/update/install` — 사용자가 "지금 설치" 가능
3. **주기적 재체크**: 1시간마다 GitHub/하트비트 재확인

### UX 개선 (P2 — 가시성)

4. **버전 상시 표시**: OpsPanel 로컬 상태 옆에 `v0.3.0` 항상 표시
5. **Settings 버전 정보**: Bridge 섹션에 현재 버전 + 최신 버전 표시
6. **다운로드 진행률 바**: OpsPanel 배너에 프로그레스 바 추가
7. **트레이 알림**: 업데이트 가능/다운로드 완료 시 Windows 토스트

### 안정성 (P3 — 복구)

8. **설치 후 health check**: 새 버전 시작 후 /health 확인, 실패 시 백업 복원
9. **수동 재체크 버튼**: Settings에서 "업데이트 확인" 버튼
10. **릴리즈 노트 표시**: GitHub Release body를 배너 하단에 접이식 표시

### 향후 고려 (Phase 2+)

11. 차등 업데이트 (델타 패치) — 현재 인스톨러 크기가 작아 우선순위 낮음
12. 클라우드-프론트-로컬 3자 버전 매트릭스 — 복잡도 높음
13. 베타 채널 지원 — pre-release 포함 옵션
