### 이상 발생 시

| 이상 유형 | 심각도 | 자동 대응 | 사용자 개입 |
|-----------|--------|---------|----------|
| 브로커 WS 끊김 | 높음 | REST 폴백 자동 전환 | 알림 확인 |
| 주문 실패 | 높음 | 없음 (재시도 없음) | 즉시 확인 |

| 클라우드 5분 끊김 | 중간 | 로컬 캐시로 계속 | 알림 확인 |
| 엔진 평가 exception | 높음 | try-except 루프 계속 | 로그 확인 |

### 현재 감지 공백

## C-2. 알림 정책

### 알림 이벤트

| 이벤트 | WS 타입 | 브라우저 토스트 | Windows 토스트 |
|--------|---------|--------------|--------------|
| 체결 성공 | `execution` | O (미동작-버그) | X |
| 주문 실패 | `execution` | X (누락) | X |
| Kill Switch 발동 | X (없음) | X | O |
| 손실 락 발동 | `execution` (rule_id=0) | X | O |
| 브로커 끊김 | `broker_disconnected` | O | X |
| 엔진 시작/중지 | X | X | X |

### 버그 발견: WS 타입 불일치
`useLocalBridgeWS.ts`의 handleMessage가 `signal_sent`, `execution_result` 처리.
서버(`ws.py`, `trading.py`)는 `execution` 타입 브로드캐스트.
**→ 체결 알림이 현재 전혀 동작하지 않음.**

### 빈도 제어
- 체결: 같은 종목+side 60초 이내 중복 억제
- 브로커 끊김: 5분 debounce
- Kill Switch/손실 락: 상태 변경 시에만 (회당 1번)

### 채널별 적합성
- 브라우저 토스트: 체결, 주문 거부, 엔진 변경 (즉각 인지)
- NotificationCenter: 모든 이벤트 보관 (50개 capped)
- Windows 토스트: Kill Switch, 손실 락, 클라우드 30분 단절
- 텔레그램 (미래): 손실 락, Kill Switch, 일별 요약

## C-3. 레이아웃 재검토

### 문제점
1. Kill Switch/손실 락 상태가 화면에 없음
2. 당일 PnL 없음 (account.dailyReturn = 0 하드코딩)
3. OpsPanel 1줄 vs 종목 정보 전체화면 → 비상 시 가시성 부족

### OpsPanel 3단계 확장 권장
```

주의: 경고 배너 (노란색 — 브로커 미연결, 규칙 0개)

  └ 재개 버튼 인라인 배치
```

# D. 원격 제어 상세

## D-1. 릴레이 아키텍처

| 항목 | (a) 하트비트 명령 | (b) 클라우드→로컬 WS | (c) 명령 큐 폴링 |
|------|----------------|-------------------|---------------|
| 지연 | 최대 30초 | 100ms | 3~10초 |
| 구현 복잡도 | **낮음** | 높음 | 중간 |
| 안정성 | **높음** | 낮음 (NAT 문제) | 높음 |
| 오프라인 | 자연스러움 | 재연결 필요 | 자연스러움 |
| 방화벽 | **불필요** | 인바운드 포트 필요 | 불필요 |
| 보안 | 기존 JWT + 서명 | 별도 인증 | JWT + nonce |

### 권장: (a) 하트비트 응답 명령 포함

이유:
- 하트비트 인프라 이미 완성 → `command` 필드 추가만으로 구현
- 로컬은 방화벽 안 → 클라우드→로컬 직접 연결 현실적 불가
- Kill Switch 30초 지연 허용 (자동 손실 락이 즉각 대응 담당)
- 긴급 시 폴링 간격 30초→10초 단축이 더 간단

### 하트비트 명령 설계
```json
{
  "rules_version": 42,
  "command": {
    "id": "cmd-uuid",
    "type": "kill_switch",
    "payload": {"mode": "STOP_NEW"},
    "issued_at": "...",
    "expires_at": "..."
  }
}
```

로컬 처리:
- `executed_command_ids` set → 중복 실행 방지
- `expires_at` 지난 명령 무시 (오프라인 보호)
- 다음 하트비트에 실행 결과 포함

클라우드 UI:
- `POST /api/v1/commands` → DB pending 저장 → 하트비트에서 수신

## D-2. 재개 UX

### 정지 상태 화면
```
[빨간 배너]
엔진 정지됨 — Kill Switch STOP_NEW (수동, 14:23)
미체결: 2건 | 마지막 체결: 14:21 삼성전자 매수 100주
브로커: 연결됨 (실전)

□ 정지 사유를 확인했습니다
□ 브로커 연결 상태를 확인했습니다
[재개] ← 체크 완료 시 활성화
```

### 사유별 재개 조건

| 정지 사유 | 재개 조건 | 자동 재개 |
|----------|---------|---------|
| 수동 Kill Switch | 체크박스 확인 | X |
| 손실 락 자동 | 확인 + "손실 사유 이해" 추가 체크 | X |
| 엔진 오류 | 오류 로그 확인 + 체크박스 | X |
| 일일 예산 초과 | **다음날 자동 리셋** | O (자정) |
| 주문 속도 제한 | 1분 자동 해제 (이미 구현) | O |

## 핵심 발견 사항

1. **버그**: `useLocalBridgeWS.ts` WS 타입 불일치 → 체결 알림 전혀 동작 안 함
2. **운영 안전 누락**: Kill Switch/손실 락 상태가 OpsPanel에 미표시
3. **원격 제어**: 하트비트 `command` 필드 방식 권장 (구현 최소, 방화벽 무관)
4. **캐싱 범위**: 시스템 상태만 허용, 잔고/포지션/PnL은 클라우드 전송 금지
