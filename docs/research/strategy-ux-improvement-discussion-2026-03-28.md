# 전략 빌더 UX 개선 논의 종합

> 작성일: 2026-03-28 | 상태: 초안 | 세션 기반 논의 정리

---

## 1. 현재 상태 평가

### 1.1 현재 구현된 기능

| 영역 | 상태 | 설명 |
|------|------|------|
| DSL v2 파서 | 완료 | `매수: RSI(14) < 30 AND 골든크로스()` |
| Strategy Builder UI | 완료 | 폼 에디터 + DSL 직접 편집 모드 |
| 백테스트 엔진 | 완료 | 멀티 타임프레임, 수익률/샤프/MDD 등 |
| 실시간 전략 엔진 | 완료 | 1분 주기 평가 → 시그널 → 주문 실행 |
| 주문 실행 파이프라인 | 완료 | 시장가/지정가, 중복 방지, 예산 체크 |
| 안전장치 | 완료 | Kill Switch, Loss Lock, 가격 검증 |
| 지표 | 완료 | RSI, MA, EMA, MACD, 볼린저, 평균거래량 |

전체 플로우: 규칙 생성(클라우드) → 로컬 sync → 엔진 시작 → 1분마다 DSL 평가 → 조건 충족 시 자동 주문 → WebSocket 프론트 알림

### 1.2 DSL 문법 문제점

**한영 혼용이 어색하다:**
```
매수: 골든크로스() AND RSI(14) < 30 OR 수익률 >= 5
       ^^^^한글     ^^^영어  ^^^영어    ^^^^한글
```

**함수 이름 일관성 없음:**
- `RSI(14)` — 영문 약자
- `볼린저_상단(20)` — 한글 + 언더스코어
- `평균거래량(20)` — 한글 붙여쓰기
- `MACD_SIGNAL()` — 영문 + 언더스코어

**타겟이 불분명:**
- 코더에게는 Pine Script보다 불편
- 일반인에게는 여전히 코딩

**결론:** DSL은 내부 IR(중간 표현)로만 쓰고, 사용자는 직접 안 건드리게 만드는 게 현실적.

## 2. 제품 벤치마킹

### 2.1 전략 빌더 UI/UX

| 제품 | 입력 방식 | 핵심 강점 | 배울 점 |
|------|-----------|-----------|---------|
| **Composer.trade** | 자연어 → AI 전략 | 60초 백테스트, 플레이북 공유 | "설명하면 만들어줌" 플로우 |
| **TradingView** | Pine Script 코드 | 차트-에디터 통합, 커뮤니티 마켓 | 백테스트 결과를 차트 위에 직접 시각화 |
| **3Commas** | 폼 + 프리셋 | 차트 드래그 TP/SL, 프리셋 자동입력 | 프리셋 → 파라미터 자동채움 |
| **TrendSpider** | 비주얼 + 자연어 | No-go 경고, 변형 테스트 | 수익 불가능 전략 사전 경고 |
| **키움 영웅문** | 수식 DSL | 한국 관행, 시험/실매매 분리 | 시험적용 vs 실매매 모드 명확 분리 |

### 2.2 차트 드로잉 → 자동매매

| 플랫폼 | 도형 그리기 | 알림 | 실제 주문 | 비고 |
|--------|-----------|------|----------|------|
| **NinjaTrader 8** | O | O | **네이티브** | 유일하게 그리기→실주문 완전 지원 |
| TradingView | O | O | 웹훅만 | 써드파티 브릿지 필요 |
| TrendSpider | O (자동감지+수동) | O | 브로커 봇 | 자동 추세선 AI 스코어링 |
| MetaTrader | O | EA 필요 | EA 필요 | 코딩 필수 |
| **국내 플랫폼** | **없음** | **없음** | **없음** | **차별화 기회** |

### 2.3 추세선/회귀선 기반 매매

| 선 종류 | 지원 플랫폼 | 비고 |
|---------|-----------|------|
| 수동 추세선 | 모든 플랫폼 | 2점 연결, 기울기 있는 선 |
| 자동 추세선 | TrendSpider (AI 상위 1%), TradingView (스크립트) | TrendSpider가 가장 완성도 높음 |
| 선형 회귀채널 | TradingView, NinjaTrader, thinkorswim, MT4 등 | 대부분 지원 |
| 다항식/이차 회귀 | thinkorswim (1~6차 내장), TradingView (스크립트) | 3차 이상은 과적합 위험 |

**국내에 "선 그리기 → 조건 연결" 직접 UI가 없음 — StockVision이 구현하면 명확한 차별화 포인트.**

### 3.1 데스크톱 전문 플랫폼

## 4. 전략 공유 포맷

### 4.1 핵심 결론: 범용 표준 교환 포맷은 존재하지 않는다

| 영역 | 포맷 | 한계 |
|------|------|------|
| 기관 | FIXatdl (XML) | 파라미터+UI 정의만. 로직 미포함 |
| 리테일 | Pine Script, MQL5, EasyLanguage 등 | 플랫폼 완전 종속 |
| 오픈소스 | Python 클래스 | JSON/YAML은 설정만, 로직은 코드 |
| 마켓플레이스 | 바이너리 배포 또는 신호 카피 | 로직 비공개가 표준 |
| 한국 | 키움 서버 저장, API 실행만 | Export 불가 |

### 4.2 StockVision 전략 공유 포맷 제안

```jsonc
{
  "version": "1.0",
  "name": "골든크로스 RSI 전략",
  "symbol": "005930",
  "timeframe": "1d",
  "script": "매수: 골든크로스() AND RSI(14) < 35\n매도: 수익률 >= 7",
  "execution": { "order_type": "MARKET", "qty": 10 },
  "trigger_policy": "ONCE_PER_DAY",
  "backtest_summary": { "return_pct": 12.3, "mdd": -5.1, "sharpe": 1.8 }
}
```

DSL 스크립트 + 메타데이터 + 백테스트 요약을 하나의 JSON으로 묶는 자체 포맷.

## 5. EasyLanguage 호환성 분석

### 5.1 법적 결론

| 항목 | 가능? | 근거 |
|------|-------|------|
| 문법/키워드 유사 구현 | **가능** | 언어 문법은 특허/저작권 비보호 |
| "EasyLanguage" 이름 사용 | **불가** | 등록 상표 |
| 공식 문서 참고 설계 | **가능** | 공개 PDF 다수 |

선례: MultiCharts → "PowerLanguage", 키움 → "Signal Language". 둘 다 법적 문제 없음.

### 5.2 StockVision DSL vs EasyLanguage 비교

| | EasyLanguage | StockVision DSL |
|---|---|---|
| 변수 선언 | `Input:`, `Vars:` | 없음 |
| 파라미터화 | `Input: Len(14)` → 외부 변경 가능 | 하드코딩 |
| 주문 표현 | `Buy 100 Shares at Market` | 별도 execution JSON |
| 이전 봉 참조 | `Close[1]`, `High[3]` | 없음 |
| 바 기반 시계열 | `Close`, `Open`, `High`, `Low`, `Volume` | `현재가`, `거래량`만 |
| 복합 로직 | if/then/else, for, 함수 정의 | AND/OR/NOT만 |
| 생태계 | 수만 개 공개 전략 | 없음 |

### 5.3 결론

EasyLanguage 전체 호환은 오버킬. 대신 **핵심 아이디어 2개만 차용**:

| EL 아이디어 | 적용 방안 | 효과 |
|------------|----------|------|
| `Input: Len(14)` | DSL에 파라미터 변수 추가 | 최적화, UI 슬라이더 연동 |
| `Close[1]` | `현재가[1]` 이전 봉 참조 | 패턴 인식 능력 대폭 향상 |

## 6. 전문 트레이더 시각 — 부족한 점

### 6.1 현재 DSL은 전략의 10%만 커버

```
프로가 결정하는 것:          현재 지원:
1. 언제 사나?              ← 있음
2. 언제 파나?              ← 있음 (고정 조건만)
3. 얼마나 사나?            ← 없음 (고정 수량)
4. 어떻게 나누어 사나?      ← 없음
5. 리스크 관리?            ← 없음
6. 여러 전략 조합?         ← 없음
7. 시장 상황 대응?         ← 없음
```

### 6.2 핵심 부재 기능

**포지션 사이징:**
- 현재: 수량 10주 (고정)
- 필요: 계좌의 N% 리스크, ATR 기반 변동성 조절, 손절폭 역산

**동적 청산:**
- 현재: `수익률 >= 5` (고정 %)
- 필요: 트레일링 스탑, 시간 기반 청산, 분할 청산, ATR 기반 동적 손절, 브레이크이븐 스탑

**시장 환경 필터:**
- 현재: 없음
- 필요: KOSPI 추세 필터, 변동성 필터, 섹터 로테이션

**백테스트 지표 부족:**
- 현재: 수익률, MDD, 승률, 샤프, 거래횟수, CAGR
- 추가 필요: 최대 연속 손실, R:R 비율, 월별 수익 분포, 슬리피지 민감도, **파라미터 민감도** (과적합 방지 핵심)

## 7. UX 개선 방향 (우선순위순)

### 7.1 사용자 경험 관점 재평가

화려한 UI보다 **"지금 이 전략이 뭘 하고 있는지" 투명하게 보여주는 것**이 트레이딩 도구의 핵심 UX.

| 기능 | 가치 | 이유 |
|------|------|------|
| 조건별 실시간 상태 + 트리거 이력 | **높음** | "왜 안 사는지" 즉시 파악 |
| 백테스트 차트 오버레이 + 월별 분포 | **높음** | 매매 포인트가 차트에 보이면 10배 직관적 |
| 시험/실매매 모드 분리 | **높음** | 안전. 사고 방지 |
| 동적 청산 (트레일링 스탑 등) | **높음** | 수익에 직결. DSL 확장 + 엔진 수정 필요 |
| 파라미터 민감도 분석 | **중간** | 과적합 방지. 프로와 아마추어의 차이 |
| 포지션 사이징 (리스크 기반) | **중간** | 고정 수량 → 계좌 비율 기반 |
| LLM 자연어 → DSL 생성 | **중간** | 서비스화할 때 의미 |
| 카드형 조건 프리뷰 | **낮음** | 기존 폼 + ✅❌로 충분히 대체 가능 |
| 프리셋 템플릿 | **낮음** | 이해 없는 실행 유도 위험 |
| 비주얼 블록 빌더 | **낮음** | 구현 비용 대비 효과 불확실 |

### 7.2 프로 트레이더 기준 개선 우선순위

| 순서 | 기능 | 난이도 | 임팩트 |
|------|------|--------|--------|
| 1 | 조건별 실시간 상태 + 트리거 이력 | 낮음 | 디버깅 능력 대폭 향상 |
| 2 | 동적 청산 (트레일링 스탑, 시간 청산) | 중간 | 수익에 직결 |
| 3 | 백테스트 차트 오버레이 + 월별 분포 | 중간 | 전략 신뢰도 판단 필수 |
| 4 | 파라미터 민감도 분석 | 중간 | 과적합 방지 |
| 5 | DSL 확장 (파라미터 변수, 이전 봉 참조) | 중간 | 표현력 대폭 향상 |
| 6 | 포지션 사이징 (리스크 기반) | 중간 | 자금 관리 |
| 7 | 시장 환경 필터 | 높음 | KOSPI 지수 연동 필요 |
| 8 | 차트 드로잉 → 자동매매 | 높음 | 국내 유일 차별화 |

## 8. 포지셔닝 결론

```
         코딩 깊이
           ↑
           │  QuantConnect       Sierra Chart
           │  (Python/C#)        (C++)
           │
           │         MultiCharts (EasyLanguage)
           │
           │    ★ StockVision
           │    (한국어 DSL + AI + 차트 드로잉)
           │
           │              TrendSpider (비주얼+자연어)
           │
           │    3Commas (폼)      Composer (자연어→AI)
           │
           └──────────────────────────────→ 접근성
```

**두 가지 방향:**
- **A. 접근성** — LLM 자연어 + 카드 프리뷰 + 차트 드로잉 (일반 트레이더)
- **B. 깊이** — 멀티 종목 + 포지션 사이징 + Walk-Forward 최적화 (전문 트레이더)
