## 2. 원격 제어 — 킬스위치/재개 법률 검토 완료

### 킬스위치 (정지)
- **법적 리스크 없음**: 투자판단 대행이 아닌 안전장치
- 전체 정지(all-or-nothing)가 안전. 종목 선별 정지는 투자판단에 가까워짐
- `stop_new`(신규 차단) + `stop_all`(미체결 취소) 둘 다 OK

### 재개 (resume)
- **법적 리스크 없음**: ON/OFF 스위치일 뿐
- 2단계 확인은 법적 요건이 아니라 운영 안전 장치
- **개선 제안**: 코드 입력보다 "현재 상태 요약 확인 + 체크박스"가 더 실용적
  - 정지 사유, 브로커 상태, 미체결 건수 등을 보여주고 확인

### 보안 관점
- 로컬 제어: `local_secret`으로 충분 (매 실행 생성, 메모리 전용)
- 원격 제어: 클라우드 JWT + 릴레이
- "클라우드가 로컬에 명령" 구조이지만, 매매 명령이 아닌 상태 제어이므로 기존 원칙과 충돌 없음

## 3. 온보딩 — 벤치마킹 및 사용자 관점

### 젠포트 참고점
- 가입 → 전략 만들기 → 백테스트까지 **증권사 연동 없이** 가능
- "일단 써보게 하고, 실전은 나중에" → 이탈률 감소

### 핵심 문제
- 현재: 온보딩 끝나도 대시보드에서 아무 일도 안 일어남 (규칙 없음, 종목 없음)
- **첫 번째 성공 경험까지의 거리**가 너무 김

### 개선 방향 (미확정)
- 증권사 없이 체험 가능한 구간 만들기 (샘플 전략, 모의 데이터)
- "왜 로컬 서버를 깔아야 하는가" → 보안/신뢰 메시지 명확히
- 온보딩 순서 재검토: 설치 동기 → 체험 → 설치 → 실전

### "왜 로컬인가" 사용자 메시지
- "증권사 키가 우리 서버를 거치지 않습니다"
- "주문은 당신의 PC에서만 나갑니다"
- 보안/신뢰가 로컬 설치의 핵심 이유
