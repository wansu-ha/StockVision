## 1. 한 줄 정의

`System Trader`는 여러 전략이 만든 신호를 받아, 포트폴리오 관점에서 실제로 어떤 매매 의도를 채택할지 결정하는 규칙형 결정 모듈이다.

쉽게 말하면:

- `Rule Evaluator`는 각 전략이 사고 싶은지 판정한다.
- `System Trader`는 그 판정들 중 무엇을 실제 행동으로 옮길지 정한다.
- `Execution Layer`는 정해진 행동을 브로커에 제출한다.

## 2. 왜 따로 불러야 하는가

아래 세 가지는 서로 다른 책임이다.

1. 전략 조건 평가
2. 포트폴리오 차원의 최종 판단
3. 주문 제출과 체결 추적

이 세 가지를 한 모듈에 몰아넣으면:

- 규칙 충돌이 꼬이고
- 테스트가 어려워지고
- 체결 상태와 전략 상태가 섞이고
- 나중에 Assistant/LLM을 붙일 때 경계가 무너진다

그래서 `System Trader`는 이름만 멋있는 개념이 아니라, 실제로 필요한 중간 계층이다.

## 3. System Trader라고 부르려면 최소 있어야 하는 것

### 3.1 후보 신호 수집

전략이 조건을 만족했다고 바로 주문하면 안 된다.

필요한 것:

- 전략별 후보 신호 생성
- 후보를 한 주기 단위로 모으기
- 후보를 공통 포맷으로 정규화하기

예:

- `rule_id`
- `symbol`
- `side`
- `priority`
- `desired_qty`
- `reason`

### 3.2 포트폴리오 스냅샷

한 주기 안에서 아래 상태를 함께 봐야 한다.

- 현재 포지션
- 현금
- 미체결 주문
- 이미 채택된 후보
- 일일 예산 사용량
- 종목별, 전략별 제한

이게 없으면 `전략별 주문기`일 뿐 `포트폴리오 판단기`는 아니다.

### 3.3 충돌 해소

같은 순간에 여러 전략이 동시에 신호를 낼 수 있다.

System Trader는 적어도 아래를 정해야 한다.

- 같은 종목 중복 진입을 허용할지
- 우선순위 높은 전략만 채택할지
- 분할 진입으로 합칠지
- 기존 포지션 청산과 신규 진입이 동시에 나오면 어떤 순서로 처리할지

### 3.4 자금 배분과 포지션 크기

단순히 `qty=1`로 내려보내는 것만으로는 부족하다.

적어도 아래 중 일부는 책임져야 한다.

- 종목별 최대 비중
- 전략별 최대 자금
- 일일 예산
- 남은 포지션 슬롯
- 분할 진입/분할 청산 규칙

### 3.5 Order Intent 상태 기계

`주문 접수`와 `체결`을 분리해야 한다.

최소 상태 예시:

- `PROPOSED`
- `APPROVED`
- `SUBMITTED`
- `PARTIAL_FILLED`
- `FILLED`
- `CANCELLED`
- `REJECTED`
- `EXPIRED`

이 상태 기계가 없으면 System Trader는 실행 레이어와 분리되지 않는다.

### 3.6 정합성과 재조정

실전에서는 로컬 앱이 낸 주문 말고도 아래가 생긴다.

- MTS 수동 주문
- HTS 수동 주문
- 브로커 서버 사정으로 인한 지연 체결
- 부분 체결 후 잔량

System Trader는 브로커 상태와 자신의 의도를 주기적으로 맞출 수 있어야 한다.

### 3.7 설명 가능성

System Trader는 아래 질문에 답할 수 있어야 한다.

- 왜 이 신호는 채택했는가
- 왜 저 신호는 버렸는가
- 왜 수량이 줄었는가
- 왜 신규 진입이 차단됐는가

이건 LLM용 설명이 아니라 운영 로그와 감사성 로그다.

## 4. 하지 말아야 하는 것

System Trader는 아래를 하면 안 된다.

- 사용자와 자연어 대화
- 전략 DSL 생성
- 뉴스 요약
- 개인화 비서 메모리 관리
- 브로커 키 저장

즉 `판단 모듈`이지 `비서`도 아니고 `코파일럿`도 아니다.

## 5. 현재 구현과의 비교

현재 구현은 좋은 출발점이지만, 엄밀히 말하면 아직 `System Trader`라기보다 `Strategy Engine + Executor`에 더 가깝다.

이미 있는 것:

- 전략 평가 루프
- 우선순위 정렬
- 중복 방지 일부
- 예산/포지션 수 제한 일부
- 킬스위치와 손실 제한
- 브로커 주문 실행

아직 부족한 것:

- 후보 신호를 모아 한 번에 판단하는 단계
- 포트폴리오 충돌 해소
- 주문 의도 상태 기계
- `SUBMITTED`와 `FILLED` 분리
- 외부 주문과의 정합성 반영

즉 지금은 `System Trader의 싹`은 보이지만, 이름을 붙이려면 한 단계 더 필요하다.

## 6. MVP에서의 현실적인 범위

MVP의 목표는 `완전한 포트폴리오 매니저`가 아니라 `얇은 System Trader`다.

MVP에서 꼭 넣을 것:

1. 한 주기 안의 후보 신호 수집
2. 같은 종목 중복 진입 차단
3. 남은 포지션 슬롯 기준 우선순위 선택
4. 일일 예산/종목별 한도 적용
5. `SUBMITTED`와 `FILLED` 분리
6. 주문 의도와 차단 이유 로그

MVP 이후로 미뤄도 되는 것:

- 섹터별 노출 제한
- 변동성 기반 동적 사이징
- 전략 간 성과 기반 동적 가중치
- 포트폴리오 전체 리밸런싱 엔진

## 7. 권장 모듈 경계

초기에는 폴더를 새로 쪼개지 않아도 된다.

권장 경계:

- `evaluator.py`
  - 전략별 BUY/SELL 후보 생성
- `system_trader.py`
  - 후보 신호 수집, 충돌 해소, order intent 생성
- `executor.py`
  - intent 제출, 취소, 체결 추적
- `safeguard.py`
  - 절대 차단 규칙
- `reconciler.py`
  - 브로커 상태와 로컬 의도 정합

## 8. 이름을 붙이는 기준

아래 질문에 `예`라고 답할 수 있으면 `System Trader`라고 불러도 된다.

1. 전략 신호를 바로 주문하지 않고 모아서 판단하는가
2. 포트폴리오 상태를 보고 어떤 신호를 버릴지 정하는가
3. 주문 접수와 체결을 구분하는가
4. 충돌과 차단 이유를 기록하는가
5. 실행 엔진과 책임이 분리되어 있는가

하나라도 빠지면 아직은 `Engine`, `Executor`, `Rule Runner` 쪽 이름이 더 정확하다.

## 9. 한 줄 권고

지금 시점에는 `System Trader`를 별도 문서로 분리하는 것이 맞다. 이 개념이 앞으로 StockVision의 핵심 실행 아키텍처가 될 가능성이 높고, LLM 문서와 섞어두면 오히려 경계가 흐려진다.
