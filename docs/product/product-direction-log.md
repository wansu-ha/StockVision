# 제품 방향 로그

대화 중 나온 제품 방향과 가설을 잃지 않기 위한 살아있는 문서입니다.
구현 spec으로 내려가기 전, 현재 판단과 열린 질문을 계속 누적합니다.

## 사용 방법

- 대화에서 중요한 제품 판단이 나오면 짧게 추가합니다.
- `현재 결정`은 지금 기준으로 가장 적절하다고 보는 방향입니다.
- `열린 질문`은 아직 검증, 실험, 법률 검토가 필요한 항목입니다.
- 구현 단계로 넘어가면 `/spec` 아래 개별 명세로 분리합니다.

## 스냅샷 - 2026-03-10

### 제품 가설

StockVision의 미래 포지셔닝은 `자동매매 툴`보다 `사용자 특화 주식 비서 + 로컬 실행 엔진`에 가깝습니다.

핵심 문장:

> 사용자의 투자 원칙과 계좌 상태를 기억하고,
> 매일 브리핑하고, 전략 엔진을 돌리고, 위험을 감시하며,
> 필요할 때만 안전하게 실행을 연결해주는 한국형 주식 비서.

### 현재 결정

1. 제품의 핵심 가치는 `추천`보다 `개인화`에 둡니다.
2. 제품의 핵심 가치는 `완전 자동화`보다 `안전한 실행`에 둡니다.
3. 제품은 일회성 챗봇이 아니라 `지속적으로 붙어 있는 비서`처럼 설계합니다.
4. 클라우드는 공통 데이터와 컨텍스트 생성에 집중합니다.
5. 로컬 서버는 개인화 판단과 실제 실행에 집중합니다.
6. 커스텀 LLM은 초기에 `메인 신호 생성기`보다 `개인화 심사기` 또는 `후보 필터`로 사용합니다.
7. 메신저는 첫 단계에서 `읽기 전용 알림`과 `주문 초안 전달` 중심으로 시작합니다.
8. 실제 주문은 항상 `강한 2차 확인`을 거치며, 이를 우회하지 않습니다.
9. 제품 흐름은 `전략 세우기 -> 엔진 시동 -> 비서 감시 -> 경고/초안 -> 승인/실행 -> 복기`로 설계합니다.
10. 차별화 포인트는 `한국 주식`, `사용자별 기억`, `메신저 비서 경험`, `로컬 실행 안전성`의 결합입니다.
11. 전략 동기화, 알림 전달, 메모리 동기화 같은 `클라우드 서버 제공 기능`은 유료화의 핵심 후보입니다.

### 비서가 맡을 일

- 아침 브리핑
- 장중 리스크 경고
- 관심 종목 및 보유 종목 중심 요약
- 사용자 원칙 위반 감지
- 주문 초안 생성
- 장마감 복기

### 사용자별 입력 정보

개인 비서가 공통 AI와 달라지려면 아래 정보를 반영해야 합니다.

- 보유 종목, 평균 단가, 손익, 현금 비중
- 당일 손실 한도와 현재 리스크 상태
- 사용자 투자 성향과 매매 스타일
- 반드시 지켜야 하는 금지 규칙
- 선호 섹터, 관심 종목, 피하고 싶은 패턴
- 과거 실수와 메모

### 권장 단계

#### 1단계 - 실행 코어

- 브로커 연동 안정화
- 전략 엔진 완성
- 백테스트, 로그, 리스크 가드 강화

#### 2단계 - 비서 레이어

- 아침 브리핑
- 장중 경고
- 장마감 리포트
- 사용자 프로필 및 메모리 구조

#### 3단계 - 메신저 인터페이스

- 알림 발송
- 명령 해석
- 주문 초안
- 확인형 액션

#### 4단계 - 커스텀 LLM 개인 비서

- 사용자 LLM 연결
- 프롬프트 템플릿
- 개인화 필터링
- 설명 생성

#### 5단계 - 데이터 모트

- 사용자 행동 이력
- 전략 성과 히스토리
- 비서 메모리
- 개인화 품질 개선

### 수익 모델 가설

`무료 오픈소스 트레이딩 엔진 + 유료 개인 비서 클라우드`

- 오픈소스: 로컬 엔진, 키 보관 방식, 브로커 어댑터, 기본 전략 프레임워크
- 유료 클라우드: 전략 동기화, 버전 히스토리, 백업, 메신저 알림, 브리핑, 리포트, 비서 메모리
- 핵심 과금 포인트는 `저장` 자체보다 `운영 편의`와 `지속적 경험`입니다.

### 오픈소스 방향

로컬 서버 기반 제품은 사용자가 구조를 상당 부분 확인할 수 있으므로,
`코드를 숨기는 전략`보다 `신뢰를 공개적으로 구축하는 전략`이 더 적합합니다.

방어력은 아래에서 나옵니다.

- 공개된 구조에 대한 신뢰
- 편한 설치와 운영 경험
- 누적되는 사용자 데이터와 히스토리
- 금융 서비스 수준의 안정성

### 오픈/유료 경계선

#### 오픈소스로 공개할 것

- 로컬 실행 엔진
- 브로커 어댑터와 로컬 키 보관 방식
- 전략 DSL과 평가 프레임워크
- 백테스트 러너와 로컬 로그
- 킬 스위치, 손실 제한, 중복 주문 방지 같은 핵심 안전장치
- BYO LLM 연결 레이어
- 엔진 상태를 보고 제어하는 기본 로컬 대시보드
- 파일 포맷과 API 스키마

#### 유료로 남길 것

- 전략 클라우드 동기화
- 전략 버전 히스토리, 롤백, 백업
- 아침 브리핑, 장중 알림, 장마감 리포트
- 비서 기본 메모리 동기화는 기본 경험으로 두고, 장기 기억/복구/고급 회상은 유료 후보로 둔다.
- 팀 워크스페이스, 공유 폴더, 권한 관리
- 관리형 뉴스, 공시, 요약 컨텍스트 묶음
- 업데이트 채널, 헬스체크, 운영 지원 도구
- 메신저 알림 전달과 메시지 라우팅

#### 이 경계선이 좋은 이유

- 오픈되는 부분은 사용자가 실행 구조와 키 안전성을 직접 확인할 수 있어 신뢰를 만듭니다.
- 유료 부분은 저장, 동기화, 알림, 운영 부담을 줄여줘서 시간 가치를 만듭니다.
- 가치의 경계는 `숨겨진 코드`가 아니라 `지속성`, `편의성`, `관리형 경험`에 있습니다.

#### 무료로 남겨도 좋은 선택 기능

- 단일 기기 사용
- 전략 수동 import/export
- 기본 로컬 리포트
- 로컬 전용 비서 메모리

#### 유료 핵심으로 두면 안 되는 것

- 단순 전략 저장만 과금하기
- 기본 브로커 연결을 유료로 잠그기
- 안전과 직결된 실행 로직을 비공개로 숨기기
- 핵심 거래 기능이 클라우드 없이는 동작하지 않게 만들기

### 기억 과금 원칙

- 비서 제품에서 `나를 기억해주는 최소 경험`은 유료 잠금으로 보이지 않게 설계한다.
- 로그인했을 때 기본 프로필, 관심종목, 비서 성향, 핵심 설정이 복원되는 것은 기본 경험에 가깝다.
- 유료는 장기 기억, 깊은 회상, 검색, 복구, 팀 공유, 관리형 브리핑처럼 `기억을 활용하는 고급 서비스`에 둔다.
### 클라우드 과금 원칙

- 유료화의 중심은 `오픈소스로도 할 수 있는 기능`이 아니라 `우리가 운영하는 클라우드 서비스`에 둔다.
- 로컬에서 가능한 기능은 무료/오픈소스로 남겨도 된다.
- 유료 가치는 전략 동기화, 백업, 메모리 동기화, 브리핑 생성, 메시지 전달, 관리형 컨텍스트 파이프라인에서 만든다.
- 퍼널이 필요하면 영구 무료 제공보다 `기간 제한 체험`이 경계가 더 선명하다.
### BYO LLM 수익화 원칙

- 사용자의 AI API 키 연결 자체는 누구나 할 수 있는 기본 기능으로 본다.
- 오픈소스로 공개되면 `모델 연결`과 `기본 프롬프트 실행`은 복제 가능하다고 가정한다.
- 따라서 유료 포인트를 `LLM 연결`이나 `프롬프트 문구 자체`에 두지 않는다.
- 유료 가치는 메모리 동기화, 컨텍스트 관리, 메시지 전달, 버전 관리, 운영 편의, 팀 협업에서 만든다.
- 프롬프트는 중요하지만 moat 자체는 아니며, 구조화된 사용자 데이터와 운영 경험이 더 중요하다.
### 당장은 피할 것

- 자유형 메신저 문장만으로 즉시 실주문 체결
- LLM이 전체 시장을 실시간으로 혼자 스캔하는 구조
- 클라우드가 개인별 매수/매도 판단을 직접 내리는 구조
- `전략 저장소`만 단독으로 파는 약한 수익 모델

### 열린 질문

1. 사용자 프로필은 어떤 스키마로 저장할지
2. 비서 메모리는 로컬 중심으로 둘지, 암호화된 클라우드 동기화를 둘지
3. 메신저 1순위 채널은 무엇으로 잡을지
4. 커스텀 LLM 응답 스키마를 얼마나 제한할지
5. 오픈소스 라이선스는 AGPL, 듀얼 라이선스, 오픈코어 중 무엇이 맞는지
6. 유료 플랜의 첫 과금 포인트를 무엇으로 둘지

### 지금 제품에서 부족한 것

기능 아이디어는 충분히 많지만, 실제 사용자가 매일 쓰게 만들기 위해 아직 부족한 것은 아래와 같다.

#### 1. 신뢰

- 이 엔진이 진짜 안전하게 주문을 다룬다는 확신
- 브로커 장애, 네트워크 끊김, 중복 주문, 재시작 상황에서 어떻게 보호되는지에 대한 신뢰
- `내 키가 진짜 로컬에만 있나`에 대한 투명성

#### 2. 안전한 원격 조작 구조

- 텔레그램 같은 원격 채널에서 어디까지 허용할지
- 읽기 전용, 주문 초안, 엔진 온오프, 실주문을 어떻게 구분할지
- 2차 확인, 허용 chat_id, 명령 로그, 킬 스위치 우선순위 설계

#### 3. 사용자 기억 모델

- 비서가 정확히 무엇을 기억해야 하는지
- 프로필, 금지 규칙, 관심종목, 메모, 과거 실수, 전략 선호를 어떤 스키마로 저장할지
- 로컬과 클라우드 동기화 경계

#### 4. 전략 수명주기

- 전략 생성 -> 백테스트 -> 드라이런 -> 실전 -> 복기까지 이어지는 루프
- 사용자가 전략을 더 좋아지게 만들 수 있는 피드백 구조
- 성과가 안 좋을 때 전략을 멈추고 수정하는 흐름

#### 5. 비서 품질

- 단순 요약이 아니라 `내 계좌 기준으로 중요한 것만 말하는 능력`
- 장황하지 않고 행동으로 이어지는 브리핑 품질
- 시장 동향과 내 포지션을 연결해서 설명하는 능력

#### 6. 온보딩

- 초보자도 설치 후 10분 안에 첫 브리핑을 받아볼 수 있는지
- 브로커 연결, LLM 연결, 텔레그램 연결, 전략 선택이 얼마나 쉽게 되는지
- 설치부터 첫 성공 경험까지의 마찰 제거

#### 7. 왜 돈을 내는지에 대한 명확성

- 사용자가 `좋다`고 느끼는 것과 `돈 내겠다`고 느끼는 것이 다를 수 있음
- 유료 포인트가 기능 잠금이 아니라 운영 편의와 지속성으로 자연스럽게 느껴져야 함
- 개인용 Pro만으로 충분한지, Team/운영형 서비스가 더 큰 매출원인지 판단 필요

#### 8. 관제와 복구

- 로컬 서버가 꺼져 있거나 비정상일 때 어떻게 감지하고 알릴지
- 재로그인, 재동기화, 재연결, 재구독, 복구 절차를 얼마나 자동화할지
- 사용자가 `신경 안 써도 굴러간다`고 느끼게 하는 운영성

#### 9. 데이터 품질과 설명 가능성

- 시장 동향, 뉴스, 공시, 브리핑 데이터의 신뢰성
- 비서가 왜 그런 경고나 제안을 했는지 설명 가능해야 함
- 특히 LLM이 관여할수록 근거와 로그가 중요함

#### 10. 제품의 감정적 가치

- 단순히 기능이 많은 도구가 아니라 `내 계좌를 챙겨주는 느낌`
- 매일 아침 먼저 찾아오고, 위험할 때 먼저 경고해주는 존재감
- 사용자가 앱이 아니라 `비서`를 쓰고 있다고 느끼게 하는 경험

### 사업 모델에 대한 냉정한 판단

- 사용자가 `부족하다`고 느끼는 것을 해결하지 못하면 유료화 명분은 약하다.
- 로컬 서버만으로 사용자가 원하는 대부분의 가치를 얻는다면, 많은 사용자는 무료로 머무를 가능성이 높다.
- 따라서 이 제품의 유료화는 `잠금`보다 `대체 불가능한 부족함 해결`에 기반해야 한다.
- 만약 클라우드가 제공하는 가치가 사용자의 체감 부족함보다 약하면, B2C 구독 모델은 강하지 않을 수 있다.
- 이 경우 전략은 세 가지다:
  1. 무료 오픈소스 중심 제품으로 신뢰와 사용자 기반을 먼저 확보한다.
  2. 정말로 귀찮고 어렵고 불안한 운영 문제를 클라우드가 대신 해결한다.
  3. 개인 구독보다 팀, 운영, 관리형 서비스로 무게중심을 옮긴다.

### 사용자가 실제로 부족함을 느낄 가능성이 큰 영역

- 여러 기기에서 끊김 없이 이어지는 경험
- 원격에서 상태를 확인하고 안심하는 경험
- 로컬 서버 장애나 브로커 이슈를 대신 감지해주는 운영성
- 뉴스/공시/맥락을 내 계좌 기준으로 정리해주는 시간 절약
- 전략을 계속 개선하고 복기하게 만드는 기록과 루프
- 팀 단위 공유, 승인, 감사 로그 같은 협업성

### 피해야 할 착각

- `좋은 기능`이 곧 `유료 기능`은 아니다.
- `기술적으로 가능`하다고 해서 사용자가 직접 하고 싶어하는 것은 아니다.
- 반대로 `오픈소스로도 할 수 있음`이 곧 `아무도 돈 안 냄`도 아니다.
- 다만 사용자가 직접 해도 크게 불편하지 않은 것은 유료 전환이 약할 가능성이 높다.

### 사용자가 실제로 돈 낼 만한 부족함 10개

아래 항목은 `로컬에서도 이론상 가능하지만`, 실제 사용자 입장에서 직접 운영하기 귀찮거나 불안해서
클라우드 서비스에 돈을 낼 명분이 생기기 쉬운 부족함이다.

#### 1. 내 로컬 서버가 지금 멀쩡한지 모른다

- 사용자는 엔진이 살아 있는지, 브로커 연결이 붙어 있는지, 전략이 실제로 돌고 있는지 항상 확인하고 싶어한다.
- 직접 운영하면 결국 원격 접속해서 확인해야 하거나, 이상이 생겨도 늦게 알게 된다.
- `원격 헬스체크`, `장애 감지`, `즉시 알림`은 돈을 받을 명분이 강하다.

#### 2. 밖에서는 내 상태가 끊긴다

- PC 앞에서는 다 되는데, 밖에 나가면 현재 포지션, 전략 상태, 최근 경고를 매끄럽게 보기 어렵다.
- 사용자는 어디서 로그인하든 `내 비서가 이어지는 느낌`을 원한다.
- `원격 대시보드`, `기기 간 연속성`, `상태 동기화`는 체감 가치가 크다.

#### 3. 중요한 이벤트를 놓칠까 불안하다

- 실적 발표, 급등락, 손절선 근접, 브로커 오류, 전략 중단 같은 이벤트는 놓치면 바로 손실 체감으로 이어진다.
- 로컬에서 직접 알림을 붙일 수는 있어도, 안정적으로 오래 굴리는 것은 귀찮다.
- `관리형 이벤트 감시 + 전달 보장형 알림`은 과금 명분이 분명하다.

#### 4. 뉴스와 공시를 내 계좌 기준으로 정리하기 귀찮다

- 사용자는 시장 전체 뉴스보다 `내 보유 종목`, `내 관심 종목`, `내 전략`에 중요한 것만 보고 싶어한다.
- 이걸 직접 파이프라인으로 엮는 건 가능하지만 오래 유지하기 어렵다.
- `계좌 기준 브리핑`, `관리형 뉴스/공시 컨텍스트`는 시간 절약 가치가 높다.

#### 5. PC를 바꾸거나 포맷하면 다시 세팅해야 한다

- 전략, 비서 설정, 관심종목, 메모, 리포트 이력이 흩어져 있으면 다시 복구하기 번거롭다.
- 사용자는 `로그인만 하면 바로 이어지는 경험`에 돈을 낸다.
- `동기화`, `백업`, `기기 교체 복구`는 전형적인 유료 명분이다.

#### 6. 왜 이런 판단이 나왔는지 근거가 부족하다

- 금융 제품은 결과만 보여주면 불안하다.
- 사용자는 `왜 이 경고가 떴는지`, `왜 이 종목이 후보가 됐는지`, `왜 이 주문이 막혔는지`를 알고 싶어한다.
- `설명 가능한 리포트`, `근거 로그`, `판단 이력`은 신뢰를 사는 기능이다.

#### 7. 전략이 언제부터 망가졌는지 추적하기 어렵다

- 전략 성과가 나빠졌을 때, 어떤 설정을 언제 바꿨는지 기억 못 하면 복기가 안 된다.
- 로컬 파일로 남길 수는 있지만 실제로는 정리되지 않기 쉽다.
- `전략 히스토리`, `변경 이력`, `성과 시점 비교`, `롤백`은 실사용자일수록 가치가 커진다.

#### 8. 설치와 운영이 생각보다 번거롭다

- 사용자는 처음엔 의욕적으로 설치하지만, 업데이트, 연동 깨짐, 브로커 정책 변경, 의존성 문제에서 지친다.
- `계속 굴러가는 상태`를 유지하는 것은 기능 개발과 별개의 가치다.
- `관리형 업데이트`, `연동 상태 점검`, `운영 지원`은 직접 해보면 돈 낼 이유가 생긴다.

#### 9. 혼자서 쓰기엔 기록과 승인 체계가 약하다

- 지인과 같이 전략을 보거나, 팀으로 검토하거나, 누가 무엇을 바꿨는지 확인해야 하는 순간이 온다.
- 로컬 단독 구조는 협업과 책임 분리가 약하다.
- `공유 워크스페이스`, `권한`, `승인`, `감사 로그`는 Team 유료화의 핵심이다.

#### 10. 비서는 있는데 나를 계속 챙겨주지는 않는다

- 단순 챗봇은 질문할 때만 반응하지만, 사용자가 돈을 내는 비서는 먼저 챙겨줘야 한다.
- 아침 브리핑, 장중 경고, 장마감 복기처럼 `내가 안 열어도 먼저 오는 경험`이 있어야 비서 가치가 생긴다.
- `지속적 운영형 비서 경험`은 무료 도구와 유료 서비스의 차이를 만든다.

### 한 줄 정리

- 사용자가 돈을 내는 이유는 `기능이 없어서`보다 `직접 운영하기 귀찮고, 놓치기 싫고, 불안해서`에 가깝다.
- 따라서 유료화 기준은 `무엇을 잠글까`보다 `무엇을 대신 운영해줄까`에 두는 것이 맞다.

### 구독형 서비스 기준선 재정리 (2026-03-10)

사용자 피드백 기준으로, 아래 항목은 `유료 잠금보다 기본 경험`에 가깝게 본다.

#### 기본 경험으로 두는 것이 자연스러운 것

- 로컬 서버 관제의 기본 상태 확인
- 원격 상태 확인의 기본 기능
- 기기 교체 복구의 기본 기능
- 설명 가능성의 기본 제공
- 비서가 먼저 찾아오는 최소 경험(예: 기본 아침 브리핑)
- 전략 변경 이력의 기본 보관(예: 최근 30일)

#### 유료 후보로 보기 쉬운 것

- 텔레그램보다 나은 전용 프론트 대화 경험
- 단, `채팅 그 자체`보다 `전략 설정`, `계좌 상태`, `주문 초안`, `설명`, `로그`, `승인`이 한 화면에서 이어지는 제품 경험이어야 한다.
- 고급 브리핑과 관리형 뉴스/공시 컨텍스트
- 장기 전략 히스토리, 장기 롤백, 장기 리포트 보관
- 고급 알림 전달 인프라와 다채널 운영
- 팀 협업, 권한, 감사 로그, 승인 플로우
- 운영 지원, 업데이트 관리, 장애 대응 지원

#### 이벤트 누락 방지의 해석

- `서버가 사용자의 전략을 감시하고 매수/매도를 권하는 것`으로 해석하면 위험하다.
- 더 적절한 해석은 `사용자 또는 로컬 엔진이 정의한 이벤트를 놓치지 않게 전달하는 것`이다.
- 예: 손실 한도 도달, 브로커 연결 끊김, 주문 거부, 사용자가 정한 손절선 근접, 실적 발표일 진입 금지 규칙 충돌.
- 즉, 유료 가치가 있다면 `판단`보다 `전달 보장`과 `운영성` 쪽에 둔다.

#### 사용자 데이터와 프라이버시 원칙

- 사용자의 보유 종목, 관심 종목, 전략 정보는 가능하면 `기본적으로 로컬 우선`으로 둔다.
- 클라우드가 이 정보를 다루려면 `명시적 동의`, `목적 고지`, `최소 수집`, `보관 기간`, `삭제/처리정지`가 분명해야 한다.
- 가능한 구조는 세 가지다:
  1. 전체 로컬 보관 + 클라우드는 최소 메타데이터만 사용
  2. 암호화 동기화로 클라우드는 내용을 직접 보지 못함
  3. 사용자가 관리형 브리핑을 원할 때만 선택적으로 일부 정보를 업로드
- 신뢰 관점에서는 `내가 사용자 계좌를 다 안다`보다 `필요할 때만 제한적으로 안다`가 더 좋다.

#### 전략 히스토리 과금 감각

- `무료 최근 30일 / 유료 장기 보관`은 비교적 자연스럽다.
- 다만 `평생`처럼 무제한을 전면 약속하기보다 `장기 보관`이나 `보관 한도 확장` 표현이 운영상 더 안전하다.

#### 8번과 9번의 의미

- 8번 운영 대행성: 업데이트, 연동 깨짐, 정책 변경, 의존성 문제, 장애 대응 같은 `계속 굴러가게 하는 일`
- 9번 협업/승인: 공유 전략, 권한 분리, 누가 무엇을 바꿨는지 추적, 승인 후 실행 같은 `팀 기능`

#### 현재 기준 한 줄 정리

- 무료는 `신뢰를 만드는 기본 경험`
- 유료는 `직접 운영하면 귀찮고 불안한 일을 대신해주는 관리형 경험`
- 따라서 구독형의 핵심은 `메신저 대체`가 아니라 `도메인 특화된 더 나은 운영 인터페이스`에 있다.

### 원격 비서 대화 과금 가설 (2026-03-10)

사용자 피드백 기준으로, `같은 PC의 로컬 서버 환경에서는 무료`, `모바일 및 로컬 서버가 없는 환경에서 같은 비서를 이어서 대화하는 경험은 유료`라는 경계선이 비교적 자연스럽다.

#### 이 경계선이 설득력 있는 이유

- 무료 사용자는 로컬 서버를 설치하고 같은 PC에서 충분히 핵심 가치를 체험할 수 있다.
- 유료 사용자는 `어디서나 같은 비서에 접속하는 연속성`에 돈을 내는 구조가 된다.
- 과금 포인트가 `지능 자체`가 아니라 `클라우드 릴레이와 원격 접근 경험`이라서 BYO 키 구조와도 잘 맞는다.
- 텔레그램 무료 대체재가 있어도, `설정 없이 바로 되는 웹/모바일 비서`는 충분한 차별점이 될 수 있다.

#### 권장 표현

- 나쁜 표현: `돈 내면 대화 가능`
- 좋은 표현: `Pro에서 어디서나 같은 비서에 접속 가능`

#### 무료 경험

- 로컬 PC에서 비서와 대화
- 로컬 서버 기반 전략 실행
- 기본 원격 상태 확인
- 기본 복구와 설명

#### Pro 경험

- 모바일 앱/웹에서 같은 비서와 대화
- 다른 PC에서 설치 없이 로그인 후 같은 비서 사용
- 대화 히스토리, 비서 설정, 최근 맥락의 연속성
- 로컬 서버 온라인 시 실시간 응답
- 로컬 서버 오프라인 시 마지막 동기화 정보와 메모 기반 응답

#### 중요한 UX 포인트

- 사용자가 원하는 것은 `새 비서`가 아니라 `내 비서를 같은 성격과 기억으로 이어서 쓰는 것`이다.
- 따라서 Pro의 본질은 원격 채팅 기능이 아니라 `연속성 있는 개인 비서 접속권`에 가깝다.

#### 제품 문장 초안

- 무료: `내 PC에서 안전하게 쓰는 로컬 주식 비서`
- Pro: `어디서나 같은 비서를 이어서 쓰는 개인 주식 비서`

### 로컬 키 보관의 현실적 보안 원칙 (2026-03-10)

#### 핵심 전제

- 사용자 PC가 악성코드나 원격제어 툴에 의해 이미 장악된 상황은 제품만으로 완전히 방어할 수 없다.
- 즉, `로컬에만 키를 둔다`는 것은 `서버 유출 위험을 줄이는 것`이지 `감염된 로컬 PC에서도 절대 안전`을 의미하지 않는다.

#### 제품이 솔직하게 말해야 하는 것

- 우리는 사용자의 LLM 키와 증권사 키를 서버에 저장하지 않는다.
- 하지만 사용자의 PC가 악성코드에 감염된 경우, 로컬 저장 키와 실행 세션도 탈취될 수 있다.
- 따라서 제품 약속은 `무조건 안전`이 아니라 `공격면 축소`와 `피해 완화`여야 한다.

#### 줄일 수 있는 위험

- 서버 측 대규모 키 유출 위험
- 클라우드 침해로 인한 일괄 탈취 위험
- 운영자 내부 접근으로 인한 노출 위험

#### 여전히 남는 위험

- 사용자 PC 감염에 의한 로컬 키 탈취
- 화면 훔쳐보기, 입력 가로채기, 세션 탈취
- 사용자가 잘못 설치한 서드파티 프로그램으로 인한 노출

#### 대응 원칙

1. 키는 평문 파일 대신 OS 보안 저장소 사용 우선
2. 가능하면 장기 고정 키보다 재발급/폐기 가능한 자격 증명 선호
3. 주문 전 민감 액션은 별도 확인 수단 제공
4. 원격 채널은 조회/초안 중심, 실주문은 더 강한 확인 절차 적용
5. 사용자가 키를 쉽게 폐기하고 재발급할 수 있는 UX 제공
6. 이상 징후 탐지와 세션 로그 제공
7. 보안 가이드를 제품 안에서 적극 안내

#### 제품 메시지 원칙

- `로컬이라서 절대 안전`이라고 말하지 않는다.
- 대신 `서버에 맡기지 않아 대규모 유출 위험을 줄인다`라고 말한다.
- 고급 사용자에게는 전용 PC, 최소 권한, 보안 저장소, 이중 인증 같은 운영 수칙을 권장한다.

#### 사업적 의미

- 이 이슈는 유료화 포인트라기보다 신뢰 설계 포인트다.
- 보안 현실을 솔직하게 설명하는 제품이 오히려 더 신뢰를 얻는다.

### 보안 관점 핵심 정리 (2026-03-10)

#### 1. 무엇을 지켜야 하는가

- 증권사 키, LLM 키, 클라우드 refresh token
- 실제 주문 실행 권한
- 사용자 전략과 비서 메모리
- 원격 대화 세션과 승인 흐름
- 업데이트 채널과 배포 무결성

#### 2. 현실적인 공격자 모델

1. 같은 PC의 악성코드 또는 원격제어 툴
2. 프론트엔드 XSS 또는 브라우저 세션 탈취
3. 로컬 API를 노리는 같은 호스트의 악성 프로세스
4. 클라우드 서버 침해 또는 토큰 탈취
5. 공급망 공격(가짜 업데이트, 변조된 패키지)

#### 3. 절대 막기 어려운 것

- 사용자의 PC가 이미 같은 사용자 권한으로 장악된 상태
- 키로거, 화면 탈취, 메모리 덤프, 같은 세션에서의 로컬 API 호출
- 즉, 로컬 보관은 `서버 대량 유출 위험 감소`이지 `감염 호스트에서 절대 안전`을 뜻하지 않는다.

#### 4. 제품이 반드시 막아야 하는 것

- 서버 침해만으로 전체 사용자 키가 일괄 탈취되는 구조
- 클라우드가 브로커 키를 보관하거나 직접 주문하는 구조
- 로컬 서버가 외부 인터넷에 직접 노출되는 구조
- 한 번의 토큰 탈취로 무제한 주문이 가능한 구조
- 업데이트 변조로 악성 코드가 배포되는 구조

#### 5. 현재 구조의 좋은 점

- 로컬 서버는 기본적으로 127.0.0.1에 바인딩된다.
- 로컬 보호 엔드포인트는 `local_secret` 검증을 건다.
- 브로커 자격증명은 keyring에 저장하고 설정 파일에는 평문 저장을 피한다.
- 킬스위치와 loss lock이 엔진 하단에 있다.

#### 6. 현재 구조에서 과신하면 안 되는 것

- `local_secret`은 강한 사용자 인증이 아니라 같은 로컬 세션 안의 요청을 구분하는 정도에 가깝다.
- 허용된 프론트가 XSS에 뚫리거나 호스트가 감염되면 local API도 악용될 수 있다.
- keyring은 평문 파일보다 낫지만, 같은 사용자 권한 악성코드까지 막아주지 않는다.

#### 7. 권장 보안 원칙

1. 키는 계속 로컬 전용, 클라우드 미보관
2. 자동전략 주문과 원격 자연어 주문을 권한 모델에서 분리
3. 자동전략은 `엔진 무장 세션` 단위로만 주문 가능
4. 원격 자연어 주문은 더 강한 확인 또는 더 좁은 한도 적용
5. 일일 손실 한도, 종목당 한도, 총 주문 금액 한도는 최하단에서 강제
6. `긴급 정지`는 어디서나 가능하게 하고, `실행 재개`는 더 어렵게
7. 배포물과 업데이트는 코드 서명 및 무결성 검증 전제
8. 원격 조회는 넓게, 원격 실행은 좁게
9. 로그는 충분히 남기되 비밀값은 절대 남기지 않기
10. 사용자가 보안 수준을 선택할 수 있게 `기본 모드 / 강화 모드` 제공

#### 8. 사용자에게 솔직하게 말해야 하는 것

- `우리는 키를 서버에 저장하지 않습니다.`
- `하지만 사용자의 PC가 감염되면 로컬 저장 키와 실행 세션도 탈취될 수 있습니다.`
- `따라서 제품의 목표는 절대 불침이 아니라 공격면 축소, 피해 제한, 빠른 중지와 복구입니다.`

#### 9. 제품적으로 가장 중요한 설계 한 줄

- `뚫려도 계좌 전체가 아니라, 사용자가 미리 허용한 전략/한도 안에서만 움직이게 만든다.`

### 현재까지의 합의 요약 (2026-03-10)

#### 제품 정체성

- StockVision은 `자동매매 툴`보다 `사용자 특화 주식 비서 + 로컬 실행 엔진`으로 포지셔닝한다.
- 핵심 가치는 `종목 추천`보다 `개인화`, `완전 자동화`보다 `안전한 실행`, `일회성 챗봇`보다 `지속적으로 붙어 있는 비서`에 둔다.

#### 역할 분리

- 클라우드: 공통 데이터, 동기화, 원격 접속, 브리핑/리포트, 운영 서비스
- 로컬 서버: 사용자 키 보관, 개인화 판단, 전략 실행, 주문 실행
- 프론트: UI와 대화 인터페이스

#### LLM 원칙

- LLM은 직접 주문하지 않는다.
- LLM은 전략 생성, 규칙 문장화, 파라미터 제안, 설명 생성 역할을 맡는다.
- 실제 주문은 저장된 전략이 조건을 만족할 때 엔진이 실행한다.

#### 키와 신뢰

- LLM 키와 증권사 키는 서버가 보관하지 않는다.
- 키는 로컬 서버 + OS 보안 저장소(keyring) 원칙으로 간다.
- 제품 메시지는 `절대 안전`이 아니라 `서버 대량 유출 위험을 줄인다`에 둔다.

#### 무료 / 유료 감각

- 무료는 `신뢰를 만드는 기본 경험`
- 유료는 `같은 비서를 어디서나 이어서 쓰는 경험`과 `직접 운영하면 귀찮고 불안한 일을 대신해주는 관리형 경험`
- 텔레그램 무료 대체재는 막지 않고, 가벼운 진입 채널로 활용한다.
- 자체 프론트는 단순 채팅창이 아니라 `전략/상태/설명/승인/로그`가 이어지는 운영 콘솔이어야 한다.

#### 접속 경험

- 같은 PC에서는 로컬 서버에 직접 접속하는 무료 경험을 제공한다.
- 모바일이나 다른 PC처럼 로컬 서버가 없는 환경에서 `같은 비서와 이어서 대화`하는 경험은 Pro 핵심 후보다.
- 원격 조회는 넓게, 원격 실행은 좁게 가져간다.

#### 자동매매와 원격 명령

- 자동 전략 주문은 엔진을 한 번 `무장(arm)`하면 로컬 서버가 unattended로 실행한다.
- 원격 자연어 즉시 주문은 별도 위험군으로 보고 더 엄격하게 다룬다.
- 전략 주문과 원격 수동 주문은 권한 모델을 분리한다.

#### 이벤트와 감시

- `이벤트 누락 방지`는 서버가 투자판단을 대신하는 것이 아니라,
  사용자가 정의한 규칙 및 로컬 엔진 상태 이벤트를 놓치지 않게 전달하는 것으로 해석한다.
- 예: 손실 한도 도달, 브로커 연결 끊김, 주문 거부, 외부 주문 감지, 전략 중단.

#### 프라이버시 원칙

- 사용자의 보유 종목, 관심 종목, 전략 정보는 기본적으로 `로컬 우선`으로 본다.
- 클라우드가 이를 다루려면 최소수집, 목적 고지, 선택적 업로드, 가능하면 암호화 동기화를 지향한다.

#### 외부 주문 감지

- 브로커 API의 주문/체결 조회 및 실시간 통보를 로컬 엔진 주문 ledger와 대조하여 `우리 앱이 내지 않은 주문`을 외부 주문으로 감지할 수 있다.
- 이 기능은 유용하며, 비서/보안/운영성 측면 모두에서 가치가 있다.

### 프론트 제공 LLM의 안전한 역할 정리 (2026-03-10)

#### 상대적으로 안전한 방향

- 프론트에서 `전략 DSL 코딩 보조`와 `코드 생성 LLM`을 제공하는 것 자체는 가능성이 높다.
- 이 경우 LLM의 역할은 `프로그래밍 도구` 또는 `전략 초안 작성 보조자`에 가깝다.
- 사용자가 자연어로 규칙을 말하면 DSL 초안으로 변환하고, 문법 오류를 고치고, 백테스트용 템플릿을 만드는 역할은 비교적 안전한 편이다.

#### 위험해지는 지점

- LLM이 사용자 계좌, 보유종목, 투자성향을 읽고 `당신에게 맞는 전략`을 실질적으로 추천하는 경우
- `이 전략이 지금 시장에서 가장 좋다`, `오늘은 반도체 눌림목 전략으로 가자`, `삼성은 0.5% 빠지면 사라`처럼 투자판단을 대신하는 경우
- 생성 결과가 전략 초안을 넘어 사실상 즉시 실행 지시 또는 개인별 투자조언으로 기능하는 경우

#### 권장 제품 원칙

- LLM 출력은 항상 `전략 초안`, `설명`, `테스트 코드`, `문법 수정안`까지만 허용한다.
- 생성된 결과는 반드시 사용자가 저장하고 검토하고 백테스트/드라이런을 거친 뒤에만 엔진에 반영한다.
- `추천 전략`, `수익률 높은 전략`, `당신에게 맞는 전략` 같은 문구보다 `전략 코드 생성`, `규칙 번역`, `초안 제안` 같은 문구를 사용한다.
- 서버 제공 LLM은 가능하면 사용자 계좌 원장이나 민감한 포트폴리오 데이터를 직접 받지 않는 쪽이 더 안전하다.

#### 한 줄 원칙

- `LLM은 투자조언자가 아니라 전략 DSL 코파일럿이다.`

### 비서 UX 원칙: 거절 루프를 만들지 않는다 (2026-03-10)

#### 문제의식

- 금융 규제를 너무 정직하게 그대로 노출하면 비서가 매번 `그건 안 되고 대신 이것만 됩니다`만 반복하는 답답한 UX가 된다.
- 사용자는 규제 설명을 듣고 싶은 것이 아니라 `내 의도를 가능한 한 자동으로 안전하게 처리`해주길 원한다.

#### 권장 원칙

- 비서는 금지 요청을 바로 거절하는 대신, 가장 가까운 `안전한 작업 단위`로 자동 변환한다.
- 사용자가 말한 의도를 여러 단계의 작업으로 쪼개서 가능한 부분은 바로 수행하고, 위험한 마지막 단계만 승인 또는 수동 결정으로 남긴다.

#### 예시

- `삼성 좋은 전략 적용해줘`
  - 나쁜 UX: 불가능합니다.
  - 좋은 UX: 삼성전자용 전략 초안 3개를 생성하고, 사용자가 저장한 평가 기준으로 비교한 뒤, 1위 초안을 비활성 상태로 저장하고 백테스트 준비까지 끝낸다.

- `포트폴리오 재분배해줘`
  - 나쁜 UX: 투자자문에 해당할 수 있어 불가합니다.
  - 좋은 UX: 사용자가 미리 정한 목표 비중 규칙에 따라 리밸런싱 초안을 계산하고, 예상 비용/변화량을 보여준다.

- `오늘 뭐 사?`
  - 나쁜 UX: 추천은 불가합니다.
  - 좋은 UX: 현재 활성 전략 조건에 들어온 후보와 조건 충족률, 리스크 상태를 보여준다.

#### 설계 원리

- 금지해야 하는 것은 `의도`가 아니라 `권한`이다.
- 사용자가 원하는 높은 수준의 목적은 받아들이고,
  실제로는 `전략 초안 생성`, `백테스트`, `드라이런`, `초안 저장`, `활성화 후보 등록`, `주문 초안 계산` 같은 안전한 primitives로 분해한다.

#### 제품 문장

- 비서는 `하지 못한다`보다 `여기까지는 바로 해뒀고, 마지막 결정만 남았다`처럼 응답해야 한다.
- 즉, 규제를 UX의 벽이 아니라 워크플로우의 마지막 확인 단계로 흡수한다.

### 사용자 요청을 안전한 작업으로 변환하는 예시 20개 (2026-03-10)

#### 전략 생성 / 수정

1. 사용자: `삼성 좋은 전략 적용해줘`
   비서: `삼성전자용 전략 초안 3개를 만들었어요. 각 초안의 진입/청산/손절 규칙과 최근 백테스트 조건을 정리해뒀습니다. 1위 초안을 비활성 상태로 저장할까요?`

2. 사용자: `반도체 눌림목 전략 짜줘`
   비서: `반도체 대형주용 눌림목 전략 DSL 초안을 만들었어요. 거래대금 필터, 추세 필터, 손절 규칙까지 포함했습니다. 백테스트용으로 저장할까요?`

3. 사용자: `손절을 좀 더 짧게 바꿔줘`
   비서: `현재 전략의 손절 규칙을 -2.0%에서 -1.2%로 바꾼 수정안을 만들었어요. 기존 버전과 비교해서 저장할까요?`

4. 사용자: `수익 잘 나는 전략으로 바꿔줘`
   비서: `현재 전략군에서 손익비, 승률, 최대낙폭 기준으로 비교표를 만들었어요. 가장 안정적인 상위 3개 수정안을 보여드릴게요.`

#### 백테스트 / 검토

5. 사용자: `이 전략 괜찮아?`
   비서: `백테스트 관점에서 검토해볼게요. 승률, 기대값, 최대낙폭, 거래 빈도, 최근 구간 성능을 요약해드릴게요.`

6. 사용자: `지난 6개월 기준으로 테스트해줘`
   비서: `지난 6개월 구간으로 백테스트 설정을 준비했어요. 실행 후 요약 리포트와 주의할 왜곡 포인트를 같이 보여드릴게요.`

7. 사용자: `실전 돌려도 될까?`
   비서: `바로 실전 대신 드라이런 체크리스트를 준비했어요. 최근 성능, 슬리피지 가정, 손실 한도, 충돌 규칙을 먼저 점검해볼게요.`

#### 비서 브리핑 / 상태 확인

8. 사용자: `오늘 뭐 사?`
   비서: `현재 활성 전략 기준으로 조건에 가까운 후보들을 보여드릴게요. 조건 충족률, 리스크 상태, 진입까지 남은 조건을 정리했습니다.`

9. 사용자: `오늘 장 조심해야 할 거 알려줘`
   비서: `보유종목과 관심종목 기준으로 오늘 주의 이벤트를 정리했어요. 실적 발표, 손절선 근접, 섹터 쏠림, 변동성 급증 순으로 보여드릴게요.`

10. 사용자: `내 계좌 상태 요약해줘`
    비서: `보유 비중, 평가손익, 현금 비중, 손실 한도 대비 상태를 요약했어요. 오늘 새로 주의할 항목도 같이 붙였습니다.`

11. 사용자: `삼성 지금 어때?`
    비서: `삼성전자의 현재 전략 조건 충족률, 최근 변동성, 내 보유 여부, 설정한 진입 규칙과의 거리만 정리해드릴게요.`

#### 포트폴리오 / 리밸런싱

12. 사용자: `포트폴리오 재분배해줘`
    비서: `저장된 목표 비중 규칙 기준으로 현재 드리프트를 계산했어요. 리밸런싱 초안과 예상 거래 비용을 보여드릴게요.`

13. 사용자: `현금 비중 좀 늘리게 정리해줘`
    비서: `현재 규칙 안에서 현금 비중 목표를 높이는 리밸런싱 초안을 만들었어요. 변경 후 예상 비중과 매매 건수를 같이 확인해보세요.`

14. 사용자: `반도체 비중 줄이는 전략으로 바꿔`
    비서: `섹터 최대 비중 규칙을 조정한 포트폴리오 관리 초안을 만들었어요. 적용 전 시뮬레이션 결과를 먼저 볼까요?`

#### 주문 / 실행 관련

15. 사용자: `삼성 100주 사`
    비서: `직접 주문 대신 주문 초안을 만들 수 있어요. 현재 잔고, 손실 한도, 중복 주문 여부를 점검해서 초안으로 보여드릴게요.`

16. 사용자: `조건 오면 알아서 사`
    비서: `자동 실행 전략으로 바꿔드릴게요. 진입 조건, 최대 수량, 손실 한도, 시간대를 포함한 전략 초안을 저장하겠습니다.`

17. 사용자: `오늘 엔진 켜`
    비서: `오늘 사용할 전략과 리스크 한도를 확인했어요. 엔진을 무장 상태로 전환할까요?`

18. 사용자: `외부 주문 들어오면 멈춰`
    비서: `외부 주문 감지 시 엔진 자동 정지 규칙을 추가했어요. 경고만 할지, 즉시 정지할지도 선택할 수 있어요.`

#### 설명 / 복기 / 운영

19. 사용자: `왜 이 주문 막혔어?`
    비서: `차단 사유를 정리했어요. 손실 한도, 전략 잠금, 중복 주문, 허용 수량 초과 여부를 순서대로 보여드릴게요.`

20. 사용자: `오늘 매매 복기해줘`
    비서: `오늘 주문, 체결, 규칙 위반, 아쉬운 진입/청산을 정리했어요. 내일 수정할 전략 포인트도 초안으로 남겨둘게요.`

### UX 원칙 한 줄

- 비서는 `불가능합니다`로 끝내지 않고, 사용자의 목적을 가장 가까운 `안전한 초안/계산/저장/검토 작업`으로 자동 변환해야 한다.
