## 1. 배경

현재 StockVision의 실제 연결된 프론트는 다음 화면을 중심으로 동작한다.

- 메인 대시보드
- 종목 상세
- 온보딩
- 설정

이 프론트는 `종목 중심 싱글뷰 + 로컬 실행 기반 신뢰형 자동매매 도구`라는 방향이 명확하다.
반면 아래 영역은 아직 부족하다.

- 운영 상태를 한눈에 읽는 제어판성
- 차트와 자동매매 이벤트의 연결성
- 규칙을 이해하고 수정하는 구조화된 UX
- 온보딩 마지막 구간의 설치/연결 피드백
- 향후 전략 빌더로 자연스럽게 확장되는 흐름

본 PRD는 벤치마크 문서를 바탕으로, 현재 연결된 프론트 기준의 UX 개편 우선순위를 정의한다.

## 2. 문제 정의

### 2.1 사용자가 현재 겪는 문제

1. 메인 화면에서 "자동매매가 지금 어떤 상태인지"를 빠르게 읽기 어렵다.
2. 종목 상세에서 차트는 볼 수 있지만, 신호/규칙/체결이 차트와 분리되어 있다.
3. 규칙은 수정할 수 있지만 조건, 실행, 리스크, 최근 결과가 한 구조로 보이지 않는다.
4. 온보딩은 단계는 좋지만 설치형 제품 특유의 불안감을 해소하기엔 상태 피드백이 부족하다.
5. 설정 화면은 정보는 있으나 운영 제어판 느낌이 약하다.

### 2.2 제품 관점의 문제

1. StockVision의 차별점인 `로컬 실행`, `책임 분리`, `신뢰 UX`가 헤더와 설정에서 충분히 브랜드화되지 않았다.
2. 차트 UX가 존재하지만 TradingView급 해석력으로 이어지지 않아 제품의 전문성이 약하게 느껴진다.
3. 미래 전략 빌더를 붙일 진입점이 현재 메인/상세 화면에 아직 설계되어 있지 않다.

## 3. 목표

### 3.1 제품 목표

- StockVision을 `신뢰 가능한 자동매매 운용 콘솔`로 느끼게 만든다.
- 메인 화면과 상세 화면에서 자동매매 상태를 차트/규칙/체결 맥락 안에서 연결해 보여준다.
- 미래 전략 빌더가 붙더라도 현재 IA가 깨지지 않도록 진입 흐름을 설계한다.

### 3.2 사용자 목표

- 지금 엔진이 안전하게 돌아가는지 3초 안에 알 수 있다.
- 특정 종목에서 왜 신호가 났고 최근 무엇이 실행됐는지 한 화면에서 이해할 수 있다.
- 온보딩 중 어디에서 막혔는지 즉시 파악할 수 있다.

## 4. 비목표

- 레거시 프론트 전체 개편
- 어드민 UX 재설계
- 전체 전략 DSL 엔진 재설계
- 모바일 전용 별도 UI 구축

## 5. 대상 사용자

### 5.1 1차 사용자

- 국내 주식 자동매매 입문자
- 키움/KIS 연동에 거부감은 없지만 코딩은 원하지 않는 사용자
- 실전보다 먼저 모의/관찰 중심으로 운용 상태를 확인하고 싶은 사용자

### 5.2 2차 사용자

- 전략 조건을 자주 조정하는 파워 유저
- 규칙과 실행 로그를 더 구조적으로 보고 싶은 사용자

## 6. 핵심 UX 원칙

- 상태는 감추지 말고 상시 노출한다.
- 차트는 단순 조회가 아니라 설명 중심 화면이어야 한다.
- 규칙은 코드보다 먼저 구조로 이해되어야 한다.
- 온보딩은 설치 절차가 아니라 신뢰 형성 과정이어야 한다.
- 미래 전략 빌더는 별도 세계가 아니라 상세 화면의 확장으로 붙어야 한다.

## 7. 우선순위

## P1. 운영 패널 강화

### 목적

메인 헤더를 단순 유틸리티가 아니라 운용 제어판으로 격상한다.

### 변경 범위

- 메인 헤더
- 상태 드롭다운
- 메인 리스트 상단 요약

### 요구사항

- `로컬`, `브로커`, `클라우드`, `엔진`, `실전/모의` 상태를 한 패널에 노출한다.
- 현재 상태를 색상만이 아니라 텍스트로도 설명한다.
- 오늘의 신호 수, 최근 체결 수, 최근 오류 수를 요약 노출한다.
- 드롭다운은 `운영 패널`이라는 개념이 읽히도록 섹션 구조를 강화한다.

### 참고 제품

- Stoic: 신뢰형 상태 표현
- 3Commas: 자동화 운영 배지

### 수용 기준

- 사용자가 메인 헤더만 보고 현재 운용 가능 상태를 파악할 수 있다.
- 상태가 1개라도 비정상일 때 원인이 텍스트로 보인다.

## P1. 상세 차트 워크벤치화

### 목적

상세 화면을 단순 종목 조회가 아니라 `차트 기반 자동매매 해석 화면`으로 만든다.

### 변경 범위

- 종목 상세 차트
- 차트 보조 탭
- 차트 내 마커

### 요구사항

- 차트에 매수/매도/트리거/실패 이벤트 마커를 표시한다.
- 차트 하단에 `지표`, `전략`, `체결`, `컨텍스트` 탭을 둔다.
- 최근 규칙 트리거 시점과 체결 시점을 차트와 연결한다.
- 현재 선택 종목의 실행 히스토리가 차트와 함께 읽히도록 한다.

### 참고 제품

- TradingView: 차트 중심 해석 UX
- Composer: 전략과 성과 연결성

### 수용 기준

- 사용자는 상세 화면에서 "왜 이 종목이 매수/매도되었는지"를 추론할 수 있다.
- 차트 밖 표를 보지 않고도 최근 실행 이벤트를 따라갈 수 있다.

## P1. 규칙 카드 구조화

### 목적

상세 화면의 규칙을 `빠른 편집`에서 `구조적 이해 + 빠른 편집`으로 끌어올린다.

### 변경 범위

- 규칙 카드
- 인라인 편집
- 규칙 메타 정보

### 요구사항

- 규칙 카드를 `조건`, `실행`, `리스크`, `최근 결과` 4개 블록으로 나눈다.
- 규칙 활성 상태 외에 최근 실행 성공/실패를 보여준다.
- 최근 실패 이유가 있으면 카드에서 바로 확인 가능해야 한다.
- 편집 진입 전에도 조건 요약 문장이 읽혀야 한다.

### 참고 제품

- Composer: 전략 구조 노출
- 3Commas: 운영 가능성 중심 표현

### 수용 기준

- 규칙 상세를 열지 않고도 무엇을 하는 규칙인지 이해할 수 있다.
- 실패한 규칙이 활성 규칙 사이에 묻히지 않는다.

## P1. 온보딩 신뢰 강화

### 목적

설치형 자동매매 제품 특유의 불안감을 낮추고, 단계별 완료 확신을 높인다.

### 변경 범위

- 온보딩 단계 카드
- 위험고지
- 브릿지 설치 단계

### 요구사항

- 각 단계마다 `왜 필요한지`, `완료 조건`, `다음 단계에서 얻는 것`을 보여준다.
- 브릿지 설치 단계는 `다운로드`, `실행`, `연결 감지` 상태를 분리해 노출한다.
- 실패 시 재시도와 도움말 링크가 보여야 한다.
- 위험고지에서 `로컬 실행`, `비밀번호 미보관`, `사용자 책임`을 더 선명하게 시각화한다.

### 참고 제품

- Stoic: 전략 선택 전 신뢰 형성
- 3Commas: 연결 단계 시각화

### 수용 기준

- 사용자는 각 단계가 무엇 때문에 필요한지 설명할 수 있다.
- 브릿지 단계에서 막혀도 다음 액션이 즉시 보인다.

## P1. 미래 전략 빌더 진입 구조 설계

### 목적

향후 전략 빌더가 별도 제품처럼 보이지 않도록 현재 IA에 자연스럽게 붙일 준비를 한다.

### 변경 범위

- 상세 화면 CTA
- 규칙 추가 플로우
- 미래 전략 빌더 IA

### 요구사항

- `이 종목 전략 추가`를 상세 화면의 자연스러운 다음 단계로 둔다.
- 미래 전략 빌더는 위저드형 구조를 기본으로 한다.
- 흐름은 `목표 -> 진입 -> 청산 -> 수량 -> 리스크 -> 검증 -> 활성화` 순서를 따른다.
- 고급 사용자를 위해 DSL/JSON 보기 모드는 유지하되 기본은 문장형/구조형 편집기로 제공한다.

### 참고 제품

- Composer: 시각 전략 편집
- 3Commas: 자동화 설정 흐름
- TradingView: 차트 옆 튜닝 경험

### 수용 기준

- 사용자는 별도 교육 없이 전략 추가 흐름을 끝까지 진행할 수 있다.
- 빌더 진입 전후의 맥락이 메인/상세 화면과 끊기지 않는다.

## P2. 설정 화면 제어판화

### 목적

설정 화면을 정보 표시 페이지가 아니라 운용 환경 제어판으로 만든다.

### 변경 범위

- 설정 정보 구조
- 섹션 그룹화
- 최근 이벤트/로그 미리보기

### 요구사항

- `계정`, `브로커`, `엔진`, `보안` 4개 섹션으로 분리한다.
- API 키 저장 상태와 실전/모의 상태를 더 선명하게 보여준다.
- 최근 엔진 이벤트 또는 상태 변경 로그를 짧게 미리 보여준다.

### 참고 제품

- Stoic: 단순하고 신뢰감 있는 설정 UX
- QuantConnect: 명확한 정보 구조

### 수용 기준

- 설정 화면만으로 현재 연결 환경을 점검할 수 있다.
- 사용자는 브로커 상태와 엔진 상태를 별도로 혼동하지 않는다.

## P2. 실행 로그 타임라인화

### 목적

향후 실행 로그를 운영 신뢰의 핵심 화면으로 만든다.

### 변경 범위

- 실행 로그 정보 구조
- 상태 단계 모델
- 상세 패널

### 요구사항

- 실행 단계를 `Triggered -> Submitted -> Filled/Failed/Cancelled`로 표현한다.
- 로그 목록은 표만이 아니라 타임라인 해석이 가능해야 한다.
- 실패 이유, 브로커 응답, 재시도 여부를 표시한다.

### 참고 제품

- 3Commas: 운영 관점 로그
- QuantConnect: 고급 사용자 정보 밀도

### 수용 기준

- 사용자는 실패 로그를 보고 원인을 분류할 수 있다.
- 체결 여부와 주문 제출 여부를 혼동하지 않는다.

## 8. 출시 순서 제안

### Phase 1

- 운영 패널 강화
- 온보딩 신뢰 강화
- 브릿지 설치 피드백 개선

### Phase 2

- 상세 차트 워크벤치화
- 규칙 카드 구조화

### Phase 3

- 미래 전략 빌더 진입 구조 설계
- 설정 화면 제어판화
- 실행 로그 타임라인화

## 9. 성공 지표

정량 지표:
- 온보딩 단계 이탈률 감소
- 브릿지 설치 단계 완료율 증가
- 상세 화면 체류 시간 증가
- 규칙 수정 후 저장 완료율 증가

정성 지표:
- "지금 시스템이 안전하게 돌고 있는지 알겠다"는 피드백 증가
- "왜 이 주문이 발생했는지 이해된다"는 피드백 증가
- "설치형이라 불안했는데 상태가 잘 보인다"는 피드백 증가

## 10. 오픈 질문

- 메인 헤더 운영 패널에서 클라우드 상태를 어떤 데이터 소스로 가장 안정적으로 표시할 것인가
- 상세 차트 이벤트 마커를 어느 수준까지 실시간 반영할 것인가
- 미래 전략 빌더를 상세 화면 내부 플로우로 붙일지, 전용 화면으로 분리할지
- 실행 로그의 단계 모델을 현재 엔진 상태 모델과 어떻게 맞출 것인가

## 11. 한 줄 요약

이번 UX 개편의 목표는 `StockVision을 종목 조회 화면에서 신뢰 가능한 자동매매 운용 콘솔로 끌어올리는 것`이다.
