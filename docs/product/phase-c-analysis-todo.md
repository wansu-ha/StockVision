# Phase C 분석 목록

Phase C spec 재설계를 위한 분석 항목. 구현 전에 이 분석을 먼저 완료.

## A. 상태별 UI 매트릭스

사용자 상태에 따라 어떤 UI를 보여줄지 전수 조사.

### A-1. 상태 정의
| 상태 | 조건 |
|------|------|
| 미가입 | 계정 없음 |
| 가입만 | 로그인 O, 로컬 서버 X, 증권사 X |
| +로컬 서버 | 로그인 O, 로컬 서버 O, 증권사 X |
| +증권사 | 로그인 O, 로컬 서버 O, 증권사 O, 전략 X |
| 완전체 | 로그인 O, 로컬 서버 O, 증권사 O, 전략 O, 엔진 실행 중 |

### A-2. 각 상태에서 가능/불가능한 기능
- 종목 검색/차트 (클라우드 yfinance → 가입만으로 가능)
- 전략 생성/편집 (클라우드 저장 → 가입만으로 가능?)
- 실시간 시세 (로컬 WS → 로컬 서버 필요)
- 계좌 잔고/수익률 (증권사 키 필요)
- 자동매매 실행 (전략 + 증권사 + 로컬 서버)
- 체결 로그 (증권사 + 로컬 서버)

### A-3. 현재 컴포넌트별 데이터 소스 매핑
조사 대상:
- `MainDashboard.tsx` — 어떤 카드가 어떤 API를 호출하는지
- `OpsPanel.tsx` — 증권사 의존 데이터 vs 독립 데이터
- `DetailView.tsx` — 종목 상세에서 뭘 보여주는지
- `Settings.tsx` — 증권사 연동 상태 표시
- `ExecutionLog.tsx` / `ExecutionTimeline.tsx` — 데이터 소스

## B. 온보딩 재설계

### B-1. 첫 번째 성공 경험 정의
- "성공 경험"이란? → 후보: 차트 보기, 전략 만들기, 백테스트, 첫 자동 주문
- 증권사 없이 도달할 수 있는 가장 가치 있는 경험은?

### B-2. 온보딩 순서 재검토
현재: 가입 → 로컬 설치 → 증권사 → 위험고지 → 대시보드
제안: 가입 → (증권사 없이) 체험 → 설치 동기 부여 → 로컬 설치 → 증권사 → 실전
- "체험"에 뭘 넣을 수 있는지 구체화 필요

### B-3. 상태별 CTA (Call-to-Action)
각 불완전 상태에서 "다음에 뭘 해야 하는지" 안내:
- 증권사 미연동 → "증권사 API 키를 등록하세요" + 이유
- 로컬 서버 미설치 → "로컬 서버를 설치하세요" + 이유
- 전략 없음 → "첫 전략을 만들어보세요"

### B-4. 음영/잠금 UI 패턴
증권사 미연동 시 카드 정보 음영 처리:
- 어떤 카드/섹션이 증권사 의존인지 매핑 (A-3과 연동)
- 음영 + 안내 멘트 디자인
- 카드 내 혼합 정보 분리 필요 여부 확인

## C. 운영 대시보드 심화

### C-1. 실제 운영 시나리오
- 정상 운영 중 봐야 하는 것
- 이상 발생 시 봐야 하는 것
- 하루 종료 후 리뷰 시 봐야 하는 것

### C-2. 알림 정책
- 어떤 이벤트에 알림이 필요한지 (체결, 오류, 연결 끊김 등)
- 알림 채널: 토스트? 뱃지? 소리?

## D. 원격 제어 상세

### D-1. 릴레이 아키텍처
- 클라우드가 로컬에 명령을 전달하는 방식 (하트비트 응답에 명령 포함? 별도 채널?)
- 오프라인 시 대기열 처리

### D-2. 재개 UX
- 상태 요약 + 체크박스 확인 (2단계 코드 입력 대신)
- 정지 사유, 브로커 상태, 미체결 건수 표시

## E. 백테스트 시점 검토

### E-1. 온보딩 체험용 간이 백테스트?
- 증권사 없이 yfinance 과거 데이터로 전략 시뮬레이션
- 구현 난이도 vs 온보딩 효과
- Phase C에 포함할지 별도 Phase로 뺄지

## F. 전략 생성 UX

### F-1. 증권사 없이 전략을 만들 수 있는가?
- 현재 StrategyBuilder가 증권사에 의존하는 부분이 있는지
- 전략 = 클라우드 저장이면 가입만으로 가능해야 함

### F-2. 전략 → 실전 전환 흐름
- 전략을 만들었는데 증권사가 없으면? → "실전 배포하려면 증권사를 연동하세요"
- 모의투자로 먼저 테스트 → 실전 전환 흐름
