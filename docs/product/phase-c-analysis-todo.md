# Phase C 분석 목록

Phase C spec 재설계를 위한 분석 항목. 구현 전에 이 분석을 먼저 완료.

## A. 상태별 UI 매트릭스

사용자 상태에 따라 어떤 UI를 보여줄지 전수 조사.

### A-1. 상태 정의
| 상태 | 조건 |
|------|------|
| 미가입 | 계정 없음 |
| 가입만 | 로그인 O, 로컬 서버 X, 증권사 X |
| +로컬 서버 | 로그인 O, 로컬 서버 O, 증권사 X |
| +증권사 | 로그인 O, 로컬 서버 O, 증권사 O, 전략 X |
| 완전체 | 로그인 O, 로컬 서버 O, 증권사 O, 전략 O, 엔진 실행 중 |

### A-2. 각 상태에서 가능/불가능한 기능 매트릭스
모든 기능을 나열하고 각 상태에서 가능 여부를 O/X/△(제한적)로 표기:
- 종목 검색
- 차트 보기 (일봉/분봉)
- 전략 생성/편집/삭제
- 전략 목록 조회
- 전략 로컬 배포
- 실시간 시세 (WS)
- 계좌 잔고 조회
- 계좌 수익률 조회
- 보유종목 목록
- 자동매매 엔진 시작/정지
- 체결 로그 조회
- 설정 변경
- 증권사 키 등록/삭제
- AI 분석 (향후)

### A-3. 현재 컴포넌트별 데이터 소스 매핑
각 컴포넌트가 어떤 API(cloud/local)를 호출하는지 코드 분석:
- `MainDashboard.tsx` — 렌더링하는 카드/섹션 목록, 각각의 데이터 소스
- `OpsPanel.tsx` — 4개 상태 표시기, 각각 어디서 데이터를 가져오는지
- `DetailView.tsx` — 종목 상세 내 섹션별 데이터 소스
- `PriceChart.tsx` — 시세 데이터 출처 (cloud? local? yfinance?)
- `RuleCard.tsx` — 규칙 데이터 출처, 결과 데이터 출처
- `RuleList.tsx` — 규칙 목록 출처
- `StrategyList.tsx` — 전략 목록 출처
- `StrategyBuilder.tsx` — 저장 대상 (cloud API), 배포 대상 (local API)
- `Settings.tsx` — 각 설정 항목의 데이터 소스
- `ExecutionLog.tsx` / `ExecutionTimeline.tsx` — 로그 데이터 출처
- `BridgeInstaller.tsx` — 로컬 서버 감지 로직
- `TrafficLightStatus.tsx` — 3개 상태의 데이터 소스
- `Header.tsx` — 검색, 엔진 제어 버튼의 API 대상
- `StockSearch.tsx` — 검색 API 출처

### A-4. 각 API 엔드포인트가 어떤 사전조건을 필요로 하는지
- `cloudClient.ts`의 모든 API 호출 목록 + 필요 조건 (JWT만? 추가 조건?)
- `localClient.ts`의 모든 API 호출 목록 + 필요 조건 (로컬 서버 실행? 증권사 연동?)
- `logs.ts`의 API 호출 목록

### A-5. 현재 에러 처리 현황
- 로컬 서버 미연결 시 각 컴포넌트가 어떻게 반응하는지
- 증권사 미연동 시 API가 어떤 에러를 반환하는지
- 프론트엔드가 에러를 어떻게 표시하는지 (빈 화면? 에러 메시지? 무한 로딩?)

## B. 온보딩 재설계

### B-1. 첫 번째 성공 경험 정의
- "성공 경험" 후보 목록:
  - 차트 보기 (가장 쉬움, 가입만으로)
  - 전략 만들기 (가입만으로 가능해야 함)
  - 백테스트 돌리기 (미구현)
  - 모의투자 체결 (로컬 서버 + mock 브로커)
  - 실전 첫 체결 (전체 셋업 완료)
- 각 후보의 도달 난이도, 소요 시간, 사용자 만족도 비교

### B-2. 온보딩 순서 재검토
현재: 가입 → 로컬 설치 → 증권사 → 위험고지 → 대시보드
제안: 가입 → (증권사 없이) 체험 → 설치 동기 부여 → 로컬 설치 → 증권사 → 실전
- "체험"에 넣을 수 있는 것: 차트, 전략 생성, 샘플 전략 백테스트?
- 각 단계에서 이탈 가능성 평가
- "왜 로컬 서버를 깔아야 하는지" 메시지

### B-3. 상태별 CTA (Call-to-Action) 설계
각 불완전 상태에서 "다음에 뭘 해야 하는지" 안내 설계:
- 위치: 배너? 카드 내부? 모달?
- 메시지 톤: 강제적? 부드러운 안내?
- 닫기/나중에 가능 여부
- 재방문 시 동일 CTA 반복 여부

### B-4. 음영/잠금 UI 패턴
증권사 미연동 시 카드 정보 음영 처리:
- 어떤 카드/섹션이 증권사 의존인지 매핑 (A-3 결과 활용)
- 음영 + 안내 멘트 디자인 패턴 조사 (Figma 커뮤니티, Material Design 등)
- 카드 내 혼합 정보 분리 필요 여부 (A-3에서 혼합 데이터가 있는 카드 식별)
- 잠금 해제 조건별 아이콘/색상 통일

### B-5. 온보딩 위저드 vs 인라인 온보딩
- 위저드 (별도 페이지 플로우): 집중적, 하지만 강제적
- 인라인 (대시보드 내 가이드): 자연스럽, 하지만 놓칠 수 있음
- 하이브리드: 첫 방문은 위저드, 이후는 인라인 CTA
- 현재 구현(`OnboardingWizard.tsx`)의 구조 분석

### B-6. "왜 로컬인가" 메시지 검증
- 사용자에게 전달할 핵심 메시지 후보:
  - "증권사 키가 우리 서버를 거치지 않습니다"
  - "주문은 당신의 PC에서만 나갑니다"
  - "인터넷 끊겨도 기존 주문은 유지됩니다"
- 경쟁사 대비 차별점으로 쓸 수 있는 포인트

## C. 운영 대시보드 심화

### C-1. 실제 운영 시나리오별 필요 정보
**정상 운영 중 (장중)**:
- 엔진 상태 (실행 중/대기/정지)
- 오늘 신호 수, 체결 수, 오류 수
- 현재 보유종목 + 평가손익
- 미체결 주문 수
- 브로커 연결 상태
- 실시간 시세 수신 상태

**이상 발생 시**:
- 어떤 이상인지 (브로커 끊김? 엔진 오류? 네트워크?)
- 마지막 정상 시점
- 영향받은 주문/규칙
- 복구 방법 안내

**장 종료 후 리뷰**:
- 오늘 체결 내역 요약
- 규칙별 성과 (어떤 규칙이 체결됐는지)
- 오류/실패 내역
- 계좌 변동 요약

**장 시작 전 확인**:
- 모든 연결 정상인지
- 활성 규칙 수, 대상 종목 수
- 예산 잔여량

### C-2. 알림 정책
- 알림이 필요한 이벤트 목록:
  - 체결 성공/실패
  - 브로커 연결 끊김/복구
  - 엔진 오류
  - 킬스위치 발동
  - 일일 한도 도달
  - 손절 발생
- 알림 채널별 적합성:
  - 브라우저 토스트: 앱 사용 중
  - Windows 토스트 (트레이): 앱 미사용 중
  - (향후) 텔레그램/카카오: 외출 중
- 알림 빈도 제어 (같은 알림 반복 방지)

### C-3. 대시보드 레이아웃 재검토
- 현재 `OpsPanel` + `ListView` + `DetailView` 구조
- 운영 정보와 종목 정보의 비중
- 모바일 반응형 고려

## D. 원격 제어 상세

### D-1. 릴레이 아키텍처 설계
- 방식 후보:
  - (a) 하트비트 응답에 명령 포함 (폴링, 지연 있음)
  - (b) 클라우드→로컬 WebSocket (항시 연결, 복잡)
  - (c) 클라우드에 명령 큐, 로컬이 폴링 (중간)
- 각 방식의 지연시간, 구현 복잡도, 안정성 비교
- 오프라인 시 동작: 명령 대기열? 타임아웃?
- 보안: 명령 위조 방지 (서명?), 재전송 공격 방지

### D-2. 재개 UX 확정
- 상태 요약 화면 설계:
  - 정지 시각, 정지 사유 (수동/자동/오류)
  - 현재 브로커 상태, 미체결 건수, 마지막 체결 시각
  - "현재 상태를 확인했습니다" 체크박스
  - "재개" 버튼
- 자동 정지 사유별 재개 조건:
  - 수동 킬스위치 → 체크박스만
  - 오류로 자동 정지 → 오류 해소 확인 후 재개
  - 일일 한도 도달 → 다음 날 자동 해제? 수동?

### D-3. 원격 조회 범위
- 클라우드에 캐싱할 정보:
  - 로컬 상태 스냅샷 (하트비트마다 갱신)
  - 오늘 체결 로그 요약
  - 계좌 요약 (잔고, 수익률) — 개인정보 이슈?
- 클라우드에 안 올려야 하는 정보:
  - 증권사 키
  - 상세 체결 내역 (개인 금융정보)
- "개인정보 로컬만" 원칙과의 충돌 검토

## E. 백테스트 시점 검토

### E-1. 온보딩 체험용 간이 백테스트
- yfinance 과거 데이터로 전략 시뮬레이션
- 최소 구현 범위: 전략 1개 + 종목 1개 + 기간 선택 → 수익률 차트
- 구현 위치: 클라우드? 프론트엔드? 로컬?
  - 클라우드: 서버 부하, 하지만 가입만으로 가능
  - 프론트엔드: WASM? 무거움
  - 로컬: 설치 필요 → 온보딩 체험 목적에 안 맞음
- Phase 2 백테스트 코드 재활용 가능성

### E-2. 본격 백테스트 (Phase D 후보)
- 전 종목 스크리닝 + 기간 백테스트
- 데이터 소스: yfinance? 증권사 API? 별도 데이터 제공업체?
- 결과 저장/비교
- 이건 Phase C 범위 밖이므로 메모만

## F. 전략 생성 UX

### F-1. 현재 전략 생성 흐름 분석
- `StrategyBuilder.tsx` 코드 분석:
  - 어떤 필드를 입력하는지
  - 저장 시 어디로 보내는지 (cloudClient? localClient?)
  - 증권사에 의존하는 필드가 있는지
- `StrategyList.tsx` 코드 분석:
  - 전략 목록 출처
  - 로컬 배포 상태 표시
  - 배포/해제 동작

### F-2. 증권사 없이 전략 만들기
- 전략 정의(규칙 DSL)는 증권사와 무관해야 함
- "종목 선택"은 증권사 없이 가능? (클라우드 종목 DB)
- "배포"만 증권사+로컬 필요
- 전략 생성 → 저장 → (나중에) 배포 흐름이 자연스러운지

### F-3. 전략 → 실전 전환 흐름
- 모의투자로 먼저 테스트 (mock broker)
- 모의 → 실전 전환 시 확인 사항 (위험고지, 예산 설정)
- 실전 전환 UI

### F-4. 샘플/템플릿 전략
- "처음 만드는 사용자"를 위한 샘플 전략 제공
- "RSI 30 이하 매수, 70 이상 매도" 같은 기본 전략
- 사용자가 복사 후 수정 가능

## G. 시세 데이터 아키텍처

### G-1. 현재 시세 데이터 흐름
- 클라우드: yfinance로 실시간 조회, 캐싱 없음
- 로컬: 증권사 WS 실시간 수신 (키움 0B)
- 프론트: 어디서 시세를 가져오는지 (cloud? local?)
- IndicatorProvider: yfinance 일봉 → 지표 계산

### G-2. 클라우드 시세 캐싱 설계
- 캐싱 대상: 일봉 (수십 년), 분봉 (최근 N개월?)
- 저장소: PostgreSQL? 별도 시계열 DB?
- 갱신 주기: 장중 실시간? 장 마감 후 일괄?
- LLM 분석용 데이터 포맷: 어떤 형태로 Claude에 넘길지
- 용량 예측: 코스피+코스닥 전 종목 일봉 = 대략 몇 GB?

### G-3. 시세 데이터 모델 (DB 스키마)
- 현재 cloud_server의 DB 모델 중 시세 관련 확인
- `StockPrice`, `DailyBar` 같은 모델이 있는지
- 없으면 설계 필요

## H. 인증/권한 흐름

### H-1. 현재 인증 흐름 전수 조사
- 가입 → 로그인 → JWT 발급 → 로컬 전달 흐름
- 토큰 갱신 (refresh) 흐름
- 로컬 서버의 cloud token 사용처

### H-2. 증권사 없이 동작하는 인증
- 현재: 클라우드 JWT 없으면 로컬 서버가 규칙 sync 불가
- 오픈소스 독립 동작 시: JWT 없이도 로컬 CRUD 가능하게?
- 이건 open-questions와 연동
