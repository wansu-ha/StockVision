# 미결 논의사항

Phase C 이전에 방향을 잡아야 하는 것들. 급하지 않지만, M3(레포 분리) 전에는 결정 필요.

## 1. 오픈소스 독립성 — 로컬 서버가 클라우드 없이 동작해야 하는가?

**현재 문제**: 로컬 서버는 클라우드 JWT 없이 규칙 동기화 불가. 오픈소스 사용자가 클라우드 없이 쓸 수 없음.

**선택지**:
- A) 로컬에 규칙 CRUD API 추가 → 클라우드 없이 독립 동작, 클라우드는 부가 기능
- B) 클라우드 셀프 호스팅 허용 → 비공개의 의미 퇴색
- C) API 프레임워크로 포지셔닝 → "엔진 라이브러리, UI는 직접 만드세요"

**영향**: 이 결정에 따라 로컬 서버 아키텍처, 인증 흐름, 레포 분리 범위가 달라짐.

## 2. 공개 시 UI 포함 여부

**현재 문제**: stockvision-local을 공개할 때 UI를 뭘 넣을지 미정.

**선택지**:
- A) 미니 웹 UI를 localhost에서 서빙 (FastAPI StaticFiles)
- B) CLI 도구만 제공
- C) API 문서만 공개, UI 없음

**의존**: #1에서 A를 선택하면 미니 UI가 자연스럽고, C를 선택하면 UI 불필요.

## 3. 오픈소스의 실제 타겟 사용자

**질문**: 이 오픈소스를 가져다 쓸 사람은 누구인가?
- 한국 주식 자동매매를 Python으로 하려는 개발자 (작은 풀)
- 브로커 어댑터를 확장하려는 해외 개발자 (BrokerAdapter ABC)
- 포트폴리오/신뢰 목적의 공개 (사용자 없어도 가치 있음)

**영향**: 타겟에 따라 문서 수준, API 설계, 확장성 투자량이 달라짐.

## 4. roadmap M1 재정의

**결정된 것**: M1(프론트 경계 리팩터링) 사실상 불필요. 어드민 lazy import 정도만.

**남은 결정**: M1을 삭제하고 M3 직전에 통합할지, 아니면 축소된 M1을 유지할지.

## 5. repo-mapping-table 업데이트

**결정된 것**:
- "Local UI 분리" 항목 삭제 (사용자 앱에서 로컬 호출은 정상)
- stockvision-web 레포 삭제 (비공개 유지, 별도 레포 불필요)
- 어드민은 물리적 분리 불필요

**남은 작업**: `docs/open-source/repo-mapping-table.md` 문서에 반영.
