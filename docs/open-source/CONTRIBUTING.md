# Contributing Guide (초안)

StockVision에 관심을 가져주셔서 감사합니다.
이 문서는 오픈소스 공개를 가정한 기여 가이드 초안입니다.
이 프로젝트는 1인 유지관리 체계로 운영되므로, 모든 기여가 즉시 리뷰되거나 병합되지는 않습니다.

## 시작하기 전에

먼저 아래 문서를 읽어 주세요.

- `README`
- `SECURITY.md`
- `SUPPORT.md`
- `OPEN_SOURCE_SCOPE.md`
- `TRADEMARKS.md`

보안 취약점은 PR이나 공개 issue로 올리지 말고 `SECURITY.md`의 비공개 채널로 제보해 주세요.

## 프로젝트 방향

StockVision은 아래 방향을 우선합니다.

- 로컬 실행 엔진의 투명성과 안전성
- 사용자가 직접 정의한 규칙의 실행 자동화
- 클라우드 의존 없이도 핵심 거래 기능이 동작하는 구조
- 유지관리 가능한 단순성

아래 제안은 신중하게 다룹니다.

- 자유형 자연어 주문 기능
- 과도하게 공격적인 자동매매 기능
- 법적 포지셔닝을 흐릴 수 있는 기능
- 유지관리 비용이 큰 브로커 확장

## 기여 전 원칙

1. 큰 변경은 먼저 issue로 방향을 맞춰 주세요.
2. 작은 버그 수정은 바로 PR을 열어도 됩니다.
3. 프로젝트 방향과 맞지 않는 기능은 병합되지 않을 수 있습니다.
4. 유지관리 비용이 큰 설계는 거절될 수 있습니다.

## 개발 원칙

- 변경은 가능한 한 작고 명확하게 유지해 주세요.
- 한 PR에는 하나의 목적만 담아 주세요.
- 회귀를 막는 테스트를 함께 추가해 주세요.
- 문서 변경이 필요하면 같이 포함해 주세요.
- 사용자의 민감정보, 자격증명, 실거래 데이터가 코드나 테스트에 포함되면 안 됩니다.

## 코드 스타일

- 기존 코드 스타일과 구조를 따릅니다.
- 공개 API, 라우트, 설정 키를 바꾸는 경우 영향 범위를 설명해 주세요.
- 설명이 필요한 로직에는 짧은 주석을 추가해 주세요.
- 불필요한 리팩터링과 기능 변경을 한 PR에 섞지 말아 주세요.

## 테스트

PR에는 가능한 한 아래를 포함해 주세요.

- 변경 영역의 단위 테스트
- 계약을 깨지 않는다는 확인
- 수동 검증 방법 또는 재현 절차

테스트가 없더라도 유지관리자가 필요하다고 판단하면 병합을 보류할 수 있습니다.

## 보안 및 민감정보 규칙

아래 정보는 issue, PR, commit, 스크린샷, 테스트 픽스처에 넣지 마세요.

- API Key
- Secret
- refresh token
- 계좌번호
- 개인 식별 정보
- 실거래 주문 원본 로그

localhost 인증, WebSocket 인증, 토큰 저장, 규칙 주입, 브로커 키 보관 관련 코드는 특히 보안 민감 영역으로 봅니다.
이 영역의 변경은 더 엄격하게 리뷰될 수 있습니다.

## 법적/도메인 경계

아래와 같은 기여는 병합되지 않거나 추가 검토가 필요합니다.

- 투자 추천처럼 보이는 기능
- 수익 보장 또는 성과 오인 표현
- 증권사 약관 위반 가능성이 있는 시세 재배포 기능
- 서버에서 사용자 키를 수집하거나 처리하는 설계
- 사용자 대신 투자 판단을 내리는 구조

## 라이선스

별도 명시가 없는 한, 기여된 코드는 프로젝트 라이선스와 동일한 조건으로 배포될 수 있어야 합니다.
최종 라이선스는 프로젝트 루트의 `LICENSE` 파일을 기준으로 합니다.

타인의 코드를 가져올 때는 반드시 재사용 가능 여부와 라이선스를 확인해 주세요.
출처 불명 코드나 재배포 권한이 불명확한 에셋은 받지 않습니다.

## PR 설명에 포함해 주세요

- 무엇을 바꿨는지
- 왜 바꿨는지
- 어떤 대안을 검토했는지
- 테스트 또는 검증 방법
- 사용자 영향 또는 호환성 영향

## 유지관리자 재량

유지관리자는 아래 이유로 PR을 닫거나 보류할 수 있습니다.

- 프로젝트 방향과 맞지 않는 경우
- 유지관리 비용이 과도한 경우
- 보안 또는 법적 리스크가 큰 경우
- 테스트와 설명이 부족한 경우
- 같은 목표를 더 작은 변경으로 달성할 수 있는 경우

## 커뮤니케이션 원칙

- 친절하고 간결하게 소통해 주세요.
- 상대를 설득하기보다 사실과 재현 가능한 근거를 제시해 주세요.
- 공격적이거나 반복적으로 소모적인 커뮤니케이션은 종료될 수 있습니다.

## 한 줄 원칙

좋은 기여는 `기능을 많이 더하는 기여`보다 `안전하고 유지관리 가능한 변화를 만드는 기여`입니다.
