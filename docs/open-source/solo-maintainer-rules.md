# 1인 개발자가 감당 가능한 오픈소스 운영 규칙 초안

이 문서는 StockVision 같은 프로젝트를 1인 개발자가 무너지지 않고 운영하기 위한 현실적인 오픈소스 운영 규칙 초안입니다.
핵심 목표는 `친절함`이 아니라 `지속 가능성`입니다.

## 기본 원칙

1. 오픈소스는 공개이지, 무제한 무료 지원 약속이 아니다.
2. 유지관리자는 사용자를 돕되, 자신의 일정과 건강을 우선한다.
3. 응답하지 못하는 요청을 열어두기보다, 지원 범위를 명확히 제한하는 것이 더 친절하다.
4. 보안 이슈는 공개 이슈가 아니라 비공개 제보 흐름으로 받는다.
5. 로컬 엔진의 신뢰와 안전성은 공개하되, 운영 비밀과 클라우드 내부는 분리한다.

## 운영 목표

이 프로젝트의 오픈소스 운영 목표는 아래 정도로 제한하는 것이 현실적입니다.

- 로컬 엔진을 투명하게 공개한다.
- 설치와 실행이 가능한 최소 문서를 유지한다.
- 심각한 버그와 보안 이슈에는 대응한다.
- 외부 기여를 받을 수 있는 통로는 열어두되, 리뷰 의무는 약속하지 않는다.
- 유료 클라우드/운영 지원과 오픈소스 커뮤니티 지원을 분리한다.

## 하지 않을 일

1인 개발자는 아래를 기본적으로 약속하지 않는 편이 좋습니다.

- 24시간 이내 응답 보장
- 모든 이슈에 개별 답변
- 모든 PR 리뷰
- 사용자의 로컬 환경별 설치 문제를 전부 원격 지원
- 포크/변형판에 대한 호환성 보장
- 구버전 무기한 지원
- 투자 결과, 매매 로직, 수익률에 대한 개별 상담

이걸 문서에 명시해야 나중에 덜 소모됩니다.

## 권장 운영 규칙

### 1. 지원 범위는 좁게 적는다

권장 문구 예시:

- 이 프로젝트는 best-effort로 유지됩니다.
- 모든 이슈와 PR에 대한 응답 또는 병합을 보장하지 않습니다.
- 보안 취약점은 공개 이슈가 아니라 별도 채널로 제보해 주세요.
- 사용 환경, 증권사 정책, 네트워크 상태에 따라 동작 차이가 있을 수 있습니다.

### 2. 응답 속도는 넓게 잡는다

1인 개발자에게 맞는 현실적인 기준 예시:

- 일반 이슈 triage: 주 1회
- 보안 제보 1차 응답: 3영업일 이내 목표
- 긴급 취약점 공지: 수정 준비 후 별도 공지
- PR 리뷰: 가능할 때만, 기한 약속 없음

핵심은 `빨리 답하겠다`보다 `느려도 예측 가능하게 하겠다`입니다.

### 3. 이슈 템플릿으로 입력을 제한한다

최소한 아래 4종은 분리하는 게 좋습니다.

- 버그 리포트
- 기능 제안
- 보안 제보 안내
- 질문/지원 요청

버그 리포트에는 아래를 강제합니다.

- OS 버전
- 앱 버전
- 브로커 종류
- 재현 단계
- 기대 결과 / 실제 결과
- 민감정보 제거 여부

### 4. 보안 제보는 공개 이슈로 받지 않는다

반드시 문서화해야 할 것:

- 제보 메일 또는 비공개 제보 채널
- 어떤 정보가 필요한지
- 응답 목표 시간
- 공개 전 협의 원칙
- 지원하는 버전 범위

보안 제보자가 공개 이슈로 올렸을 때도, 유지관리자가 비공개 채널로 유도하는 원칙이 필요합니다.

### 5. 브랜치와 릴리스 규칙을 단순하게 유지한다

1인 운영에서는 복잡한 브랜치 전략이 오히려 부담입니다.

권장:

- `main`: 항상 배포 가능한 상태
- 기능 작업은 짧은 브랜치
- 릴리스는 태그 기반
- 패치 노트는 짧아도 반드시 작성

### 6. 지원 버전은 1~2개만 유지한다

권장 정책:

- 최신 안정 버전만 지원
- 필요하면 직전 minor 또는 직전 installer만 추가 지원
- 그 이전 버전은 커뮤니티 사용은 가능하되 공식 지원 없음

### 7. PR 수용 기준을 미리 적어둔다

권장 기준:

- 프로젝트 방향과 맞아야 함
- 보안/안정성을 해치지 않아야 함
- 테스트가 있어야 함
- 유지관리 비용을 과하게 늘리지 않아야 함
- 증권사 약관, 법적 리스크, 비밀정보 노출 문제가 없어야 함

### 8. 기능 요청보다 안전과 단순성을 우선한다

특히 이 프로젝트는 아래 요청을 쉽게 받게 됩니다.

- 더 공격적인 자동매매 기능
- 더 넓은 브로커 지원
- 더 많은 실시간 알림 채널
- 자유형 자연어 주문 기능

1인 운영에서는 이런 요청보다 아래 우선순위가 맞습니다.

- 주문 안전장치
- 인증/권한 검증
- 로컬 데이터 보호
- 설치 안정성
- 문서 정확성

### 9. 커뮤니티와 유료 지원을 분리한다

혼자 운영할 때 가장 중요한 경계입니다.

문서 원칙 예시:

- GitHub 이슈는 커뮤니티 지원 채널입니다.
- 응답과 해결은 보장되지 않습니다.
- 운영, 배포, 조직 맞춤 지원은 별도 계약 또는 유료 지원 범위입니다.

### 10. 유지관리자 시간을 보호하는 종료 규칙을 둔다

아래 같은 종료 기준을 적어두면 좋습니다.

- 재현 정보가 없으면 이슈를 닫을 수 있음
- 장기간 응답 없는 이슈는 닫을 수 있음
- 범위를 벗어난 지원 요청은 토론 없이 닫을 수 있음
- 공격적이거나 무례한 커뮤니케이션은 차단할 수 있음

## StockVision에 특히 중요한 추가 규칙

### 투자/매매 관련 문의 경계

반드시 선을 그어야 합니다.

- 특정 종목 추천 요청은 받지 않음
- 개별 전략의 수익성 평가를 약속하지 않음
- 손실 책임 상담을 하지 않음
- 사용자 계좌나 키를 받아서 디버깅하지 않음

### 민감정보 제출 금지

이슈/PR/로그 공유 시 금지 항목을 명시해야 합니다.

- API Key
- Secret
- refresh token
- 계좌번호
- 주문 내역 원본
- 개인 식별 정보

### 로컬 브리지 보안 우선

이 프로젝트는 localhost와 브로커 연동이 있으므로, 새로운 기능보다 아래를 우선 처리해야 합니다.

- localhost mutation 인증
- WebSocket 인증/Origin 검증
- 규칙 주입 방지
- 토큰 저장 하드닝
- 로그 민감정보 마스킹

- `README`

- `CONTRIBUTING.md`
- `SUPPORT.md`
- `CODE_OF_CONDUCT.md` 또는 최소한의 커뮤니케이션 규칙
- `DISCLAIMER.md`
- `TRADEMARKS.md`

## 권장 자동화

사람 시간을 줄이려면 자동화가 중요합니다.

최소 자동화 권장:

- 테스트 CI
- lint/typecheck
- dependency audit
- secret scan
- 릴리스 태그 자동화 또는 체크리스트
- issue template

## 현실적인 운영 선언문 예시

아래 정도 톤이 1인 개발자에게 적당합니다.

- 이 프로젝트는 1인 유지관리 체계로 운영됩니다.
- 응답과 수정은 best-effort로 제공되며, 일정 보장은 하지 않습니다.
- 보안 이슈는 공개 이슈가 아닌 비공개 채널로 제보해 주세요.
- 투자 판단, 수익 보장, 개별 전략 자문은 제공하지 않습니다.
- 민감정보가 포함된 로그나 자격증명은 제출하지 마세요.
- 프로젝트 방향과 유지관리 가능성을 기준으로 이슈와 PR을 선별합니다.

## 한 줄 결론

1인 개발자가 감당 가능한 오픈소스 운영의 핵심은 `다 받는 것`이 아니라 `무엇을 안 받는지 먼저 적는 것`입니다.
오픈소스는 코드 공개이고, 유지관리자의 시간까지 무제한 공개하는 것은 아닙니다.
