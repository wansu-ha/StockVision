# StockVision 오픈소스 라이선스 정리 초안

이 문서는 StockVision의 오픈소스 공개 방향을 검토하기 위한 내부 초안입니다.
`docs/product/product-direction-log.md`에서 정리된 제품 방향, 특히 `무료 오픈소스 트레이딩 엔진 + 유료 개인 비서 클라우드` 가설을 기준으로 작성했습니다.
법률 자문을 대체하지 않으며, 실제 공개 전에는 변호사 검토와 제3자 라이선스 점검이 필요합니다.

## 결론 요약

- 권장 라이선스: `MPL-2.0`
- 권장 공개 대상: 로컬 실행 엔진, 브로커 어댑터, 전략 DSL/평가기, 로컬 대시보드, 공개 가능한 테스트와 스키마
- 권장 비공개 대상: 클라우드 운영 코드, 관리자 도구, 운영 자동화, 내부 연구 문서, 비밀정보, 재배포 권한이 불명확한 데이터/모델
- 권장 저장소 전략: 현재 모노레포 전체를 그대로 공개하지 말고, 공개 범위를 분리한 공개 저장소를 별도로 운영

## 제품 방향과 라이선스의 연결

제품 방향 로그에서 StockVision은 아래 조합으로 정의되어 있습니다.

- 오픈소스 쪽 가치: 로컬 실행 엔진, 키 보관 방식, 브로커 어댑터, 전략 프레임워크, 안전장치의 투명성
- 유료 쪽 가치: 전략 동기화, 백업, 버전 히스토리, 메신저 알림, 브리핑, 리포트, 비서 메모리 같은 지속 서비스

즉, 차별화 포인트는 `숨겨진 코드`보다 아래에 가깝습니다.

- 한국 주식에 맞춘 실행 경험
- 사용자별 기억과 개인화
- 안전한 로컬 실행 구조
- 운영 편의가 높은 클라우드 비서 경험

이 방향에 가장 잘 맞는 라이선스가 `MPL-2.0`입니다.

## 왜 `MPL-2.0`인가

### 1. `MIT`보다 방어력이 있다

`MIT`는 출처와 라이선스 고지만 유지하면 수정본을 닫아서 재배포해도 됩니다.
반면 `MPL-2.0`은 공개된 파일을 수정해 배포하면 그 수정 파일은 다시 `MPL-2.0`으로 공개해야 합니다.

### 2. `GPL/AGPL`보다 도입 장벽이 낮다

`MPL-2.0`은 파일 단위 copyleft입니다.
즉, 공개 파일을 고친 부분은 열리지만, 별도 파일로 결합된 더 큰 작업물 전체를 강하게 묶지는 않습니다.
StockVision처럼 로컬 코드와 비공개 운영 코드의 경계를 나누고 싶은 프로젝트에 더 잘 맞습니다.

### 3. 로컬 실행 제품의 신뢰를 높이기 좋다

이 프로젝트의 경쟁력은 단순히 `코드를 숨기는 것`보다 아래에 더 가깝습니다.

- 로컬에서 실제로 어떤 로직이 실행되는지 사용자가 검증할 수 있는 신뢰
- 브로커 연동과 키 보관 방식의 투명성
- 안정적인 운영 경험
- 클라우드 동기화, 알림, 히스토리, 관리형 비서 경험 같은 지속 서비스

`MPL-2.0`은 공개 코드에 대한 신뢰를 만들면서도, 운영 자산 전체를 강제로 공개하지 않게 해줍니다.

### 4. 오픈소스의 제약을 정면으로 받아들인 선택이다

오픈소스 라이선스는 상업적 사용 금지, 경쟁사 사용 금지, 특정 업종 금지 같은 제한을 걸 수 없습니다.
그런 제한을 넣고 싶다면 더 이상 OSI 기준의 오픈소스가 아닙니다.
따라서 `막을 수 없는 사용`은 받아들이고, 대신 `배포 시 수정 파일 공개`와 `브랜드 분리`에 무게를 두는 편이 현실적입니다.

## `MPL-2.0`이 해결해주는 것

- 공개한 파일을 누가 수정해 배포하면 그 수정 파일 공개를 요구할 수 있습니다.
- 실행 파일만 배포해도 대응하는 소스 코드 접근 경로를 제공하게 만들 수 있습니다.
- 특허 사용 허가가 포함됩니다.
- 상표권은 자동으로 주어지지 않으므로 코드 공개와 브랜드 보호를 분리할 수 있습니다.

## `MPL-2.0`이 해결해주지 못하는 것

- 남이 코드를 상업적으로 쓰는 것 자체를 막을 수는 없습니다.
- 남이 사내에서만 수정해서 쓰는 것을 공개시키기 어렵습니다.
- 서버에서만 돌리고 배포하지 않는 경우까지 강하게 공개시키지 못합니다.
- 현재 모노레포에 섞여 있는 비공개 자산을 자동으로 구분해주지 않습니다.

## 이 프로젝트에 대한 실무 해석

StockVision은 현재 아래 성격이 섞여 있습니다.

- 사용자가 결국 들여다볼 수 있는 로컬 실행 코드
- 서비스 운영 품질이 가치가 되는 클라우드 코드
- 내부 연구/법률/전략 문서
- 브로커/데이터/모델/배포 등 재배포 검토가 필요한 자산

그래서 가장 중요한 결정은 `라이선스 선택`보다 `공개 범위 분리`입니다.

### 공개 후보

- `local_server/`
- `sv_core/`
- 로컬 제어/상태 확인에 필요한 `frontend/` 일부
- 공개 가능한 테스트 코드
- 공개 가능한 API 스키마, 포맷 문서, 설치 문서

### 비공개 유지 후보

- `cloud_server/`
- `backend/` 레거시 또는 운영 참고 코드
- `docs/legal/`, `docs/research/` 등 내부 검토 문서
- 운영 인프라, 비밀 키, 관리자 기능
- 재배포 권한이 정리되지 않은 데이터/모델/에셋

## 채택 시 꼭 필요한 문서와 작업

### 필수에 가까운 것

1. `LICENSE`
   - 최종 공개 저장소 루트에 `MPL-2.0` 전문을 수정 없이 둡니다.
2. 파일별 라이선스 고지
   - 각 소스 파일에 Exhibit A 헤더 또는 `SPDX-License-Identifier: MPL-2.0`를 넣습니다.
3. 실행 파일 배포 시 소스 접근 안내
   - 설치본, 릴리스 노트, About 화면, README 중 하나 이상에 소스 코드 위치를 명시합니다.
4. `README` 라이선스 섹션
   - 어떤 부분이 오픈소스인지, 어떤 문서가 기준인지, 상표는 별도라는 점을 적습니다.

### 강하게 권장되는 것

1. `OPEN_SOURCE_SCOPE.md`
   - 정확히 무엇을 공개하고 무엇을 공개하지 않는지 정의합니다.
2. `TRADEMARKS.md`
   - 이름, 로고, 앱 아이콘, 배포 명칭 사용 규칙을 분리합니다.
3. `THIRD_PARTY_NOTICES.md`
   - 외부 라이브러리, 폰트, 아이콘, 데이터, 모델, SDK의 재배포 조건을 정리합니다.
4. `CONTRIBUTING.md`
   - 외부 기여의 라이선스 귀속과 개발 규칙을 정리합니다.
5. `SECURITY.md`
   - 취약점 제보 경로를 둡니다.
6. `DISCLAIMER.md`
   - 투자 자문이 아니라는 점, 무보증, 실거래 책임 범위를 명시합니다.

## 적용 전에 먼저 할 일

1. 공개 저장소 범위를 따로 분리합니다.
2. `cloud_server/`, 내부 문서, 운영 도구, 비밀정보가 공개 경로에 섞이지 않게 정리합니다.
3. 모델 파일, 데이터 파일, 아이콘, 폰트, 브로커 연동 자산의 재배포 권한을 확인합니다.
4. 소스 파일 헤더 정책을 정합니다.
5. 루트 `README.md`와 릴리스 프로세스에 소스 제공 안내를 넣습니다.
6. 상표 정책을 함께 공개합니다.

## 이 초안 세트의 용도

이 폴더의 문서는 `바로 배포`가 아니라 `검토 후 채택`을 위한 초안입니다.
실제 적용 시에는 아래처럼 옮겨야 합니다.

- `docs/open-source/LICENSE` -> 공개 저장소 루트 `LICENSE`
- `docs/open-source/OPEN_SOURCE_SCOPE.md` -> 공개 저장소 루트 `OPEN_SOURCE_SCOPE.md`
- `docs/open-source/TRADEMARKS.md` -> 공개 저장소 루트 `TRADEMARKS.md`
- `docs/open-source/README-license-section.md` -> 공개 저장소 루트 `README.md`의 라이선스 섹션으로 반영
