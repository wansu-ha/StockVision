# Security Policy (초안)

이 문서는 StockVision 프로젝트의 보안 취약점 제보 및 대응 원칙 초안입니다.
이 프로젝트는 1인 유지관리 체계로 운영되므로, 보안 이슈는 반드시 비공개 채널로 제보해 주세요.

## 지원 범위

현재 지원 범위는 아래로 제한합니다.

- 최신 안정 버전
- 필요 시 직전 안정 버전 1개
- 개발 브랜치나 오래된 비공식 빌드는 지원 대상이 아닐 수 있습니다

## 제보 방법

보안 취약점은 공개 GitHub issue로 올리지 말고 아래 채널로 제보해 주세요.

- 보안 제보 이메일: `[보안 제보 이메일 주소 입력]`
- 또는: `[비공개 취약점 제보 채널 입력]`

공개 issue, PR, discussion에 취약점을 올리면 사용자를 위험에 노출할 수 있습니다.
유지관리자는 공개 제보를 비공개 채널로 전환하도록 요청하거나 삭제할 수 있습니다.

## 제보 시 포함해 주세요

가능하면 아래 정보를 함께 보내 주세요.

- 영향 받는 버전
- 재현 단계
- 예상 영향 범위
- PoC 또는 로그
- 필요한 전제 조건
- 제안하는 완화 방법

다만 아래 정보는 보내지 마세요.

- API Key
- Secret
- refresh token
- 계좌번호
- 실거래 주문 원본
- 개인 식별 정보

## 응답 목표

1인 유지관리 체계 기준의 목표입니다.

- 1차 접수 응답: 3영업일 이내 목표
- triage 완료: 가능한 한 빠르게
- 수정 일정: 영향도와 재현 가능성에 따라 결정
- 공개 시점: 수정 또는 완화책 준비 후 협의

이 일정은 보장 SLA가 아니라 best-effort 목표입니다.

## 대응 우선순위

StockVision에서는 아래 항목을 특히 우선적으로 봅니다.

- localhost mutation API 인증 우회
- WebSocket 인증/Origin 검증 누락
- 규칙 주입 또는 엔진 강제 실행
- 토큰 저장 및 인증 정보 유출
- 브로커 키 또는 계정 정보 노출
- 클라우드 인증 우회, 세션 탈취, 권한 상승

## 공개 원칙

- 가능한 한 제보자와 협의하여 공개 시점을 정합니다.
- 수정 전 상세 exploit 정보 공개는 지양합니다.
- 필요 시 보안 공지 또는 릴리스 노트에 영향 버전과 수정 버전을 명시합니다.
- 제보자가 원하면 크레딧 표기를 고려할 수 있습니다.

## 범위 밖 항목

아래는 보안 취약점이 아니라 일반 지원 또는 기능 요청으로 분류될 수 있습니다.

- 설치 환경 차이로 인한 일반 오류
- 증권사 API 자체 장애
- 투자 손실 또는 전략 성과 관련 문의
- 지원 종료 버전의 일반 버그
- 문서 오탈자, UI 개선 제안

## 법적/운영 참고

- 이 프로젝트는 투자자문이나 투자일임 서비스를 제공하지 않습니다.
- 보안 제보 처리 과정에서도 사용자의 계좌 접근 정보나 거래 정보는 받지 않습니다.
- 취약점 대응 과정에서 필요한 최소한의 정보만 요청합니다.

## 유지관리자 권리

유지관리자는 아래를 할 수 있습니다.

- 공개 제보를 비공개 채널로 이동 요청
- 민감정보가 포함된 내용을 편집 또는 삭제
- 재현 정보가 부족한 제보를 추가 정보 요청 상태로 보류
- 범위를 벗어난 요청을 일반 이슈로 전환
