# 전략 목록 정보 표시

## 목표

전략 카드/리스트에서 "이 전략이 뭘 하는지" 한눈에 파악할 수 있게 핵심 정보를 보강한다.

현재 문제:
- 종목 코드만 표시 (`005930`) — 종목명 없음
- 전략이 로컬 엔진에서 실행 중인지 알 수 없음 (클라우드 `is_active`와 로컬 실행 상태는 다름)
- 매수/매도 방향이 카드에서 바로 안 보임

## 요구사항

| ID | 요구사항 | 우선순위 |
|----|---------| ---------|
| S1 | 종목 코드 옆에 종목명 표시 (`삼성전자 005930`) | P0 |
| S2 | 전략 방향 표시 (매수/매도/양방향) — DSL script에서 파싱 | P0 |
| S3 | 로컬 엔진 실행 상태 표시 (실행 중/OFF) | P1 |

## 설계

### 카드/리스트 레이아웃

**RuleCard (그리드):**
```
┌──────────────────────────────────┐
│ RSI 과매도 전략            [●━━] │
│ 삼성전자 005930                  │  ← 종목명 추가
│                                  │
│ 매수: kospi_rsi_14 < 30 | ...   │
│                                  │
│ 수정  삭제                       │
└──────────────────────────────────┘
```

**RuleList (리스트):**
```
[●━━] RSI 과매도 전략  삼성전자 · 매수 · 실행 중    수정  삭제
```

| 요소 | 출처 | 비고 |
|------|------|------|
| 전략 이름 | `rule.name` | 기존 |
| 종목명 | 종목 코드 → 이름 매핑 | 아래 §종목명 매핑 참조 |
| 방향 | `rule.script` 파싱 (`매수:` / `매도:` 키워드) | v1은 `buy_conditions` / `sell_conditions` 유무로 판단 |
| 실행 상태 | 로컬 서버 엔진 상태 조회 | `is_active`와 별개 |

### 종목명 매핑

종목 코드 → 이름 변환 방법 후보:

1. **클라우드 서버 stocks API 활용** — `GET /api/v1/stocks` 등 기존 엔드포인트에서 종목 목록 캐싱
2. **프론트엔드 로컬 매핑 테이블** — 정적 JSON 또는 초기 로딩 시 fetch

> 구현 시 plan에서 확정. 핵심은 API 호출을 매 렌더마다 하지 않는 것.

### 방향 파싱

DSL script에서 `매수:` / `매도:` 라인 존재 여부로 판단:
- `매수:` 만 있음 → "매수"
- `매도:` 만 있음 → "매도"
- 둘 다 있음 → "양방향"
- script 없음 (v1) → `buy_conditions` / `sell_conditions` 유무로 동일 판단

### 실행 상태 (S3)

`is_active`는 클라우드 DB의 ON/OFF (사용자 의도). 실행 상태는 로컬 엔진이 이 규칙을 실제로 평가 중인지 여부.

- 로컬 서버 `GET /api/status`의 엔진 상태 + 동기화된 규칙 목록에서 판단
- 엔진 미실행 → 모든 규칙 "OFF"
- 엔진 실행 중 + `is_active: true` → "실행 중"
- 엔진 실행 중 + `is_active: false` → "OFF"
