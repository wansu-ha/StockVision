## 1. 목표

전략 엔진의 표현력과 실행 능력을 프로 수준으로 끌어올리되,
사용자 경험은 카드 UI + AI 생성 + 실시간 투명성으로 차별화한다.

**철학:**

```
DSL = 강력한 IR (파워유저도 직접 편집 가능)
차별화 = AI 전략 생성 + 실시간 투명성 + 한국 특화 로컬 실행
```

| 계층 | 사용자 | 인터페이스 |
|------|--------|-----------|
| 겹 1 | 초급~중급 (80%) | 카드 UI / 자연어 → AI 생성 |
| 겹 2 | 중급~고급 (15%) | DSL 직접 편집 |
| 겹 3 | 프로 (5%) | Python 커스텀 함수 연동 (후속 spec) |

### 2.2 프로 트레이더 기준 부족한 점

**리서치 결론: "현재 DSL은 전략의 10%만 커버한다."**

| 영역 | 현재 | 필요 |
|------|------|------|
| 진입 | 단일 조건 매수 | DCA, 파라미터화, 이전 봉 참조, 시간 필터 |
| 청산 | `매도: 수익률 >= 5` (고정) | 트레일링, 다단계, 시간 청산, 브레이크이븐, ATR 동적 |
| 투명성 | 블랙박스 (결과만) | 조건별 실시간 상태, 트리거 이력 |
| 파라미터 | 하드코딩 | 변수 선언 + 최적화 스윕 |
| 상태 추적 | 크로스오버만 | 연속 봉 수, 래치, 실행 이력 |

### 2.3 차별화 방향 (EasyLanguage/Pine Script와의 차이)

| | EasyLanguage / Pine Script | StockVision v2 |
|---|---|---|
| 패러다임 | 절차적 코딩 | 선언적 규칙 (카드 UI) |
| 디버깅 | Print문 수동 | 조건별 실시간 자동 |
| 단위 | 달러 기반 | % 기반 (한국 주식 네이티브) |
| 청산 | 내장 함수 호출 | 선언적 카드 |
| 진입장벽 | 높음 (프로그래밍) | 낮음 (자연어/폼/카드) |
| 코드 직접 편집 | 필수 | 선택 (파워유저 escape hatch) |

**핵심**: DSL 문법으로 차별화하지 않는다. 문법은 강력한 IR일 뿐.
차별화는 AI 생성 + 카드 UI + 실시간 투명성 + 한국 특화에서 나온다.

### 2.4 선언적 트레이딩 제품 조사 결과

업계 제품을 텍스트/비주얼 × 선언적/절차적으로 분류하면:

```
       텍스트
         ↑
EasyLanguage  │  ★ StockVision DSL
Pine Script   │  (이 조합이 비어있음)
              │
  절차적 ─────┼───── 선언적
              │
  Python      │  Composer, Tradetron
              │  TrendSpider, Vestinda
         ↓
       비주얼
```

| 제품 | 위치 | 투명성 | 직접 편집 |
|------|------|--------|----------|
| Composer.trade | 선언적 + 비주얼 | 블록 상태 표시 | 불가 (비주얼만) |
| Tradetron | 선언적 + 비주얼 | 규칙 실행 로그 | 불가 |
| TrendSpider | 선언적 + 비주얼 | 차트 오버레이 | 불가 |
| EasyLanguage | 절차적 + 텍스트 | Print문 수동 | 가능 |
| Pine Script | 절차적 + 텍스트 | Plot 수동 | 가능 |

**비주얼 빌더는 투명하지만 편집 불가, 텍스트 언어는 편집 가능하지만 불투명.**
StockVision DSL은 선언적(→ 구조 추출 가능 → 투명) + 텍스트(→ 직접 편집 가능) 조합.
이 위에 AI 생성 + 카드 UI + 한국 특화(원화 %, KIS/키움, 한글)가 합쳐져서 차별화.

**왜 자체 DSL인가 (Python/EasyLanguage/JSON이 아닌 이유):**
- **Python**: 사용자 코드 샌드박스 없이 실제 돈으로 주문 실행 → 안전성 불가
- **EasyLanguage**: 절차적이라 조건 구조 추출 불가 → 투명성 불가. 파서 구현 비용 막대
- **JSON만**: 구조적이지만 사람이 직접 편집 불가 → 파워유저 15%가 못 씀

### 2.5 DSL 설계 원칙

1. **`이름 = 값`은 상수.** 전략 실행 중 불변. 변수는 없다. 내장 필드/함수와 동일한 네임스페이스.
2. **상수 값은 숫자 또는 문자열.** 숫자면 프론트에서 슬라이더 UI, 문자열이면 드롭다운 UI 자동 생성.
3. **변하는 것은 전부 함수.** `수익률`, `고점 대비`, `실행횟수`, `횟수(조건, 기간)` — 매 사이클 평가되는 값은 내장 필드 또는 함수로 제공. 사용자가 커스텀 변수를 갱신하는 것은 허용하지 않는다.
4. **새로운 추적 패턴이 필요하면 → 내장 함수/필드를 추가.** DSL 문법 변경 없이 확장.
5. **인자 없는 함수는 괄호 선택.** `골든크로스`과 `골든크로스()`는 동일. 괄호 생략을 권장하되, 붙여도 에러가 아님. "이름 쓰면 값이 나오는 것"으로 통일.
6. **`[]`는 이전 봉 참조 전용.** 업계 표준(EasyLanguage, Pine Script) 일치.
7. **이름 해석 순서:** 사용자 상수 → 내장 필드 → 내장 함수 → 에러. 충돌 시 사용자 상수 우선 + 파서 경고.

### 2.6 DSL v2 문법 정리

```
스크립트 = 상수선언* 규칙+

상수선언 = 이름 '=' 값
  값 = 숫자 | 문자열

규칙 = 조건식 ('→' | '->') 행동

행동 = ('매수' | '매도') 수량
  수량 = 숫자 '%' | '전량' | '나머지'

조건식 = OR식
  OR식 = AND식 ('OR' AND식)*
  AND식 = NOT식 ('AND' NOT식)*
  NOT식 = 'NOT' NOT식 | 비교식
  비교식 = 산술식 (비교연산자 산술식 | 'BETWEEN' 산술식 'AND' 산술식)?
  산술식 = 곱셈식 (('+' | '-') 곱셈식)*
  곱셈식 = 단항식 (('*' | '/') 단항식)*
  단항식 = '-' 단항식 | 후위식
  후위식 = 기본식 ('[' 숫자 ']')?
  기본식 = 숫자 | 문자열 | 불리언
         | 이름 '(' 인자목록 ')'      -- 함수 호출
         | 이름                       -- 필드/상수/무인자함수
         | '(' 조건식 ')'             -- 괄호 그룹

인자목록 = 조건식 (',' 조건식)*
비교연산자 = '>' | '>=' | '<' | '<=' | '==' | '!='
숫자 = 정수 | 소수
문자열 = '"' 문자* '"'
불리언 = 'true' | 'false'
이름 = 한글/영문/숫자/밑줄 (첫 글자는 숫자 불가)
주석 = '--' 줄끝까지
```

**예시:**

```
-- 상수
기간 = 14
손절 = -2
tf = "1d"

-- 규칙 (위에서 아래로 = 우선순위)
MA(20) <= MA(60) AND 보유수량 > 0           → 매도 전량
RSI(기간) < 30 AND 골든크로스 AND 보유수량 == 0  → 매수 100%
수익률 <= 손절                               → 매도 전량
수익률 >= 3 AND 실행횟수 < 1                 → 매도 50%
고점 대비 <= -1.5                            → 매도 나머지
횟수(수익률 >= 2, 보유봉) >= 1 AND 수익률 <= 0            → 매도 전량
보유일 >= 3 AND 수익률 > -1 AND 수익률 < 1   → 매도 전량
```

## 3. 요구사항

### 3.1 DSL 표현력 (IR로서)

| ID | 요구사항 | 우선순위 |
|----|---------|---------|
| D1 | 상수 선언 (`이름 = 기본값`). `@` 없음 — 내장 필드/함수와 동일한 문법. 값이 스칼라(숫자)면 프론트에서 슬라이더 UI 자동 생성. 이름 해석 순서: 사용자 상수 → 내장 필드 → 내장 함수 (충돌 시 경고) | P0 |
| D2 | 이전 봉 참조: `식[N]` (N은 정수, 0=현재, 최대 60). `현재가[1]` = 1봉 전 종가, `RSI(14)[3]` = 3봉 전 RSI. `[]` 안에는 **정수만** 허용 (§2.6 EBNF 참고). 타임프레임은 함수 인자로 지정: `RSI(14, "1d")[1]` = 전일 RSI. §5.5 데이터 소스 참고 | P0 |
| D3 | (조건 → 행동) × N 규칙 구조 | P0 |
| D4 | 행동에 방향+수량 포함. **매수 N% = 일일 할당 예산의 N%, 매도 N% = 보유수량의 N%**. `매도 전량` = 보유 전량, `매도 나머지` = `매도 전량`의 의미적 별칭 (이전 사이클 부분 매도 후 남은 전량을 청산할 때 사용. 실행 시점에는 `매도 전량`과 동일하게 현재 보유수량 전량 매도) | P0 |
| D5 | 위→아래 순서 평가 (순서가 우선순위, §5.1 충돌 해소 참고) | P0 |
| D6 | 시간 필드 (`시간`, `장시작후`, `요일`) | P1 |
| D7 | 상태 함수 `횟수(조건, 기간)` — 기간 내 조건 True 봉 수. 래치 대체: `횟수(조건, 보유봉) >= 1` | P1 |
| D8 | 상태 함수 `연속(조건)` — 현재 연속 True 봉 수 | P1 |
| D9 | 상태 함수 `실행횟수` — 이 규칙의 현재 포지션 내 실행 횟수 | P1 |
| D10 | `BETWEEN A AND B` — 레인지 조건 syntactic sugar. 파서에서 `>= A AND <= B`로 변환 | P1 |
| D12 | 파라미터 있는 커스텀 함수 | P2 |

> **수식어 폐지 + 함수 통합 (2026-03-29 논의):**
> - 수식어 `[1회]`, `[래치]`, `[유지 N봉]`, `[N봉내]` 전부 폐지
> - `달성` + `카운트` → `횟수`로 통합. 래치는 `횟수(조건, 보유봉) >= 1`로 표현
> - 상태 함수 3개만 남음: `횟수`, `연속`, `실행횟수`
> - `[]`는 이전 봉 참조 전용 — 업계 표준 일치

### 3.3 포지션 상태 관리

엔진이 포지션별로 추적해야 하는 상태:

```
entry_price: float          # 평균 진입가 (DCA 시 가중평균)
entry_time: datetime        # 최초 진입 시각
highest_price: float        # 보유 중 최고가 (트레일링용, 매 사이클 갱신)
pnl_high: float             # 보유 중 최고 수익률% (수익률고점 필드용, 매 사이클 갱신)
remaining_ratio: float      # 잔여 비율 (1.0 시작, 다단계 청산용)
bars_held: int              # 보유 봉 수 (매 사이클 +1)
days_held: int              # 보유 영업일 수
func_state: dict             # 상태 함수 내부 상태 (실행횟수 등)
```

**DCA 진입가 계산:**

```
진입가 = Σ(체결가 × 체결수량) / 총 보유수량

예: 1차 50주 × 72,000 + 2차 30주 × 70,000
    = (3,600,000 + 2,100,000) / 80 = 71,250원
```

`수익률`, `고점 대비` 등 진입가 기반 필드는 모두 이 가중평균 기준.

**전량 청산 정의:** `보유수량 == 0`. 부분 매도로 수량이 줄어도 0이 아니면 청산이 아님 — 상태 함수(`실행횟수`)는 리셋되지 않는다. `횟수`는 윈도우 기반이라 자동 이동.

**보유봉 초기값:** 진입 사이클의 `보유봉 = 1`. 0이면 `횟수(조건, 보유봉)`이 첫 봉에서 항상 0을 반환하여 래치 체크 불가.

**재진입:** 전량 청산 후 매수 조건이 다시 True면 재진입. PositionState 전체 리셋(진입가, 최고가, 보유봉, 실행횟수 등). 새 포지션으로 취급.

**일일 실행 제한:** trigger_policy(ONCE_PER_DAY, ONCE 등)는 규칙 메타데이터(execution JSON)로 관리하며 DSL 밖에서 엔진(SignalManager)이 적용. 하루에 같은 규칙이 여러 번 진입/청산을 반복하는 것을 방지.

### 3.4 내장 필드 확장

| 필드 | 타입 | 의미 | 신규 |
|------|------|------|------|
| `현재가` | 숫자 | 현재 가격 (+ `[N]` 이전 봉) | [N] 신규 |
| `거래량` | 숫자 | 현재 거래량 | 기존 |
| `수익률` | 숫자 | 진입가 대비 % | 기존 |
| `보유수량` | 숫자 | 현재 보유 수량 | 기존 |
| `고점 대비` | 숫자 | 보유 중 최고가 대비 하락률% | **신규** |
| `수익률고점` | 숫자 | 보유 중 최고 수익률% (매 사이클 갱신) | **신규** |
| `진입가` | 숫자 | 평균 진입 단가 | **신규** |
| `보유일` | 숫자 | 진입 후 경과 영업일 | **신규** |
| `보유봉` | 숫자 | 진입 후 경과 봉 수 | **신규** |
| `시간` | 시간 | 현재 시각 HH:MM (KST) | **신규** |
| `장시작후` | 숫자 | 장 시작 후 경과 분 | **신규** |
| `요일` | 숫자 | 1=월 ~ 5=금 | **신규** |

### 3.5 내장 함수 확장

기존 유지 (RSI, MA, EMA, MACD, MACD_SIGNAL, 볼린저_상단/하단, 평균거래량, 상향돌파, 하향돌파, 8개 패턴), 추가:

| 함수 | 시그니처 | 반환 | 용도 |
|------|---------|------|------|
| `ATR` | `ATR(기간, [타임프레임])` | 숫자 | 변동성, 동적 손절 |
| `최고가` | `최고가(기간, [타임프레임])` | 숫자 | N봉 최고가 |
| `최저가` | `최저가(기간, [타임프레임])` | 숫자 | N봉 최저가 |
| `이격도` | `이격도(기간, [타임프레임])` | 숫자 | (현재가 - MA) / MA × 100 |

**상태 함수** (조건을 인자로 받는 특수 함수 — 평가기에서 지연 평가):

| 함수 | 시그니처 | 반환 | 용도 |
|------|---------|------|------|
| `횟수` | `횟수(조건, 기간)` | 숫자 | 기간 내 조건 True 봉 수. `횟수(조건, 보유봉) >= 1`로 래치 대체 |
| `연속` | `연속(조건)` | 숫자 | 현재 연속 True 봉 수 |
| `실행횟수` | `실행횟수` | 숫자 | 이 규칙의 현재 포지션 내 실행 횟수. 포지션 청산 시 리셋 |

## 4. DSL 예시 — 실전 패턴

### 4.1 기본 추세 추종

```
기간 = 14

-- 추세 이탈 → 최우선 청산
MA(20) <= MA(60) AND 보유수량 > 0   → 매도 전량

-- 진입
RSI(기간) < 30 AND MA(20) > MA(60) AND 보유수량 == 0  → 매수 100%

-- 청산
수익률 <= -2                       → 매도 전량
수익률 >= 5                        → 매도 전량
```

### 4.2 다단계 청산 + 트레일링

```
-- 손절 (최우선)
수익률 <= -2                       → 매도 전량

-- 1차 익절 (1회만 실행)
수익률 >= 3 AND 실행횟수 < 1     → 매도 50%

-- 잔여분 트레일링
고점 대비 <= -1.5                  → 매도 나머지

-- 시간 청산 (횡보 방지)
보유일 >= 3 AND 수익률 BETWEEN -1 AND 1  → 매도 전량
```

### 4.3 DCA (분할 매수)

```
기간 = 14

-- 1차 진입
RSI(기간) < 30 AND 보유수량 == 0   → 매수 50%

-- 2차 추가 매수 (더 떨어지면)
RSI(기간) < 20 AND 보유수량 > 0    → 매수 30%

-- 청산
수익률 >= 5                        → 매도 전량
수익률 <= -5                       → 매도 전량
```

### 4.4 브레이크이븐 + 트레일링

```
-- 손절
수익률 <= -2                       → 매도 전량

-- 브레이크이븐: 2% 달성한 적 있으면 본전에서 청산
횟수(수익률 >= 2, 보유봉) >= 1 AND 수익률 <= 0  → 매도 전량

-- 트레일링
고점 대비 <= -1.5                  → 매도 전량
```

### 4.5 시간 필터 + 순차 조건

```
-- 장 시작 10분은 진입 금지
-- 골든크로스 발생 후 3봉 이내에 RSI 눌림목이면 진입

장시작후 >= 10 AND 횟수(골든크로스, 3) >= 1 AND RSI(14) < 35 AND 보유수량 == 0  → 매수 100%

-- 청산
수익률 <= -2                       → 매도 전량
고점 대비 <= -1.5                  → 매도 전량
```

### 4.6 ATR 기반 동적 청산

```
배수 = 2

-- ATR 기반 동적 손절
현재가 <= 진입가 - ATR(14) * 배수   → 매도 전량

-- ATR 기반 동적 익절
현재가 >= 진입가 + ATR(14) * 배수 * 1.5  → 매도 전량

-- 진입
RSI(14) < 30 AND 보유수량 == 0      → 매수 100%
```

### 4.7 거래량 + 이격도 패턴

```
-- 거래량 폭발 + 이격도 과매도 → 반등 매수
거래량 > 평균거래량(20) * 2 AND 이격도(20) < -5 AND 보유수량 == 0  → 매수 100%

-- 청산
이격도(20) > 3                      → 매도 전량
수익률 <= -3                       → 매도 전량
```

## 5. 평가 모델

### 5.1 사이클 흐름

```
매 사이클 (1분):
  1. 시세 데이터 수집 (가격, 거래량, 지표 — 전부 로컬 캐시, §5.5 참고)
  2. 포지션 상태 갱신 (최고가, 보유봉, 보유일)
  3. 모든 규칙을 위→아래 순서대로 평가
     - 실행 여부와 무관하게 전 규칙의 조건 값 기록 (투명성 API용)
     - 상태 함수 평가 (횟수, 연속, 실행횟수 등의 내부 상태 갱신)
  4. 실행 규칙 결정 (True인 규칙들 중 우선순위):
     우선순위 1: 전량 매도 — True인 것 중 가장 위 하나
     우선순위 2: 부분 매도 — 전량 매도가 없을 때만, 가장 위 하나
     우선순위 3: 매수   — 매도가 하나도 없을 때만, 가장 위 하나
     → 한 사이클에 실행되는 행동: 최대 1개
  5. 실행
  6. 포지션 전량 청산(보유수량 == 0) 시 상태 함수 리셋 (실행횟수)
  7. 조건 상태 API에 결과 저장 (모든 조건의 평가 결과, 실행 안 된 것 포함)
```

**왜 "최대 1개"인가:**
- 전량 매도 + 부분 매도가 동시 True → 전량 매도가 이김. 부분 매도를 먼저 실행하면 전량 손절이 1분 늦어질 수 있다.
- 매수 + 매도가 동시 True → 매도가 이김. 손실 확대 방지가 신규 진입보다 우선.
- 부분 매도 후 다음 사이클에서 잔여분에 대해 다음 규칙이 평가되므로, 다단계 청산은 사이클 단위로 진행된다.

### 5.2 상태 함수 동작 정의

| 함수 | 동작 | 리셋 시점 |
|------|------|----------|
| `횟수(조건, 기간)` | 기간 내 조건 True 봉 수 반환. 기간에 `보유봉` 넣으면 포지션 전체 = 래치 | 자동 (윈도우 이동) |
| `연속(조건)` | 현재 연속 True인 봉 수 반환 | 연속 끊기면 0 |
| `실행횟수` | 이 규칙이 현재 포지션에서 실행된 횟수 반환 | 포지션 전량 청산 시 |

상태 함수는 **조건을 인자로 받는 특수 함수**. 평가기에서 인자를 즉시 평가하지 않고 표현식 자체를 매 사이클 평가(지연 평가). 파서는 일반 함수 호출과 동일하게 처리.

### 5.3 `매도 나머지`의 동작

`매도 나머지`는 `매도 전량`의 **의미적 별칭**이다. 실행 시 현재 보유수량 전량을 매도한다.

사이클당 최대 1개 행동 원칙에 의해, 부분 매도(`매도 50%`)와 `매도 나머지`가 같은 사이클에서 동시 실행되는 일은 없다. 다단계 청산은 사이클 단위로 진행된다:

```
사이클 1: 수익률 >= 3 → 매도 50%     (보유 100주 → 50주 매도)
사이클 2: 고점 대비 <= -1.5 → 매도 나머지  (남은 50주 전량 매도)
```

`나머지`라는 표현은 가독성을 위한 것이지, 실행 로직상 `전량`과 다르지 않다.

### 5.4 기존 v1과의 호환성

| v1 DSL | v2 호환 |
|--------|--------|
| `매수: RSI(14) < 30` | 파서가 `RSI(14) < 30 AND 보유수량 == 0 → 매수 100%`로 변환 |
| `매도: 수익률 >= 5` | 파서가 `수익률 >= 5 → 매도 전량`으로 변환 |
| `buy_conditions` JSON (v1) | 엔진 폴백 유지 |

v1 스크립트는 v2 파서가 자동 변환. 기존 규칙이 깨지지 않는다.

**커스텀 함수:** `내조건 = RSI(14) < 30 AND MA(5) > MA(20)` — 상수와 동일한 문법. 값이 조건식이면 무인자 함수로 취급. 사용 시 괄호 선택: `내조건` = `내조건()`. v1의 `내조건() = ...` 형태도 파서가 인식 (호환).

### 5.5 데이터 소스와 `[N]` 참조

**원칙:** 모든 평가 데이터는 로컬 캐시에서 읽는다. 평가 시점에 외부 네트워크 요청 없음.

| 데이터 | 소스 | `[N]` 참조 시 | 지연 |
|--------|------|--------------|------|
| `현재가`, `거래량` | WS push → BarBuilder (메모리) | MinuteBarStore (로컬 SQLite) | 없음 |
| 일봉 지표 (예: `RSI(14, "1d")[1]`) | IndicatorProvider 캐시 (yfinance, 1일 갱신) | 동일 캐시 | 없음 |
| `RSI(14)` 등 지표 현재값 | IndicatorProvider 캐시 | — | 없음 |
| `RSI(14)[N]` 등 지표 히스토리 | **신규: 지표 히스토리 링버퍼** (최근 60봉) | 링버퍼 조회 | 없음 |
| `진입가`, `보유일` 등 포지션 필드 | PositionState (엔진 메모리) | — | 없음 |

**`[N]`의 봉 기준:**
- 타임프레임 미지정: 엔진 평가 주기 = **1분봉** (`현재가[1]` = 1분 전 종가)
- 타임프레임 지정: 함수 인자로 (`RSI(14, "1d")[1]` = 전일 RSI, `RSI(14, "5m")[3]` = 3봉(15분) 전 RSI)
- 필드(`현재가`, `거래량`)의 타임프레임별 이전 봉 참조는 이번 범위에서 미지원. 현재는 `현재가[N]` = 엔진 평가 주기(1분봉) 기준만 가능. 일봉 종가 참조가 필요하면 `MA(1, "1d")` 등 기존 함수로 우회

**신규 필요: 지표 히스토리 링버퍼**
- IndicatorProvider가 매 사이클 지표를 계산할 때 최근 60개를 링버퍼에 보관
- `RSI(14)[3]` → 링버퍼에서 3번째 이전 값 조회
- 메모리만 사용, 외부 요청 없음
- 엔진 재시작 시 링버퍼는 비어있음 → `[N]`이 None → 조건 False (안전)

## 7. 수용 기준

### P0 (필수)

- [ ] DSL v2 파서가 (조건 → 행동) × N 규칙 구조를 파싱
- [ ] 상수 선언 (`이름 = 값`) 및 이름 해석 (§2.5)
- [ ] 이전 봉 참조 (`현재가[1]`, `RSI(14)[3]`)
- [ ] 트레일링 스탑 (`고점 대비 <= -N%`)
- [ ] 고정 손절/익절
- [ ] 다단계 청산 (`매도 50%` → 다음 사이클에서 `매도 나머지`)
- [ ] 같은 방향 규칙은 사이클당 하나만 실행 (§5.1)
- [ ] 매수+매도 동시 시 매도 우선
- [ ] 조건 상태 API (매 사이클 조건별 결과 + 현재 값)
- [ ] 카드 UI로 전략 시각화 (조건별 ✅❌ + 현재 값)
- [ ] DSL 직접 편집 모드
- [ ] 기존 v1 DSL/JSON 호환

### P1

- [ ] 시간 필드 (`시간`, `장시작후`, `요일`)
- [ ] 상태 함수 (`횟수`, `연속`, `실행횟수`)
- [ ] ATR, 최고가, 최저가, 횟수, 연속, 이격도 함수
- [ ] 브레이크이븐 (`횟수(수익률 >= N, 보유봉) >= 1` + 조건 조합으로 표현)
- [ ] DCA / 추가 매수 (진입가 가중평균 갱신)
- [ ] 시간 청산 (`보유일`, `보유봉`)
- [ ] 프리셋 템플릿 (§4 실전 패턴 기반)
- [ ] 파라미터 슬라이더 UI
- [ ] 트리거 이력 타임라인 UI
- [ ] 카드 UI ↔ DSL 양방향 변환
- [ ] `BETWEEN A AND B` 레인지 조건

### P2

- [ ] 파라미터 있는 커스텀 함수
- [ ] 린터/검증 경고 ("매수 조건에 추세 필터가 있는데 역조건 청산이 없습니다")

## 8. 범위 밖 (후속 spec으로 명시)

| 항목 | 이유 | 후속 |
|------|------|------|
| AI 자연어 → DSL 생성 | BYO LLM 연동 필요 | Phase E 별도 spec |
| Python 커스텀 함수 연동 | DSL에서 Python 함수 호출 (프로용 escape hatch) | 후속 spec |
| 절차적 함수 레이어 (루프, 변수) | 커스텀 함수 내부 절차적 로직 | Python 연동과 통합 |
| 포지션 사이징 (리스크 기반 수량) | execution 설정 확장 | 별도 spec |
| 복수 종목 참조 (`KOSPI지수 > MA(20)`) | 데이터 파이프라인 필요 | 시장 필터 spec |
| Walk-Forward / Monte Carlo | backtest-server 완료 후 | backtest-server spec |
| 전략 공유 포맷 / 마켓플레이스 | 커뮤니티 기능 | 별도 spec |
| 시험 모드 (모의 실행) | 실주문 없이 전략 동작 검증. 영웅문 "시험 적용"과 유사 | 별도 spec (온보딩/안전성) |
| 카드 UI 빌더 (전략 생성/편집) | AI 생성과 함께 설계해야 최적 | AI spec과 통합 |
| 포트폴리오 관리 (섹터 비중, 리밸런싱, 전략 간 예산 배분) | SystemTrader 거부권 확장으로 구현 가능. 개별 전략이 "제안"하고 포트폴리오 레이어가 "승인/거부/조정"하는 계층 구조. 현재 v2의 CandidateSignal→SystemTrader 흐름이 이 확장을 지원함 | 별도 spec |
| 스탑/IOC 주문 타입 | 증권사 API 확인 필요 | 별도 spec |
| Pine Script / EasyLanguage 변환 | AI 변환 의존 | AI spec과 통합 |
