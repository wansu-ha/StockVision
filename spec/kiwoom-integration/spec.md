# 키움 API 연동 명세서 (kiwoom-integration)

> 작성일: 2026-03-04 | 상태: 초안 | 담당: 로컬 브릿지 아키텍처

---

## 1. 개요

### 목표
StockVision AI 매매 신호를 키움 OpenAPI+를 통해 실제 거래로 실행하는 통합 시스템 구축.

### 법적 제약 (키움 G5 제5조②)
- **API 키/시크릿키는 제3자(서버) 저장 금지**
- 사용자 PC(로컬 브릿지)에서만 안전하게 보관·사용 가능
- 백엔드 서버는 **매매 신호만 전송**, 실제 주문은 로컬 브릿지가 실행

### 아키텍처 개요
```
StockVision Backend (FastAPI)
        ↓ (매매 신호: 종목, 수량, 매수/매도)
로컬 브릿지 (사용자 PC, Node.js/Python)
        ↓ (API 키로 인증)
키움 OpenAPI+ (Windows COM/OCX)
        ↓ (주문 전송)
키움 증권 거래소
```

## 2. 키움 OpenAPI+ 제약 사항

### 2.1 플랫폼 제약
- **Windows 전용** — COM/OCX 기반 (Linux/macOS 미지원)
- 사용자 PC에 키움증권 API 설치 필수
- 로그인은 자동 로그인(암호 저장) 또는 매번 ID/PW 입력 필요

### 2.2 API 호출 제한 (Rate Limiting)
| 기능 | 제한 |
|------|------|
| 조회 API (시세, 잔고 등) | 초당 5건 |
| 주문 API (매수/매도) | 초당 5건 |
| 실시간 등록 | 최대 200개 |

**대응 방안**
- 조회 큐 구현 (대기열 관리)
- 주문은 1초 간격으로 처리
- 급속 주문 시도는 클라이언트 검증으로 사전 차단

### 2.3 인증

**키움 오픈 API+는 별도 API 키가 없다.**
인증 = 키움증권 계좌 ID/PW + 공동인증서(실거래 시 필수).

| 인증 항목 | 모의투자 | 실거래 |
|-----------|:--------:|:------:|
| 키움 계좌 ID | ✅ | ✅ |
| 비밀번호 | ✅ | ✅ |
| 공동인증서 | ❌ | ✅ |

**채택 방식: 옵션 C — 영웅문 HTS 자동로그인 위임**
- 영웅문 HTS 자체 자동로그인 기능을 사용자가 직접 설정
- 우리 로컬 서버는 이미 로그인된 COM 세션에 **연결만** 수행
- 자격증명을 우리 앱이 직접 보관하지 않음 → 보안 위험 최소화
- 우리 `config.json`에 ID/PW 저장 없음

### 2.4 거래 시간 제약
- 평일 09:00 ~ 15:30 (주중)
- 야간/주말 거래 불가
- 공휴일 미운영

### 2.5 주문 타입
- **지정가**: 가격 입력 필수
- **시장가**: 즉시 체결
- **조건부 주문**: 일부 상품에만 지원
- **정정/취소**: 체결 전에만 가능

## 3. 로컬 브릿지 내 구현 범위

### 3.1 로컬 브릿지 역할
사용자의 PC에서 실행되는 애플리케이션 (Node.js 또는 Python)

**책임**
1. 키움 OpenAPI+ 인증 (앱키, 시크릿키)
2. 백엔드에서 수신한 매매 신호 처리
3. 주문 실행 (매수/매도)
4. 실시간 잔고·체결 정보 조회 및 로컬 저장
5. 통신 오류 시 재연결 및 상태 동기화
6. 보안: API 키는 메모리에만 보관, 디스크 저장 금지 (또는 암호화)

### 3.2 백엔드와의 통신
- **WebSocket** 또는 **HTTP 폴링**으로 신호 수신
- 신호 형식:
  ```json
  {
    "signal_id": "sig_abc123",
    "action": "buy|sell",
    "ticker": "005930",
    "quantity": 10,
    "price": "MARKET|70000",
    "timestamp": "2026-03-04T10:30:00Z"
  }
  ```
- 신호 전달 확인 (ACK) 백엔드로 반환

### 3.3 로컬 브릿지에서 처리하지 않는 것
- ❌ AI 신호 생성 (백엔드에서만)
- ❌ 포트폴리오 최적화 (백엔드에서만)
- ❌ 사용자 인증 (백엔드에서만)

## 4. 주요 기능 (로그인, 주문, 조회)

### 4.1 로그인 & 세션 관리

#### 4.1.1 초기 설정
1. 사용자가 로컬 브릿지 UI에서 "키움 연결" 버튼 클릭
2. 기존 자동 로그인 설정이 있으면 자동 진행
3. 없으면 ID/PW 입력 또는 자동 로그인 설정 후 진행

#### 4.1.2 세션 유지
- 로그인 후 세션 유지 (재로그인 불필요)
- 세션 타임아웃 시 자동 재로그인 시도 (최대 3회)
- 실패 시 사용자 알림

#### 4.1.3 모의투자 vs 실거래
```
로컬 브릿지 설정파일 (config.json — 우리 앱 자동저장)
{
  "kiwoom": {
    "mode": "demo|real"   // 모의투자 또는 실거래
    // ID/PW 저장 없음 — 영웅문 HTS 자동로그인에 위임
  }
}
```

**계정 전환 (재시작 없이 가능)**
1. UI에서 "계정 전환" → 실행 중인 전략 일시 중지
2. `KiwoomSession.logout()` 호출
3. 영웅문 HTS에서 다른 계정으로 재로그인 (사용자 직접)
4. `KiwoomSession.connect()` → 새 세션 획득
5. 새 프로파일 config 로딩 → 전략 재시작

### 4.2 주문 실행

#### 4.2.1 주문 요청 처리
백엔드 → 로컬 브릿지: 매매 신호 수신
```json
{
  "signal_id": "sig_20260304_001",
  "action": "buy",
  "ticker": "005930",    // 삼성전자
  "quantity": 10,
  "price": "MARKET",     // 또는 "70000"
  "comment": "AI Signal - Pattern Match"
}
```

#### 4.2.2 주문 검증 (Client-side)
- 잔고 확인 (충분한 자금/보유주 있는지)
- 거래 시간 확인 (09:00 ~ 15:30)
- 일일 거래량 제한 확인 (Rate Limit)
- 중복 신호 검사 (동일 신호 재전송 방지)

#### 4.2.3 주문 전송
```python
# 로컬 브릿지 (Python 예시)
kiwoom_api.send_order(
    rq_name="BUY_ORDER",
    screen_no="0101",
    acc_no="1234567890",
    order_type=1,  # 1=매수
    code="005930",
    qty=10,
    price=70000,
    hoga_gb="00"  # 시장가
)
```

#### 4.2.4 주문 결과 처리
- 주문 번호(ord_no) 수신
- 백엔드로 ACK 전송 (신호 처리 완료)
- 체결 시 실시간 업데이트 (WebSocket)

#### 4.2.5 주문 상태 추적
| 상태 | 설명 |
|------|------|
| PENDING | 주문 전송 대기 |
| SENT | 키움에 전송됨 |
| ACCEPTED | 거래소 접수 |
| PARTIALLY_FILLED | 일부 체결 |
| FILLED | 완전 체결 |
| CANCELLED | 취소됨 |
| REJECTED | 거부됨 |
| ERROR | 오류 발생 |

### 4.3 조회 API (시세, 잔고, 체결)

#### 4.3.1 잔고 조회
```
로컬 브릿지 → 키움 API → 계좌 잔고
응답:
{
  "account": "1234567890",
  "total_asset": 5000000,
  "cashable": 500000,
  "stocks": [
    {
      "code": "005930",
      "name": "삼성전자",
      "quantity": 100,
      "current_price": 70000,
      "eval_profit_loss": 50000
    }
  ]
}
```

#### 4.3.2 체결 조회
주문 후 체결 상태 폴링 (또는 이벤트 수신)
```
체결 정보:
{
  "order_id": "ord_123456",
  "filled_qty": 10,
  "filled_price": 70000,
  "filled_time": "10:30:45",
  "status": "FILLED"
}
```

#### 4.3.3 일봉/분봉 조회
AI 모델 재훈련을 위해 로컬 브릿지에서 실시간 캐시 유지
```
주기: 1분마다 갱신
저장: SQLite (로컬)
용도: 실시간 백테스트, 신호 검증
```

## 5. 에러 처리 및 안전장치

### 5.1 네트워크 오류
| 상황 | 처리 방식 |
|------|---------|
| 백엔드 연결 끊김 | 로컬 큐에 신호 저장, 복구 후 재전송 |
| 키움 API 타임아웃 | 5초 대기 후 재시도 (최대 3회) |
| 신호 수신 완료 ACK 미수신 | 백엔드에서 재전송, 로컬에서 중복 감지 후 무시 |

### 5.2 주문 오류
| 오류 코드 | 원인 | 대응 |
|---------|------|-----|
| A00 | 일반 오류 | 로그 기록, 사용자 알림 |
| B20 | 자금 부족 | 신호 거절, 백엔드에 알림 |
| C10 | 거래 시간 외 | 신호 큐에 저장, 시간 내에 재실행 |
| D00 | 통신 오류 | 재연결 시도 |

### 5.3 Rate Limit 관리
```python
# 조회 API: 초당 5건 제한
query_queue = []
last_query_time = time.time()

def send_query(query):
    global last_query_time
    if len(query_queue) >= 5 or \
       (time.time() - last_query_time) < 0.2:  # 200ms
        query_queue.append(query)
    else:
        send_to_kiwoom(query)
        last_query_time = time.time()

# 주기적으로 큐에서 처리
```

### 5.4 보안 안전장치
- **API 키 저장**: 메모리에만 (프로세스 종료 시 소멸)
- **전송 암호화**: HTTPS/WSS (TLS 1.3)
- **신호 검증**: 백엔드 서명(HMAC) 검증
- **감시 로깅**: 모든 주문 기록 (감시 목적)

### 5.5 복구 절차
로컬 브릿지 재시작 후:
1. 키움 자동 로그인
2. 미처리 신호 조회 (백엔드에 요청)
3. 미체결 주문 상태 확인
4. 상태 동기화 완료 알림

## 6. API 호출 제한 관리

### 6.1 조회 API 큐 (초당 5건)
```
조회 요청 타입:
1. 잔고 조회 (30초마다 자동)
2. 실시간 가격 (등록된 종목 자동 수신)
3. 체결 조회 (주문 후 1초마다)
4. 일봉/분봉 (백테스트용, 필요시)

큐 전략:
- 우선순위: 체결 > 실시간 > 잔고 > 일봉
- 버스트 방지: 요청 간격 200ms 최소 유지
```

### 6.2 주문 API 큐 (초당 5건)
```
주문 요청:
1. 신호에서 수신한 매수/매도
2. 손절/익절 (조건부 주문)
3. 정정/취소

큐 전략:
- FIFO (먼저 온 주문 먼저)
- 최대 버퍼: 100개 (초과시 오래된 신호 제거)
```

### 6.3 실시간 등록 (최대 200개)
```
실시간 데이터 등록:
- 현재 보유 종목: 자동 등록
- AI 신호 종목: 자동 등록
- 자동 해제: 포트폴리오에서 제거 후 3일

메모리 효율:
- Ticker 기준 중복 제거
- 보유 주식 변경 시 즉시 갱신
```

## 7. 기술 요구사항

### 7.1 로컬 브릿지 스택
| 항목 | 선택지 | 비고 |
|------|--------|-----|
| 언어 | Python 3.13 / Node.js 18+ | Windows COM 호출 → Python 권장 |
| 키움 API 래퍼 | pywin32 (COM) | 공식 키움 가이드 기반 |
| 백엔드 통신 | WebSocket (websockets) | 실시간 신호 수신 |
| 로컬 DB | SQLite | 오프라인 캐시, 거래 기록 |
| 설정 | JSON 파일 + 암호화 | API 키 AES-256 |

### 7.2 로컬 브릿지 API (내부)
```
POST /api/signal - 백엔드에서 신호 수신 (WebSocket)
POST /api/order - 주문 전송
GET /api/balance - 잔고 조회
GET /api/orders - 주문 상태 조회
POST /api/login - 키움 로그인
GET /api/health - 상태 체크
```

### 7.3 백엔드 연동 API
```
Signal Endpoint: /api/v1/trading/signals (WebSocket)
Response Format:
{
  "signal_id": "...",
  "action": "buy|sell",
  "ticker": "...",
  "quantity": 10,
  "price": "MARKET|70000",
  "timestamp": "...",
  "expiry": "2026-03-04T11:00:00Z"  // 신호 유효 시간
}

Bridge Callback: 주문 결과
POST /api/v1/trading/executions
{
  "signal_id": "...",
  "order_id": "ord_...",
  "status": "FILLED|REJECTED|ERROR",
  "filled_qty": 10,
  "filled_price": 70000,
  "filled_time": "2026-03-04T10:30:00Z",
  "error": null
}
```

### 7.4 설정 파일 구조 (config 분리)

**cloud_config.json** — 클라우드 동기화 가능
```json
{
  "kiwoom": {
    "mode": "demo"
  },
  "strategies": [...],
  "ui_preferences": {...}
}
```

**local_secrets.json** — 절대 클라우드 업로드 금지, .gitignore 등록
```json
{
  "account_no": "1234567890"
}
```
- ID/PW/인증서는 영웅문 HTS가 관리 → 우리 파일에 없음
- `local_secrets.json`에는 계좌번호만 (주문 시 필요)

### 7.5 로깅
```
로그 레벨: DEBUG, INFO, WARNING, ERROR
기록 대상:
- 모든 주문 (주문번호, 시간, 체결가격)
- 모든 조회 API 호출
- 오류 및 재시도
- 세션 관리 (로그인, 타임아웃)

파일: logs/kiwoom-{YYYYMMDD}.log
보관: 30일
```
