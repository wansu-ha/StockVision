## 1. 가능한 것 / 불가능한 것

### 1.1 한 장 요약

| | 우리가 해도 되는 것 | 하면 안 되는 것 |
|--|---|---|
| **데이터** | 지표, 차트, OHLCV, 스코어 제공 | "매수 추천", "상승 확률 85%" |
| **프롬프트** | 시스템 프롬프트 100% 작성 & 자동 감싸기 | — (프롬프트 자체는 도구) |
| **LLM 호출** | 사용자 API 키로 호출 | **우리 API 키로 호출** |
| **전략 코드** | 전략 프레임워크·파서·안전장치 제공 | "RSI 30 이하면 매수" 같은 판단 로직 기본 제공 |
| **실행** | 주문 래퍼, 드라이런, 킬스위치 | 사용자 동의 없는 자동 실행 |
| **비용** | 토큰 사용량 표시, 비용 추정 | **우리가 LLM 비용 부담** |

### 1.2 상세: 가능한 것

#### A. 프롬프트 완전 제공 — 합법

프롬프트는 **도구의 설계**에 해당한다. 망치의 모양을 설계하는 것이지, 대신 못을 박아주는 게 아니다.

```
가능한 것:
✅ 시스템 프롬프트를 우리가 100% 작성
✅ 항상 감싸서 보내기 (사용자가 프롬프트를 볼 필요 없음)
✅ 전략 프롬프트 템플릿 제공 (변수 바인딩 포함)
✅ 응답 JSON 스키마 정의 & 파싱
✅ 프롬프트 버전 관리 & 백테스트 연결
```

사용자 UX: API 키만 입력하고 버튼 하나로 분석 실행 가능.

#### B. 데이터 포맷팅 & 컨텍스트 빌드 — 합법

```
가능한 것:
✅ OHLCV → JSON/Markdown 자동 변환
✅ 기술적 지표 (RSI, MACD, 볼린저) 자동 포함
✅ 포트폴리오 상태 자동 첨부
✅ 뉴스 감성 점수 (원문 아닌 점수만)
✅ 스코어링 엔진 결과 참고용 첨부
```

#### C. 안전장치 프레임워크 — 합법 + 필수

```
가능한 것:
✅ 손실 한도, 포지션 한도, 쿨다운 프레임워크 제공
✅ 기본값 설정 (예: 드라이런 ON, 일일 손실 50만원)
✅ 킬스위치, 자동 중단 로직
✅ 안전장치 위반 로그 기록
```

임계값의 "기본값"은 제공해도 된다. "이 값이 적당합니다"라는 추천만 아니면 됨.

#### D. 백테스팅 & 모니터링 — 합법

```
가능한 것:
✅ 과거 데이터로 프롬프트 전략 검증
✅ 수익률, 샤프비율, MDD, 승률 등 숫자 제공
✅ 프롬프트 버전별 성과 비교
✅ 실행 이력 로그, 대시보드
✅ 토큰 사용량 & 비용 추정
```

숫자만 보여주고, "이 전략이 좋다/나쁘다" 평가는 하지 않으면 됨.

### 1.3 상세: 불가능한 것 (투자자문업 등록 없이)

#### A. 우리 API 키로 LLM 호출 — 불가

```
불가한 것:
❌ 우리 API 키로 Claude/GPT 호출 후 결과를 사용자에게 전달
❌ 프록시 서버에서 LLM 호출 (키가 우리 것이면)
❌ LLM 비용을 우리가 부담하는 "무료 AI 분석" 기능
```

**이유**: LLM 호출 주체가 우리이면, 그 결과물의 제공자도 우리.
결과에 매수/매도 판단이 포함되면 **자본시장법 제6조 제7항** 투자자문업에 해당.

#### B. 전략 판단 로직 기본 제공 — 불가

```
불가한 것:
❌ "RSI 30 이하면 매수 추천" 같은 기본 전략 프리셋
❌ "이 프롬프트가 성과가 좋습니다" 같은 전략 추천
❌ "모멘텀 전략을 추천합니다" 같은 전략 평가
❌ 백테스트 결과에 "이 전략이 우수합니다" 같은 해석
```

빈 틀과 작성법 가이드는 OK. 채워진 전략은 NO.

#### C. 서버에서 투자 판단 내려보내기 — 불가

```
불가한 것:
❌ API 응답에 "recommendation": "매수"
❌ API 응답에 "prediction": "상승 확률 85%"
❌ API 응답에 "action": "buy"
❌ API 응답에 "target_price": 85000
```

### 1.4 경계선 위의 것들 (주의 필요)

| 항목 | 판단 | 근거 |
|------|------|------|
| 프롬프트 작성 가이드 | ✅ OK | "변수 사용법, 응답 포맷 설명" 수준 |
| 전략 작성 예시 (교육용) | ⚠️ 주의 | "예시입니다, 투자 판단이 아닙니다" 명시 필요 |
| 기본 응답 스키마 | ✅ OK | 빈 구조만 (`action`, `ticker` 필드 정의) |
| 백테스트 성과 숫자 | ✅ OK | 숫자만, 해석/평가 없이 |
| 안전장치 기본값 | ✅ OK | "합리적 기본값"이지 "추천"이 아님 |
| 프롬프트 내용 공개 | ⚠️ 불가피 | 프론트에서 사용자 키로 호출하면 Network 탭에 노출 |
| 서버 프록시로 프롬프트 숨기기 | ⚠️ 위험 | "우리가 호출하는 것"으로 해석될 여지 |

## 2. 프롬프트 제공 방식 — 기술적 선택지

### 2.1 방식 A: 프론트엔드 직접 호출 (현재 spec 방향)

```
[프론트엔드]
  ├── 시스템 프롬프트 (우리가 작성, 하드코딩 또는 서버에서 받음)
  ├── 컨텍스트 데이터 (서버 API에서 받음)
  ├── 사용자 API 키 (로컬 저장)
  └── LLM API 직접 호출 → 결과 표시
```

| 장점 | 단점 |
|------|------|
| 법적으로 가장 안전 | 프롬프트가 Network 탭에 노출 |
| 구조가 명확 | CORS 이슈 (LLM API가 브라우저 호출 허용해야 함) |
| 서버 부하 없음 | API 키가 브라우저에 존재 |

**CORS 문제**: Claude API, OpenAI API 모두 브라우저에서 직접 호출 시 CORS 에러 발생 가능.
→ 이 경우 방식 B가 필요해짐.

### 2.2 방식 B: 백엔드 프록시 (사용자 키 전달)

```
[프론트엔드] → [우리 백엔드] → [LLM API]
                 ↑ 사용자 API 키를 요청에 포함
                 ↑ 우리는 키를 저장하고 대신 호출
```

| 장점 | 단점 |
|------|------|
| CORS 문제 없음 | 프롬프트 숨길 수 있지만 법적 경계 모호 |
| 프롬프트 보호 가능 | API 키가 우리 서버를 경유 |
| 로깅/모니터링 용이 | "우리가 호출했다" 해석 여지 |

**법적 분석**: 사용자 키를 사용하고, 비용도 사용자가 부담하므로, 서버가 단순 프록시 역할이면 도구 제공으로 볼 수 있다. 하지만 보수적으로 보면 위험.

### 2.3 방식 C: 하이브리드 (권장)

```
[프론트엔드]
  ├── 컨텍스트 데이터 요청 → [우리 백엔드] → JSON 반환
  ├── 프롬프트 조립 (프론트에서)
  └── LLM API 직접 호출 (사용자 키)
       ↓ 실패 시 (CORS)
       └── 백엔드 프록시 폴백 (사용자 키 전달)
```

- 기본: 프론트 직접 호출 (법적 최안전)
- 폴백: CORS 불가 시 백엔드 프록시 (사용자 키 사용 명시)
- 면책: "LLM 호출은 사용자 API 키로 수행되며, 플랫폼은 데이터 포맷팅 도구만 제공합니다"

### 2.4 프롬프트 공개 여부

**결론: 숨길 수 없고, 숨길 필요도 없다.**

- 방식 A/C: Network 탭에서 프롬프트 확인 가능
- 방식 B: 프롬프트는 숨길 수 있지만 법적 리스크 증가
- 실질적으로 프롬프트만으로는 경쟁력이 제한적 (아래 3장 참조)

## 3. 경쟁력 분석

### 3.1 StockVision 포지셔닝

```
[완전 위임]                                              [완전 자유]
핀트/Stoic ←────── Composer ←── StockVision ──→ 젠포트 ──→ QuantConnect/Alpaca
라이선스 필요        AI 전략빌더     AI+DIY       DIY 퀀트      인프라
```

**StockVision = "젠포트의 접근성 + LLM 전략 생성" → 국내 미개척 영역**

### 3.2 경쟁 우위 (유리한 점)

#### 1. LLM 통합 — 국내 최초 포지션

| 경쟁자 | LLM 지원 | 비고 |
|--------|---------|------|
| 젠포트 | ❌ | UI 기반 조건식만 |
| 인텔리퀀트 | ❌ | JavaScript 코딩만 |
| 트레이딩뷰 | ❌ (웹훅 우회) | Pine Script만 |
| 핀트 | 내부만 | 사용자 LLM 연동 불가 |
| **StockVision** | **✅** | **사용자 LLM 직접 연결** |

젠포트의 가장 큰 진입장벽인 "조건식 설계"를 LLM으로 대체할 수 있다.
→ "RSI가 30 이하이고 거래량이 급증한 종목 찾아줘"를 자연어로 입력 → LLM이 판단

#### 2. 안전장치 내장 — Composer의 반면교사

Composer.trade는 전략 빌더는 강력하지만 네이티브 리스크 관리가 없어 써드파티(Trade Conductor)가 등장.
StockVision은 **Phase 2부터 안전장치를 내장** (손실 한도, 킬스위치, 드라이런).

```
Composer: 전략 빌더 ✅ + 리스크 관리 ❌ → 써드파티 필요
StockVision: 전략 빌더 ✅ + 리스크 관리 ✅ → 원스톱
```

#### 3. 풀 파이프라인 — 데이터부터 백테스트까지

```
데이터 수집 → 지표 계산 → LLM 분석 → 안전장치 → 가상매매 → 백테스트 → 모니터링
     ↑                                                              ↑
  이미 구현                                                      이미 구현
```

OpenBB는 분석만, Alpaca는 실행만, 젠포트는 LLM이 없다. StockVision은 전 구간을 커버.

#### 4. 라이선스 불필요 — 진입장벽 낮음

사용자 키로 LLM 호출하는 구조이므로 투자자문업 등록 없이 서비스 가능.
핀트/파운트/AIM은 투자일임업 라이선스 + 코스콤 검증이 필요.

#### 5. 비용 구조 유리

| 항목 | StockVision | 핀트 | 젠포트 |
|------|------------|------|--------|
| 플랫폼 수수료 | 무료 (오픈소스) | 수익의 9.5% | 매매 0.1% |
| AI 비용 | 사용자 부담 (API 키) | 내재 | 없음 |
| 데이터 비용 | 무료 (yfinance) | 내재 | 내재 |

사용자가 부담하는 건 LLM API 비용뿐 — Claude Haiku 기준 분석 1회 ≈ $0.001~0.01.

### 3.3 경쟁 약점 (불리한 점)

#### 1. 프롬프트 = 공개 자산

프론트엔드에서 사용자 키로 호출하면 프롬프트가 노출된다.
경쟁자가 프롬프트를 복사할 수 있다.

**완화**: 프롬프트는 전체 가치의 일부. 진짜 경쟁력은 아래 참조.

#### 2. LLM 의존성 — 환각/오류 리스크

LLM은 틀린 분석을 할 수 있다. 사용자가 LLM 결과를 맹신하면 손실 발생.

**완화**:
- 드라이런 기본 활성화
- 안전장치 프레임워크
- 면책 조항 필수 표시
- 백테스트로 사전 검증 권장

#### 3. 데이터 품질 한계

yfinance는 지연 데이터 + 일봉 위주. 실시간 호가가 없다.

**완화**: Phase 4에서 키움 REST API 연동 예정. 현재는 "일봉 기반 스윙 트레이딩"에 적합.

#### 4. 사용자 진입장벽 — API 키 필요

일반 사용자에게 "API 키를 발급받으세요"는 허들이 될 수 있다.

**완화**:
- Ollama (로컬 LLM) 지원으로 API 키 없이도 사용 가능
- API 키 발급 가이드 UI 내 제공
- 스코어링 엔진 (Phase 2)은 LLM 없이도 동작

#### 5. 규모 & 신뢰 — 신생 서비스

젠포트는 검증된 사용자 베이스, 핀트는 AUM 3,000억원. StockVision은 아직 초기.

**완화**: 오픈소스로 투명성 확보, 가상 거래로 리스크 없이 시작 가능.

### 3.4 진짜 경쟁력 = 프롬프트가 아닌 것들

프롬프트가 공개되더라도 다음은 복사할 수 없다:

| 경쟁력 요소 | 복사 난이도 | 설명 |
|------------|-----------|------|
| 데이터 파이프라인 | 높음 | yfinance → 지표 계산 → 컨텍스트 빌드 전체 체인 |
| 안전장치 프레임워크 | 높음 | 손실한도/포지션한도/킬스위치/드라이런 통합 |
| 백테스팅 엔진 (LLM 전략) | 높음 | 날짜별 컨텍스트 재생성 + LLM 호출 + 성과 측정 |
| UI/UX 완성도 | 중간 | 프로바이더 관리, 프롬프트 에디터, 모니터링 대시보드 |
| 풀 파이프라인 통합 | 높음 | 데이터→분석→실행→백테스트→모니터링 원스톱 |
| 커뮤니티 & 개선 속도 | 높음 | 오픈소스 기여, 사용자 피드백 루프 |

## 4. 규제 환경 (2026년 기준)

### 4.1 적용 법령

| 법령 | 적용 여부 | StockVision 영향 |
|------|----------|-----------------|
| 자본시장법 제6조 (투자자문업) | 회피 가능 | 사용자 키로 호출 → 도구 제공 |
| 인공지능기본법 (2026.1 시행) | 적용 | AI 서비스 제공자로서 기본 의무 |
| 금융분야 AI 가이드라인 (통합 개정) | 참고 | 자율규제, 법적 강제력 없음 |
| 금융분야 AI RMF (2026 Q1) | 참고 | 자율규제, 금융회사 대상 |
| 금융소비자보호법 | 부분 적용 | 면책 조항, 위험 고지 필요 |
| 개인정보보호법 | 적용 | API 키 암호화 저장 필수 |

### 4.2 인공지능기본법 대응

2026년 1월 시행된 AI 기본법의 7대 원칙:

1. **거버넌스** → AI 관련 의사결정 체계 문서화
2. **합법성** → 관련 법령 검토 (이 문서)
3. **보조수단성** → AI는 도구, 최종 판단은 사용자 (현재 구조 부합)
4. **신뢰성** → 드라이런 기본, 안전장치 내장
5. **금융안정성** → 가상 거래 기반, 실거래 시 안전장치
6. **신의성실** → 면책 조항, 데이터 출처 표시
7. **보안성** → API 키 암호화, 실행 로그 보존

### 4.3 필수 구현 항목

```
[이미 계획됨 — spec에 포함]
✅ API 키 암호화 저장 (FR-1.4)
✅ 드라이런 기본 활성화 (FR-5.7)
✅ 킬스위치 (FR-5.6)
✅ 실행 이력 보존 (FR-6.5)

[추가 필요 — 이 분석에서 도출]
⬜ 면책 조항 UI (서비스 하단 + 첫 사용 시 동의)
⬜ "투자 자문이 아닙니다" 고지 (AI 분석 결과 표시 시 항상)
⬜ 데이터 출처 표시 ("yfinance 기반 지연 데이터")
⬜ AI 분석임을 명시 ("AI 생성 결과이며 오류가 포함될 수 있습니다")
⬜ 이용약관에 AI 서비스 관련 조항 추가
```

## 6. 결론

### 6.1 핵심 판단

| 질문 | 답 |
|------|---|
| LLM을 우리가 붙여줄 수 있나? | **사용자 키로 호출하면 가능** |
| 프롬프트를 우리가 다 작성해도 되나? | **된다** — 프롬프트는 도구 |
| 프롬프트를 항상 감싸서 보내도 되나? | **된다** — 사용자 선택 불필요 |
| 프롬프트를 비밀로 할 수 있나? | **프론트 직접 호출 시 못 숨김** |
| 전략 코드를 제공해도 되나? | **프레임워크는 OK, 판단 로직은 NO** |
| API 없이 가능한가? | **불가** — LLM 호출에 API 키 필수 (Ollama 제외) |
| 투자자문업 등록 필요한가? | **현재 구조에서는 불필요** |

### 6.2 경쟁력 요약

```
강점:
  • 국내 최초 LLM 통합 DIY 트레이딩 플랫폼
  • 안전장치 내장 (Composer 대비 차별화)
  • 풀 파이프라인 (OpenBB/Alpaca 대비 차별화)
  • 라이선스 불필요 (핀트/파운트 대비 진입장벽 낮음)
  • 비용 구조 유리 (오픈소스 + 사용자 API 키)

약점:
  • 프롬프트 비공개 불가 (경쟁자 복사 가능)
  • LLM 환각 리스크 (면책 + 안전장치로 완화)
  • yfinance 데이터 한계 (Phase 4에서 해결)
  • API 키 진입장벽 (Ollama + 가이드로 완화)
  • 신뢰/규모 부족 (오픈소스 + 가상거래로 극복)
```

| 우선순위 | 항목 | 근거 |
