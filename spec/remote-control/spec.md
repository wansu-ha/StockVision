# 원격 제어 (Remote Control)

> 작성일: 2026-03-12 | 상태: 구현 완료 (SUPERSEDED by `spec/remote-ops/`) | Phase C (C6 + C7 + C8)

## 1. 배경

현재 StockVision은 같은 PC 로컬 모드에서만 사용 가능하다. 외출 중 상태 확인이나 긴급 정지가 불가능하여 사용자가 불안을 느낀다.

권한 모델(`docs/product/remote-permission-model.md`)은 초안 상태이며, 핵심 원칙은 확정:
- 조회는 넓게, 실행은 좁게
- 정지는 쉽게, 재개는 어렵게
- 원격 수동 실주문은 초기 금지

이 spec은 원격 모드의 **첫 번째 구현 범위**를 정의한다.

## 2. 목표

밖에서 폰으로 시스템 상태를 확인하고, 위험 시 긴급 정지할 수 있다. 재개는 추가 확인을 거쳐야만 가능하다.

## 3. 범위

### 3.1 포함

**A. 원격 상태 조회**
- 원격 앱/웹에서 로컬 서버 상태 조회 (클라우드 릴레이 경유)
- 계좌 요약 (잔고, 보유 종목, 평가 손익)
- 전략 실행 상태 (활성 전략 수, 오늘 신호/체결/오류)
- 최근 실행 로그 (최근 10건)
- 로컬 서버 온라인/오프라인 상태 표시

**B. 긴급 정지 (킬스위치)**
- 원격에서 엔진 즉시 정지 (1탭)
- 정지 시 신규 주문 차단, 기존 미체결 주문 유지 또는 취소 선택
- 정지 확인 알림 (로컬 서버에서 정지 반영 확인)

**C. 엔진 재개/무장(arm)**
- 원격에서 엔진 재개 시 추가 확인 필요 (2단계 확인)
- 재개 가능 조건: 로컬 서버 온라인 + 브로커 연결 + 비정상 상태 없음
- 메신저 모드에서는 재개 금지

### 3.2 제외

- 원격 수동 실주문 (초기 금지 정책)
- 원격 설정 변경 (민감 설정)
- 메신저 봇 연동 (Phase D)
- 로컬 서버 오프라인 시 큐잉/재전달 (후속 이터레이션)

## 4. 의존성

| 의존 대상 | 상태 | 비고 |
|-----------|------|------|
| 권한 모델 (`remote-permission-model.md`) | 초안 | 원칙 확정, 세부 정책은 구현 시 확정 |
| 클라우드 서버 인증 | 완료 | JWT 기반 인증 존재 |
| 로컬 서버 API | 완료 | status, engine start/stop 엔드포인트 존재 |

**선행 작업: 권한 모델 확정 (초안 → 확정)**
- 문서 내용은 이미 구체적 (채널별 권한 표, 세부 정책 존재)
- "확정" 전환 기준: 채널별 권한 표 최종 리뷰 + 확인 코드 메커니즘 합의

## 5. 아키텍처

### 5.1 통신 경로

```
[원격 앱/웹] → [클라우드 서버] → [로컬 서버]
                 (릴레이)        (실행)
```

### 5.2 클라우드 릴레이 설계

클라우드 서버가 로컬 서버에 요청을 중계한다.
로컬 서버는 클라우드에 주기적으로 상태를 동기화(push)하거나, 클라우드가 요청 시 로컬에 pull한다.

**초기 구현 권장: Push 모델**
- 로컬 서버가 주기적으로 (30초) 상태 스냅샷을 클라우드에 전송
- 클라우드는 최신 스냅샷을 저장, 원격 클라이언트에 제공
- 킬스위치 명령은 클라우드 → 로컬 WebSocket 또는 long-poll로 전달

### 5.3 오프라인 처리

로컬 서버 오프라인 시:
- 마지막 동기화 스냅샷 표시 + "마지막 동기화: 5분 전" 경고
- 긴급 정지 명령은 클라우드에 저장, 로컬 복귀 시 즉시 적용
- 재개/무장 명령은 오프라인 시 금지

## 6. 원격 상태 조회 설계

### 6.1 상태 스냅샷 모델

```typescript
interface RemoteStatusSnapshot {
  synced_at: string              // 마지막 동기화 시각
  local_online: boolean
  broker_connected: boolean
  broker_mode: 'mock' | 'live'
  engine_state: 'running' | 'stopped' | 'halted' | 'error'
  armed: boolean                 // 엔진 무장 상태
  account_summary: {
    total_balance: number
    total_eval: number
    daily_pnl: number
    daily_pnl_pct: number
  }
  today_stats: {
    signals: number
    fills: number
    errors: number
  }
  recent_logs: LogEntry[]        // 최근 10건
}
```

### 6.2 API

```
GET /api/v1/remote/status
→ RemoteStatusSnapshot
```

인증: 클라우드 JWT + 원격 모드 권한 확인

## 7. 킬스위치 설계

### 7.1 정지 API

```
POST /api/v1/remote/kill
Body: { mode: 'stop_new' | 'stop_all' }
```

- `stop_new`: 신규 주문만 차단, 기존 미체결 유지
- `stop_all`: 신규 차단 + 기존 미체결 전부 취소 요청

### 7.2 정지 흐름

1. 사용자가 원격에서 킬스위치 탭
2. 확인 다이얼로그 ("엔진을 정지합니다. 계속하시겠습니까?") — 1회 확인
3. 클라우드 서버가 로컬 서버에 정지 명령 전달
4. 로컬 서버가 엔진 정지 + ArmSession 해제
5. 결과를 클라우드에 보고 → 원격 클라이언트에 확인 표시

### 7.3 오프라인 킬스위치

로컬 서버 오프라인 시:
- 클라우드에 정지 명령 저장 (pending_kill)
- 원격 화면에 "정지 명령 전송됨 — 로컬 서버 복귀 시 즉시 적용됩니다" 표시
- 로컬 서버 복귀 → 클라우드 동기화 시 pending_kill 확인 → 즉시 정지

## 8. 엔진 재개/무장 설계

### 8.1 재개 조건

원격에서 재개하려면 **모든 조건** 충족 필요:
- 로컬 서버 온라인
- 브로커 연결됨
- 엔진 비정상 상태 없음 (HALTED 아님)
- 미해결 경고 없음

### 8.2 재개 API

```
POST /api/v1/remote/arm
Body: { confirm_code: string }
```

- 2단계 확인: 첫 요청 → 서버가 확인 코드 생성 → 사용자가 코드 입력 → 재개
- 확인 코드는 30초 유효
- **참고**: 확인 코드 메커니즘은 기존 코드베이스에 없는 새 패턴. 구현 시 보안 검토 필요 (brute-force 방지, 시도 횟수 제한 등)

### 8.3 재개 불가 사유 표시

조건 미충족 시 구체적 사유 표시:
- "브로커 미연결 — 로컬 PC에서 연결하세요"
- "킬스위치 발동 후 — 로컬 PC에서 재무장하세요"
- "로컬 서버 오프라인 — 서버를 시작하세요"

## 9. 프론트엔드 설계

### 9.1 원격 모드 감지

로컬 서버에 직접 연결 가능 → 로컬 모드
로컬 서버 직접 연결 불가 + 클라우드 인증 → 원격 모드

### 9.2 원격 모드 UI 차이

- 헤더에 "원격 모드" 배지 표시
- 실행/설정 관련 기능 비활성화 (읽기 전용)
- 킬스위치 버튼은 항상 활성
- 재개 버튼은 조건부 활성

## 10. 수용 기준

### 원격 상태 조회
- [ ] 원격에서 로컬/브로커/엔진 상태를 확인할 수 있다
- [ ] 계좌 요약 (잔고, 평가, 손익)이 표시된다
- [ ] 오늘의 신호/체결/오류 수가 표시된다
- [ ] 로컬 서버 오프라인 시 마지막 동기화 시각이 표시된다

### 긴급 정지
- [ ] 원격에서 1탭 + 1회 확인으로 엔진을 정지할 수 있다
- [ ] 정지 후 확인 피드백이 표시된다
- [ ] 로컬 서버 오프라인 시에도 정지 명령이 저장된다

### 엔진 재개/무장
- [ ] 원격에서 2단계 확인을 거쳐 엔진을 재개할 수 있다
- [ ] 조건 미충족 시 구체적 사유가 표시된다
- [ ] 로컬 서버 오프라인 시 재개가 금지된다

### PWA + 웹 푸시
- [ ] 프론트엔드가 PWA로 설치 가능하다 (manifest.json + Service Worker)
- [ ] Android Chrome에서 웹 푸시 알림이 수신된다
- [ ] iOS Safari (16.4+)에서 홈화면 추가 후 웹 푸시가 수신된다
- [ ] 체결/오류/긴급 상황별 개인화된 알림이 전송된다
- [ ] 푸시 알림 클릭 시 해당 화면으로 이동한다

### 외부 주문 감지
- [ ] 엔진 외부에서 발생한 주문이 감지된다
- [ ] 외부 주문 감지 시 경고 알림이 표시된다
- [ ] 설정에 따라 외부 주문 감지 시 엔진 자동 정지가 가능하다

## 11. 전달 채널: PWA + 웹 푸시 (FCM)

### 11.1 채널 결정 (2026-03-12)

| 결정 | 내용 |
|------|------|
| 주 채널 | **PWA** — 기존 React SPA에 반응형 + Service Worker 추가 |
| 알림 | **웹 푸시 (FCM)** — Android 완벽, iOS 16.4+ 지원 |
| 메신저 | 당장은 안 함. iOS 푸시 불만 시 보조 채널로 검토 |
| 네이티브 앱 | 과도. 사용자 규모 커지면 검토 |

근거: 프론트엔드 코드베이스 하나로 PC(풀 기능) + 모바일(모니터링/알림/긴급 제어) 커버. 별도 앱/봇 코드베이스 불필요. 한국 Android 70%+ 점유율.

### 11.2 PWA 구성

**프론트엔드 추가 파일**:
- `public/manifest.json` — 앱 이름, 아이콘, 테마 색상, display: standalone
- `public/sw.js` — Service Worker (캐시 전략 + 푸시 수신)
- `public/firebase-messaging-sw.js` — FCM 백그라운드 메시지 처리

**기존 파일 수정**:
- `index.html` — manifest 링크, 메타 태그 (theme-color, apple-mobile-web-app-capable)
- `App.tsx` — Service Worker 등록, 푸시 권한 요청 흐름

### 11.3 푸시 알림 아키텍처

```
로컬 서버 (이벤트 발생)
    ↓ 하트비트/이벤트 채널
클라우드 서버
    ↓ FCM HTTP v1 API
Firebase Cloud Messaging
    ↓
사용자 브라우저/PWA (Service Worker)
    ↓
OS 알림 센터
```

### 11.4 푸시 토큰 관리

```
POST /api/v1/push/register
Body: { push_token: string, platform: 'web' | 'android' | 'ios' }
```

- 사용자 로그인 시 브라우저가 FCM 토큰 발급
- 프론트가 클라우드 서버에 `{ user_id, push_token }` 등록
- JWT 만료와 무관 — 푸시 토큰은 별도 채널
- 토큰 갱신 시 자동 업데이트

### 11.5 알림 유형

| 이벤트 | 제목 | 본문 예시 | 우선순위 |
|--------|------|-----------|---------|
| 체결 | "체결 완료" | "삼성전자 10주 매수 — 72,400원" | normal |
| 주문 실패 | "주문 실패" | "SK하이닉스 매수 실패 — 일일 예산 초과" | high |
| Kill Switch | "긴급: 엔진 정지" | "Kill Switch 발동 — 신규 주문 차단됨" | high |
| Loss Lock | "손실 한도 도달" | "일일 손실 한도 도달 — 자동 정지" | high |
| 엔진 중단 | "엔진 응답 없음" | "엔진이 3분째 응답 없음 — 확인 필요" | high |
| 브로커 연결 해제 | "브로커 연결 끊김" | "키움증권 연결 해제 — 재연결 시도 중" | high |

### 11.6 모바일 반응형 UI

기존 PC 레이아웃을 모바일에서도 사용 가능하도록 반응형 처리:
- OpsPanel: 가로 스크롤 또는 2줄 래핑
- ListView: 카드형 레이아웃 전환
- DetailView: 탭 기반 세로 배치
- 킬스위치 버튼: 하단 고정 FAB (Floating Action Button)

## 12. 외부 주문 감지 (C8)

### 12.1 배경

사용자가 HTS/MTS에서 직접 주문을 넣으면 StockVision 엔진이 알지 못하는 포지션 변동이 발생한다. 이로 인해:
- 엔진이 이미 매수된 종목에 중복 매수를 시도할 수 있음
- 리스크 한도 계산이 실제와 맞지 않을 수 있음
- 일일 P&L 집계가 불완전할 수 있음

### 12.2 감지 방식

엔진의 주기적 잔고 조회(reconciliation) 시:
1. 브로커 API로 현재 잔고 조회
2. 엔진이 알고 있는 포지션(`PortfolioSnapshot`)과 비교
3. 불일치 발견 시 `external_order_detected = True` 플래그 설정

### 12.3 감지 시 동작

| 설정 | 동작 |
|------|------|
| `warn_only` (기본) | 경고 로그 + WS 알림 + OpsPanel 배너. 엔진 계속 실행 |
| `auto_stop` | 경고 + 엔진 자동 정지. 사용자 확인 후 재개 |

### 12.4 API

```
GET /api/status → data.strategy_engine.external_order_detected: boolean
```

기존 status 엔드포인트에 필드 추가.

### 12.5 프론트엔드

OpsPanel 경고 배너에 추가:
```
⚠ 외부 주문 감지 — StockVision 외부에서 주문이 실행되었습니다. 잔고를 확인하세요.
```

### 12.6 수용 기준

- [ ] 엔진 외부에서 발생한 포지션 변동이 감지된다
- [ ] 감지 시 OpsPanel에 경고 배너가 표시된다
- [ ] `auto_stop` 설정 시 엔진이 자동 정지된다
- [ ] 감지 이벤트가 실행 로그에 기록된다
