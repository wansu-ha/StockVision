# /plan — 구현 계획서 작성

feature-name: $ARGUMENTS

`$ARGUMENTS`가 비어 있으면 대화 맥락과 프로젝트 상태를 분석하여 feature-name을 제안하고, 사용자 확인 후 진행한다.

## 실행 절차

1. **spec 확인**: `spec/{feature-name}/spec.md` 존재 여부 확인.
   - 없으면 경고: "spec이 없습니다. /spec $ARGUMENTS를 먼저 실행하세요." → 사용자 확인 후 계속 또는 중단.

2. **코드베이스 분석**: spec 기반으로 수정/생성 대상 파일과 영향 범위를 파악한다.

3. **plan 파일 생성**: `spec/$ARGUMENTS/plan.md` 생성. 아래 섹션 포함:
   - **아키텍처**: 컴포넌트 간 관계, 데이터 흐름
   - **수정 파일 목록**: 파일별 변경 내용 요약
   - **구현 순서**: 단계별 작업과 각 단계의 검증 방법
   - **검증 방법**: 빌드, 테스트, 브라우저 확인 항목

4. **브랜치 생성**: 현재 브랜치명에 `$ARGUMENTS`가 포함되어 있으면 그대로 사용. 아니면 `feat/$ARGUMENTS` 브랜치를 생성하고 체크아웃.

5. **커밋**: spec.md + plan.md를 함께 스테이징하고 커밋.
   - 커밋 메시지: `docs: $ARGUMENTS spec/plan 초안`
   - `git diff` 확인 후 스테이징

6. **사용자 리뷰 요청**: plan 작성 후 사용자에게 리뷰를 요청한다.

## 규칙

- plan이 이미 존재하면 덮어쓰지 말고 사용자에게 확인
- 구현 순서는 의존성 순서대로 배치 (독립 → 의존)
- 각 단계에 검증 방법을 명시 (verify 항목)
