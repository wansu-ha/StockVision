# /spec — 기능 명세서 작성

feature-name: $ARGUMENTS

`$ARGUMENTS`가 비어 있으면 대화 맥락과 프로젝트 상태를 분석하여 feature-name을 제안하고, 사용자 확인 후 진행한다.

## 실행 절차

1. **코드베이스 탐색**: feature-name 관련 기존 코드, 패턴, 의존성을 탐색한다.
   - `docs/architecture.md`, `docs/development-plan.md` 참조
   - 관련 API, 모델, 컴포넌트 파악

2. **spec 파일 생성**: `spec/$ARGUMENTS/spec.md` 생성. 아래 섹션 포함:
   - **목표**: 이 기능이 해결하는 문제
   - **요구사항**: 기능적/비기능적 요구사항 목록
   - **수용 기준**: 완료 판단 기준 (체크리스트)
   - **범위**: 포함/미포함 사항 명시
   - **API/DB 변경**: 필요한 엔드포인트, 스키마 변경 스케치
   - **참고**: 관련 기존 코드 경로

3. **사용자 리뷰 요청**: spec 작성 후 사용자에게 리뷰를 요청한다. 커밋은 `/plan`과 묶어서 한다.

## 규칙

- spec이 이미 존재하면 덮어쓰지 말고 사용자에게 확인
- 추측으로 요구사항을 채우지 말 것. 불명확하면 사용자에게 질문
- 프로젝트 맥락(기술 스택, 기존 패턴)에 맞게 작성
