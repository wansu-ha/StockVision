# 커밋 & 이터레이션 규칙

## 브랜치 커밋 순서

1. `docs: spec/plan 초안` — 문서 먼저 커밋
2. `feat: 구현 + spec/plan 수정` — 코드와 문서 변경을 하나의 커밋으로 묶음
3. 이터레이션 반복 후 `--no-ff` 병합

## 커밋 워크플로우 (Claude Code 제약으로 인한 실용 방식)

`git commit`은 Claude Code 내부 로직으로 항상 허가 요청이 발생한다.
커밋 오가락에 컨텍스트를 낭비하지 않기 위해 아래 방식 사용:

### 개발 중 (커밋 없이 진행)
- 각 단계 구현 완료 시 → 변경 파일/내용을 `spec/{feature}/reports/stepN.md`에 기록
- `git add`는 하지 않음 — 개발에 집중

### 전체 완료 후 (일괄 커밋)
1. 단계별 커밋 목록 제시 (Step N: 메시지, 파일 목록)
2. 각 단계마다 `git add` + `git commit` 순서대로 실행
3. 사용자가 단계별로 허가 클릭 → 히스토리 유지하면서 일괄 커밋

## MUST 규칙

- 각 개발 단계마다 spec/plan 먼저 수정 후 코드 변경
- research 문서는 해당 이터레이션에 같이 커밋 (나중에 따로 넣으면 히스토리 맥락 안 맞음)
- `git add` 전 diff 확인 필수 — 사용자 수정사항이 포함되지 않았는지 확인
- `/implement` 완료 후 로컬 브랜치 삭제 안 함 (`git branch -d` 생략)
