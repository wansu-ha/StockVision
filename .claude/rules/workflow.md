# 커밋 & 이터레이션 규칙

## 브랜치 커밋 순서

1. `docs: spec/plan 초안` — 문서 먼저 커밋
2. `feat: 구현 + spec/plan 수정` — 코드와 문서 변경을 하나의 커밋으로 묶음
3. 이터레이션 반복 후 `--no-ff` 병합

## MUST 규칙

- 각 개발 단계마다 spec/plan 먼저 수정 후 코드 변경
- research 문서는 해당 이터레이션에 같이 커밋 (나중에 따로 넣으면 히스토리 맥락 안 맞음)
- `git add` 전 diff 확인 필수 — 사용자 수정사항이 포함되지 않았는지 확인
- `/implement` 완료 후 로컬 브랜치 삭제 안 함 (`git branch -d` 생략)
