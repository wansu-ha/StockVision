# Spec/Plan 문서 생명주기

## 상태 정의

| 상태 | 의미 | 본문 수정 |
|------|------|----------|
| `초안` | 작성 중, 리뷰/수정 필요 | O |
| `확정` | 리뷰 완료, 구현 대기 | 최소 (오류 교정만) |
| `구현 완료` | 코드 존재. 코드가 정본 | X (헤더만 갱신) |

## 규칙

- 상태 전환: 초안 → 확정 → 구현 완료 (역행 금지)
- `구현 완료` 문서는 본문을 고치지 않는다 — 코드가 진실의 원천
- 계획과 실제 코드가 다를 수 있음 — 정상. 문서는 "왜 이렇게 설계했는지"의 히스토리
- 미구현 기능 추적: spec 수용 기준의 `- [ ]` / `- [x]` 체크박스로 관리
- 새 이터레이션이 필요하면 기존 문서를 수정하지 말고 새 spec/plan 작성 (또는 버전 올림)

## 상태 헤더 형식

spec/plan 최상단 메타데이터 블록에 명시:

```
> 작성일: 2026-03-04 | 상태: 구현 완료 | Unit 1
```

## 코드 vs 문서 우선순위

1. **코드** — 유일한 진실의 원천 (구현된 시그니처, 모델, 경로)
2. **spec 수용 기준** — 완료 추적용 체크리스트
3. **spec/plan 본문** — 설계 의도의 히스토리 (과거 시점 기록)
