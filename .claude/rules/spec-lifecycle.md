# Spec/Plan 문서 생명주기

## 상태 정의

| 상태 | 의미 | 본문 수정 |
|------|------|----------|
| `초안` | 작성 중, 리뷰/수정 필요 | O |
| `확정` | 리뷰 완료, 구현 대기 | 최소 (오류 교정만) |
| `구현 완료` | 코드 존재. 코드가 정본 | X (헤더만 갱신) |
| `부분 구현` | 일부만 완료. 괄호 안에 완료/미완료 범위 명시 | 최소 |
| `보류` | 방향 전환 또는 미래로 미룸. 괄호 안에 사유/시점 명시 | X |

## 규칙

- 상태 전환: 초안 → 확정 → 구현 완료 (역행 금지)
- 예외적 전환: `구현 완료 → 부분 구현`(초기 판정 번복), 어느 상태든 `→ 보류` 가능
- `구현 완료` 문서는 본문을 고치지 않는다 — 코드가 진실의 원천
- 계획과 실제 코드가 다를 수 있음 — 정상. 문서는 "왜 이렇게 설계했는지"의 히스토리
- 미구현 기능 추적: spec 수용 기준의 `- [ ]` / `- [x]` 체크박스로 관리
- 새 이터레이션이 필요하면 기존 문서를 수정하지 말고 새 spec/plan 작성 (또는 버전 올림)

## SUPERSEDED 마크

- 후속 spec이 기존 spec을 **대체**할 때 기존 spec의 상태 헤더 괄호에 `(SUPERSEDED by <path>)` 기입
- 후속 spec에는 `대체: <path>` 필드 추가 (본문 최상단)
- 두 spec 모두 보존. 코드 변경은 최신 spec을 따름

## plan만 존재 케이스

- 작은 작업은 spec 없이 plan만 작성하는 경우가 있음 (stability, ux-polish, docs-cleanup 등)
- 규칙 위반 아님. plan에 "목표 + 수용 기준"을 포함하면 spec 역할을 겸함
- 상태 헤더 관리 규칙은 동일하게 적용

## 상태 헤더 형식

spec/plan 최상단 메타데이터 블록에 명시:

```
> 작성일: 2026-03-04 | 상태: 구현 완료 | Unit 1
> 작성일: 2026-03-24 | 상태: 부분 구현 (H1/H2 완료, H3~ 미구현)
> 작성일: 2026-03-12 | 상태: 보류 (v2 이후로 미룸, 2026-03-24 결정)
> 작성일: 2026-03-12 | 상태: 구현 완료 (SUPERSEDED by `spec/remote-ops/`)
```

## 코드 vs 문서 우선순위

1. **코드** — 유일한 진실의 원천 (구현된 시그니처, 모델, 경로)
2. **spec 수용 기준** — 완료 추적용 체크리스트
3. **spec/plan 본문** — 설계 의도의 히스토리 (과거 시점 기록)
